## 1. Context & problem

This spec is the writeup of a forensic pass over real Claude Code session transcripts where the
codex-paired-superpowers plugin was actually used (not developed). Six genuine-usage sessions were
analyzed across three projects — Inkling, Vesikaa, and north-star-mobile — totaling ~5,000 tool calls
and ~215 Codex MCP invocations. Methodology: parse the JSONL transcripts structurally (tool_use /
tool_result pairs), count `is_error` results and codex-side failure language in *results* (not in
prose discussing bugs), and detect runs of repeated-identical tool calls. Evidence counts below are
aggregate across the analyzed transcripts unless noted.

The plugin works — full features shipped through brainstorm → plan → autopilot. But the transcripts
reveal five recurring reliability/efficiency defects that degrade the experience and, in one case,
silently undermine a headline guarantee (Codex independent audit).

## 2. What v0.12.0 already fixed (scope boundary — do NOT re-litigate)

- **Codex read-only spec/plan writes.** v0.12.0 set `sandbox_mode=workspace-write` in the codex MCP
  server args. Pre-v0.12.0 transcripts show Codex generating a full spec then failing to write it
  (*"apply_patch was rejected because this Codex session still has a read-only filesystem sandbox"*),
  forcing Claude to rewrite it. Spec/plan files live inside the workspace, so workspace-write covers
  this case. **Considered fixed; this spec does not address it except where item 3.1 overlaps.**
- **`/goal`-driven app-autopilot as the default path.** v0.12.0 marked `app-autopilot` experimental
  opt-in and pointed brainstorming's Phase 5 handoff back to `autopilot` + ralph-loop (battle-tested
  loop-prevention). The `/goal` loop-forever risk is therefore no longer on the default path.
  **Considered handled; out of scope here.**

## 3. Goals

<<<GOALS>>>
- Goal 1: When Codex reviews a slice/spec/plan, its verdict reflects tests/builds it actually executed — not a paper review. Codex either runs the project's test/build commands successfully inside its sandbox, or it explicitly declares "did not execute" and that declaration is treated as missing-audit by the gate (so a paper review cannot satisfy a SHIP).
- Goal 2: A wrong/stale Codex model id (e.g. `gpt-5.2-codex`) can never reach the Codex API. The model is pinned at a layer Claude cannot accidentally override, so the "model not supported for this account" 400 cannot happen.
- Goal 3: When the persistent Codex thread is lost mid-feature ("Session not found for thread_id"), the workflow detects it and recovers gracefully (re-establishes a thread with replayed context) instead of silently losing the review history.
- Goal 4: The v0.10.1 audit gate stops surfacing as a "PreToolUse:Bash hook error" mid-flow. Either the skills record audit evidence before attempting to log a SHIP round (so the gate never trips on the happy path), or the block is presented as an expected, actionable step rather than an error.
- Goal 5: Repeated failed edits ("File has not been read yet", "String to replace not found") stop wasting turns. The implementer/orchestrator reads before editing and does not retry an identical failing edit.

## 4. Per-defect detail

### 4.1 Codex sandbox cannot execute tests/builds (highest impact)

**Evidence (verbatim from results):**
- *"I did not rerun tests in this read-only review. Cargo was not run…"* (Inkling)
- *"this sandbox cannot write to `/Users/mkr/fvm/versions/3.44.0/bin/cache`, so Flutter exited before tests ran"* (north-star, ×3)
- *"the sandbox blocked emulator ports with `EPERM`"* / *"I still can't run the emulator in this sandbox due [to] port restrictions"* (Vesikaa, multiple)

**Root cause (inference, HIGH confidence):** `sandbox_mode=workspace-write` permits writes inside the
workspace only. It does **not** grant (a) writes to paths outside the workspace — e.g. the FVM Flutter
cache at `/Users/mkr/fvm/...` — nor (b) network/port access for the Firestore emulator. So on exactly
the project types in use (Flutter mobile, Firestore-emulator-backed), Codex still cannot run the test
suite. Its "audit" is structurally a paper review.

**Why it matters:** v0.10.1's headline feature is "Codex independently audits the codebase against
goals before SHIP." On these projects that guarantee is hollow — Codex SHIPs code it never ran, and
the verdict text quietly admits it ("I did not rerun…"). This is a correctness/trust defect, not just
inefficiency.

**Proposed direction (to be hardened in brainstorming):**
- Option A: escalate the sandbox (`sandbox_mode=danger-full-access`, or `workspace-write` +
  `network_access=true` + additional writable roots for the FVM/global caches). Verify which codex
  config keys exist and whether per-project writable-root config is feasible.
- Option B: keep the sandbox tight but make "did not execute" a **first-class, gate-enforced** state —
  if Codex's audit log shows no executed test command, the gate treats it as missing audit (cannot
  SHIP), and the skill explicitly routes test execution to Claude/subagent instead. This preserves the
  guarantee honestly rather than pretending.
- Likely answer: B as the safety floor (never let a paper review pass as a real one) + A where the
  environment allows it.

**Acceptance:** a slice review on a Flutter or emulator project either contains an executed-test audit
entry, or the gate refuses SHIP with a clear "tests were not executed by Codex — execute via <path>"
message.

### 4.2 Stale model-id 400s (`gpt-5.2-codex`)

**Evidence:** `{"status":400,"error":{"type":"invalid_request_error","message":"The 'gpt-5.2-codex'
model is not supported when using Codex with a ChatGPT account."}}` (×2). The brainstorming SKILL
already warns in prose: "do NOT substitute schema-description example values like `gpt-5.2-codex` …
always pass `gpt-5.5`." The warning did not hold.

**Root cause:** the model is a per-call MCP parameter chosen by Claude; the only guard is a prose
instruction, which is not enforcement. The MCP tool's own schema docstring offers `gpt-5.2-codex` as
an example, which leaks into calls.

**Proposed direction:** pin the model in `mcpServers.codex.args` (e.g. `-c model=gpt-5.5`), the same
layer as `sandbox_mode`. **OPEN/ASSUMED:** confirm a config-level model pin overrides the per-call
`model` param; if it does not, fall back to a wrapper or a hook that rejects non-`gpt-5.5` model values
before dispatch.

**Acceptance:** a deliberately wrong `model` in a codex MCP call still results in a `gpt-5.5` run (or a
fast local rejection), never a 400 from the API.

### 4.3 Codex thread loss — "Session not found for thread_id" (×7)

**Evidence:** `codex-reply` failing with `Session not found for thread_id: 019e3768-…` across multiple
sessions. The persistent one-thread-per-feature model is the plugin's core premise; when the thread
dies (compaction, MCP server restart, idle timeout) the 7-round review context is lost.

**Root cause (inference):** Codex MCP threads are not durable across MCP server restarts / long idle
gaps; the sidecar stores the thread id but there is no detection/recovery when that id goes stale.

**Proposed direction:** detect the "Session not found" result, then re-establish a thread and re-seed
it with the goals block + a compact replay of prior rounds (the sidecar already stores rounds), and
record the thread-id rotation in the sidecar. Surface to the user in plain English ("lost my Codex
thread, picking it back up").

**Acceptance:** injecting a stale thread id mid-feature results in automatic recovery + a sidecar note,
not a halt or lost context.

### 4.4 Audit gate surfaces as a "hook error" mid-flow (×21)

**Evidence:** `PreToolUse:Bash hook error: [bash ".../hooks/honest-reporting.sh" pretooluse]:
Honest-reporting audit gate (v0.10.1): refusing to log SHIP verdict for spec round 5. Missing audit
evidence for side(s): claude, codex.` Occurred 21× — the orchestrator repeatedly tried to log a SHIP
round *before* recording audit entries, got hard-blocked, and had to backtrack.

**Root cause:** the gate is correct, but the skills' happy-path ordering logs the round
(`sidecar-append-round`) before recording audits (`sidecar-append-audit`). The gate then trips every
time, and it presents as a scary "hook error" rather than an expected step.

**Proposed direction:** reorder the per-round procedure in `writing-plans` and `brainstorming` so audit
entries are appended *first*, then the round is logged (gate passes silently on the happy path).
Separately, soften the message so that when it does fire it reads as an actionable instruction, not a
crash.

**Acceptance:** a normal double-SHIP round produces zero "hook error" surfaces; the audit entries exist
before the round log every time.

### 4.5 Edit-retry waste — "File has not been read yet" (×31) + "String to replace not found" (×12)

**Evidence:** 43 failed edits aggregate, including a run of **9 identical failing Edits** on one file
(north-star) and 7× and 3× runs elsewhere — the implementer retried the same edit verbatim. "File has
not been read yet" is the single most common error category in the long autopilot sessions.

**Root cause:** in long sessions / across subagent boundaries, read-state for a file is lost, so an
Edit is attempted without a prior Read; or the `old_string` is stale and the identical edit is retried.

**Proposed direction:** in the implementer subagent contract + autopilot Phase B guidance, require a
Read immediately before an Edit when read-state is uncertain, and forbid retrying a byte-identical edit
that just failed (re-Read and recompute the target instead). Low-risk skill/contract wording change.

**Acceptance:** autopilot sessions show no runs of ≥3 identical failing Edits; "File has not been read
yet" rate drops to near zero.

## 5. Non-goals

- Reducing total tool-call volume as a primary objective. The 1,000–2,000 calls/feature is mostly
  inherent to 7-round × multi-phase pairing; fixing 4.4 and 4.5 will trim the waste tail without a
  redesign. A volume-reduction redesign is a separate future spec.
- Re-opening the `/goal` vs ralph-loop decision (settled in v0.12.0).
- Infra-level issues outside the plugin's control: auto-mode stalls when the model is briefly
  unavailable ("claude-opus-4-7 is temporarily unavailable"), and Bash `cwd` resets breaking
  relative-path commands. Note them; don't scope them here.

## 6. Open questions (for brainstorming)

- 4.1: Is `danger-full-access` acceptable for the user's threat model, or must we keep a tight sandbox
  and enforce "did not execute → cannot SHIP"? (Recommend the latter as the floor.)
- 4.2: Does `-c model=gpt-5.5` at the MCP-server layer override the per-call `model` param? Needs a
  quick empirical check before committing to that fix.
- 4.3: What exactly kills the thread — MCP restart, idle TTL, or compaction? Determines whether
  recovery is enough or we also need a keepalive.

## 7. Priority

1. **4.1 (sandbox/execute)** — silently guts the audit guarantee; correctness issue.
2. **4.4 (audit-gate ordering)** — 21 interruptions, cheap fix.
3. **4.2 (model pin)** — hard 400s, cheap fix, pending the override check.
4. **4.5 (edit-retry)** — wasted turns, low-risk wording fix.
5. **4.3 (thread recovery)** — highest-effort; rarer (7×) but corrupts the core premise when it hits.
