## 1. Context & problem

This spec is the writeup of a forensic pass over real Claude Code session transcripts where the
codex-paired-superpowers plugin was actually used, not developed. Six genuine-usage sessions were
analyzed across three projects — Inkling, Vesikaa, and north-star-mobile — totaling roughly 5,000 tool
calls and roughly 215 Codex MCP invocations. Methodology: parse the JSONL transcripts structurally
(`tool_use` / `tool_result` pairs), count `is_error` results and codex-side failure language in
results rather than prose discussing bugs, and detect runs of repeated-identical tool calls. Evidence
counts below are aggregate across the analyzed transcripts unless noted.

The plugin works: full features shipped through brainstorm, plan, and autopilot. The transcripts also
show five recurring reliability defects that degrade the experience and, in one case, silently
undermine the headline guarantee that Codex independently audits the codebase before SHIP.

## 2. What v0.12.0 already fixed (scope boundary — do NOT re-litigate)

- **Codex read-only spec/plan writes.** v0.12.0 set `sandbox_mode=workspace-write` in the codex MCP
  server args. Pre-v0.12.0 transcripts show Codex generating a full spec and then failing to write it:
  "apply_patch was rejected because this Codex session still has a read-only filesystem sandbox".
  Spec and plan files live inside the workspace, so workspace-write covers that case. This spec does
  not reopen it except where Goal 1 escalates review execution to full access.
- **`/goal`-driven app-autopilot as the default path.** v0.12.0 marked `app-autopilot` experimental
  opt-in and pointed brainstorming's Phase 5 handoff back to `autopilot` plus ralph-loop, the
  battle-tested continuation path. The `/goal` loop-forever risk is out of scope here.

## 5. Decisions and required changes

### 5.1 Goal 1 — full-access review execution plus phase-scoped verification floor

**Decision:** run Codex MCP review sessions with full access at the MCP-server/config layer and make
executed verification a machine-checkable audit field for code-bearing phases only. Per-call sandbox
override is technically available and empirically works, but it is not sufficient as the primary
mechanism because `codex-reply` has no `sandbox` field. The reliable default for every newly opened
review thread is the server process configuration.

Change `.claude-plugin/plugin.json` to launch Codex with these args:

```json
{
  "command": "codex",
  "args": [
    "mcp-server",
    "-c",
    "sandbox_mode=\"danger-full-access\"",
    "-c",
    "approval_policy=\"never\"",
    "-c",
    "model=\"gpt-5.5\""
  ]
}
```

Do not rely on approval prompts: reviews are launched by an MCP stdio server inside Claude Code, so an
approval prompt is a hang or a failed audit path.

Extend the audit schema in `appendAuditLog`:

```json
{
  "phase": "spec|plan|plan-slice:<slice>|review-slice:<slice>|...",
  "round": 1,
  "side": "claude|codex",
  "commands": [
    {
      "cmd": "npm test",
      "summary": "exit 0; 42 tests passed",
      "kind": "verification",
      "exit_code": 0,
      "ran_at": "ISO-8601"
    }
  ],
  "verdict_basis": "..."
}
```

Rules:

- `kind` is required for new audit entries and is one of `inspection`, `verification`, or `other`.
- `exit_code` is required for `kind: "verification"` and must be an integer.
- `hasAuditFor` remains available for back-compat, but the SHIP gate uses a new
  `hasExecutedVerificationFor(specPath, { phase, round, side })` only for code-bearing phases.
- Design-only phases stay on the v0.10.1 inspection-audit gate: a SHIP verdict in `spec`, `plan`, or
  `plan-slice:<slice>` requires `hasAuditFor`, but does not require a `verification` command. These
  phases review documents and plans; making tests mandatory here would prevent normal brainstorming
  and writing-plans from shipping.
- Code-bearing phases require executed verification: a SHIP verdict in `implement:<slice>`,
  `review-slice:<slice>`, `live-verification:<slice>`, or `post-merge-review:<slice>` is gate-eligible
  only when that side has at least one command with `kind: "verification"` and `exit_code: 0`.
- Implement the phase predicate as an explicit helper, for example
  `requiresExecutedVerification(phase)`, and test it. Unknown phases default to inspection-audit only
  unless they are added to the code-bearing allowlist.
- If Codex could not execute verification in a code-bearing phase, the sidecar entry may record
  inspection commands and a `verdict_basis` that says it did not execute, but
  `hasExecutedVerificationFor` must return false. That round cannot log a Codex SHIP.

Acceptance tests:

- Unit: `appendAuditLog` rejects missing `kind`, rejects `kind: "verification"` without integer
  `exit_code`, and preserves legacy audits only for listing, not SHIP gating.
- Unit: `hasExecutedVerificationFor` returns true only for matching `(phase, round, side)` with at
  least one `verification` command whose `exit_code` is `0`.
- Unit: `requiresExecutedVerification` returns false for `spec`, `plan`, and `plan-slice:<slice>`,
  and true for `implement:<slice>`, `review-slice:<slice>`, `live-verification:<slice>`, and
  `post-merge-review:<slice>`.
- Hook: `decideAuditGate` blocks a code-bearing SHIP round when `hasAuditFor` is true but
  `hasExecutedVerificationFor` is false, while allowing a design-phase SHIP with inspection audit.
- Installed smoke: a Codex MCP review prompt can write outside the repo to a temp path under
  `~/.codex/tmp` and can run a project command without approval prompts.

### 5.2 Goal 2 — model pin for plugin-authored calls

**Decision:** do not add a stdio proxy in v0.13.0. The omit-model probe showed that when the `codex`
tool call does not include a per-call `model`, the server-level `-c model="gpt-5.5"` default is used.
A proxy would add a new always-on JSON-RPC framing component in a reliability release. The smaller
mechanism is to pin the server config and make plugin-authored calls omit `model` entirely.

Required changes:

- Set `.claude-plugin/plugin.json` Codex args to include `-c model="gpt-5.5"`, `-c
  sandbox_mode="danger-full-access"`, and `-c approval_policy="never"`.
- Remove every plugin-authored instruction that tells Claude to pass a top-level per-call `model` to
  `mcp__plugin_codex-paired-superpowers_codex__codex`.
- Keep `config: { "model_reasoning_effort": "high" }` or `"medium"` as appropriate; reasoning effort
  is not the model id and remains allowed.
- Add a skill/doc structure test that fails if any plugin-authored MCP invocation snippet includes a
  top-level `"model"` field or `model:` key for the `codex` tool. The only allowed mentions are
  explanatory hazard text saying the raw MCP schema includes stale examples and plugin-authored calls
  must omit the field.
- Add an installed smoke that calls `codex` through the plugin's configured server with no per-call
  `model` and asserts `session_configured.model === "gpt-5.5"`.

This deliberately scopes enforcement to plugin-authored workflows. A raw manual MCP call can still
pass a per-call `model` because the upstream Codex tool exposes that field; preventing arbitrary manual
misuse would require the proxy. v0.13.0 chooses the smaller reliability fix because the observed
failures came from plugin skill instructions and schema-example leakage, not from a user intentionally
overriding the model.

Acceptance tests:

- Manifest test: `.claude-plugin/plugin.json` includes `model="gpt-5.5"`,
  `sandbox_mode="danger-full-access"`, and `approval_policy="never"` in `mcpServers.codex.args`.
- Skill-structure test: plugin-authored `codex` tool invocation examples omit top-level `model`.
- Installed smoke: omitted per-call model starts a thread whose `session_configured.model` is
  `gpt-5.5`.
- Regression: the stale `gpt-5.2-codex` example may appear only in hazard text that says it must not
  be passed.

### 5.3 Goal 3 — thread-loss detection and recovery

**Decision:** `Session not found for thread_id` is caused by the MCP server losing in-memory sessions,
at least on process restart. The sidecar's persisted thread id is not enough. Recovery is required at
every skill step that calls `codex-reply`.

Add sidecar APIs:

- `setCodexThreadId(specPath, { role = "paired-reviewer", oldThreadId, newThreadId, reason, phase,
  round })`
- CLI: `sidecar-rotate-thread-id --specPath <path> --oldThreadId <id> --newThreadId <id> --reason
  session-not-found [--phase <phase>] [--round <n>]`
- CLI: `sidecar-replay-context --specPath <path> [--phase <phase>] [--round <n>]`

`setCodexThreadId` updates both the legacy `codex_session` field and
`role_sessions["paired-reviewer"]` when the role is paired-reviewer. It appends:

```json
{
  "old_thread_id": "...",
  "new_thread_id": "...",
  "role": "paired-reviewer",
  "reason": "session-not-found",
  "phase": "plan",
  "round": 3,
  "rotated_at": "ISO-8601"
}
```

to `thread_rotations[]`. `sidecar-thread-id` must use `getCodexThreadId` rather than reading
`loadSidecar(specPath).codex_session` directly, because mixed v0.9+ sidecars can have a canonical
`role_sessions` value that differs from the legacy field.

Detection point:

- Immediately after every `codex-reply` MCP call in `brainstorming`, `writing-plans`,
  `subagent-driven-development`, and `autopilot`.
- Treat the response as stale-thread when `isError === true` and `content` or the text payload matches
  `/Session not found for thread_id:/`.

Recovery flow:

1. Read the persisted goals block with `sidecar-get-goals`; if absent, extract the current
   `<<<GOALS>>>...<<<END_GOALS>>>` block from the spec file.
2. Generate compact replay with `sidecar-replay-context`: feature name, current spec/plan path,
   `rounds[]` summarized by phase, round, Claude verdict, Codex verdict, open contentions, and any
   prior `thread_rotations[]`.
3. Invoke the initial `codex` tool, not `codex-reply`, with a recovery prompt containing:
   - the goals block;
   - replay context;
   - the current artifact path and phase;
   - the pending prompt that failed against the stale thread;
   - instruction to continue as the same paired reviewer.
4. Persist the returned `threadId` via `sidecar-rotate-thread-id`.
5. Surface one plain-English line to the user or transcript: "Codex thread was lost; opened a new
   thread and replayed the sidecar context."
6. Continue the round with the recovery response. Do not silently discard prior review history.

Acceptance tests:

- Unit: sidecar rotation updates `codex_session`, `role_sessions.paired-reviewer`, and
  `thread_rotations[]` atomically.
- Unit: `sidecar-thread-id` returns `role_sessions.paired-reviewer` when present.
- Unit: replay output includes goals, prior rounds, open contentions, and thread rotations.
- Integration with fake MCP: a `codex-reply` stale-thread response causes exactly one initial `codex`
  recovery call and one sidecar rotation, then continues with the recovery content.
- Installed smoke: create a thread, restart `codex mcp-server`, attempt reply, and verify the workflow
  recovers rather than halting on `Session not found`.

### 5.4 Goal 4 — audit gate ordering and actionable wording

**Decision:** keep the gate, but stop relying on Claude to run two separate commands in the right
order. The happy path must use one atomic CLI command that records audits and the round together.

The current brainstorming and writing-plans skills already describe `sidecar-append-audit` before
`sidecar-append-round`, but the observed 21 failures happened at runtime anyway. A documentation-order
test would only prove what is already true. v0.13.0 must make the runtime path structurally hard to
misorder.

Required changes:

- Add a CLI command:

  ```bash
  node ${CLAUDE_PLUGIN_ROOT}/lib/codex-bridge/cli.js sidecar-append-round-with-audits \
    --specPath "<spec-path>" \
    --payload '{
      "audits": [
        {
          "phase": "plan",
          "round": 2,
          "side": "claude",
          "commands": [
            {"cmd": "rg -n ...", "summary": "...", "kind": "inspection"}
          ],
          "verdict_basis": "..."
        },
        {
          "phase": "plan",
          "round": 2,
          "side": "codex",
          "commands": [
            {"cmd": "rg -n ...", "summary": "...", "kind": "inspection"}
          ],
          "verdict_basis": "..."
        }
      ],
      "round": {
        "phase": "plan",
        "round": 2,
        "claude": "SHIP",
        "codex": "SHIP"
      }
    }'
  ```

- Implement the command as a single locked sidecar mutation, not as shell-level sequencing. Under
  `withSidecarLock`, validate all audit entries, append them, validate the round against the same
  in-memory sidecar state, then append the round and save once. If any validation fails, write nothing.
- Validation inside the command must mirror the hook:
  - for SHIP in design phases, require an audit entry for that side;
  - for SHIP in code-bearing phases, require a successful `kind: "verification"` command for that
    side;
  - for REVISE, audit entries are allowed but not required.
- Update `brainstorming` and `writing-plans` to use `sidecar-append-round-with-audits` on the happy
  path instead of separate `sidecar-append-audit` then `sidecar-append-round` commands.
- Keep `sidecar-append-audit` and `sidecar-append-round` for manual recovery and back-compat. The
  PreToolUse audit gate continues to guard direct `sidecar-append-round` usage.
- Update `formatAuditGateMessage` from "refusing to log SHIP verdict" to "Audit evidence required
  before logging SHIP".
- List the exact missing condition per side:
  - no audit entry;
  - code-bearing phase audit exists but has no successful `kind: "verification"` command;
  - audit malformed/legacy and cannot satisfy v0.13.0 SHIP for a code-bearing phase.
- Keep the manual append-audit command example, but include `kind` and `exit_code` where relevant.
- Add CLI tests for atomicity: invalid second audit means neither the first audit nor the round is
  persisted; valid payload persists audits before the round in one save.
- Add skill-structure tests that the spec/plan happy path names
  `sidecar-append-round-with-audits`.
- Add hook tests for the new wording and for the phase-scoped verification requirement.

This cannot remove Claude Code's generic `PreToolUse:Bash hook error` prefix, but it prevents the
happy path from reaching the hook and makes the hook's own message actionable when it does.

### 5.5 Goal 5 — failed edit retry discipline

**Decision:** keep this as a skill/contract hardening in v0.13.0. The repeated failures were caused by
orchestrator/subagent behavior, not a missing reusable library.

Update `skills/subagent-driven-development/SKILL.md`, `skills/autopilot/SKILL.md`, and any implementer
prompt fragments used for Phase B with a short mandatory edit discipline:

- Before editing a file, read the file in the current turn if read-state is uncertain, if another
  agent may have touched it, or if the last read happened before a failed edit.
- If an edit fails with `File has not been read yet`, immediately read the file and recompute the edit.
- If an edit fails with `String to replace not found`, immediately re-read and inspect the target
  region; never retry the same `(file, old_string, new_string)` tuple byte-for-byte.
- Keep a local "last failed edit tuple" in the orchestrator's reasoning for the current turn; a
  byte-identical retry is a procedural error.
- For bulk edits, prefer a structured patch that includes enough surrounding context after the fresh
  read.

Acceptance tests:

- Skill-structure tests assert the exact phrases `File has not been read yet`,
  `String to replace not found`, and `never retry the same` appear in the relevant skill/prompt files.
- Replay metric for the six transcripts remains as baseline evidence; future transcript replay should
  flag runs of three or more identical failing edits.

## 6. Implementation plan outline (single v0.13.0 plan)

All five fixes ship together in one feature-scoped v0.13.0 plan. Suggested slices:

1. **Manifest pin and model-call cleanup.** Update `.claude-plugin/plugin.json` to pin
   `model="gpt-5.5"`, `sandbox_mode="danger-full-access"`, and `approval_policy="never"`; remove
   top-level per-call `model` from plugin-authored skill invocations; add manifest and skill-structure
   tests.
2. **Audit schema, phase predicate, and gate.** Extend `appendAuditLog`, `listAudits`,
   `hasExecutedVerificationFor`, `requiresExecutedVerification`, CLI commands, and `decideAuditGate`;
   update tests.
3. **Thread rotation and replay.** Add sidecar setters, CLI commands, recovery prompt builder, and
   fake-MCP tests for stale-thread recovery.
4. **Atomic round logging and skill updates.** Add `sidecar-append-round-with-audits`, update
   brainstorming/writing-plans to use it on the happy path, add recovery procedure, add phase-scoped
   audit instructions, and add edit-retry discipline.
5. **Installed smoke.** Add opt-in tests for real MCP sandbox/model/thread behavior; keep them out of
   default `npm test` unless guarded by an environment flag.

## 9. Acceptance criteria

- Goal 1: `.claude-plugin/plugin.json` no longer launches a workspace-write-only Codex MCP server for
  review sessions; SHIP audit gate requires successful executed verification only for code-bearing
  phases and keeps inspection audit for spec/plan design phases.
- Goal 2: plugin-authored Codex calls omit the per-call model field, `.claude-plugin/plugin.json` pins
  `model="gpt-5.5"`, and live smoke verifies omitted-model threads start on `gpt-5.5`.
- Goal 3: a restarted MCP server producing `Session not found for thread_id` causes automatic
  re-threading, replay, and a `thread_rotations[]` sidecar entry.
- Goal 4: normal SHIP round logging uses `sidecar-append-round-with-audits` and cannot log the round
  before its audits at runtime; forced direct-round gate failures are actionable and mention the
  missing audit or code-bearing verification command.
- Goal 5: implementer/autopilot instructions prohibit byte-identical failed edit retries and require
  a fresh read before recomputing edits.
