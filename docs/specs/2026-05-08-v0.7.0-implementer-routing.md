# v0.7.0 — Implementer Routing

**Status:** draft for review
**Date:** 2026-05-08
**Owner:** mkr
**Target version:** v0.7.0
**Validation:** critical
**Spec path:** `docs/specs/2026-05-08-v0.7.0-implementer-routing.md`

## 1. Goal

Extend autopilot's Phase B so implementation work can route to either Sonnet subagents or Codex MCP calls, defaulting to Codex, falling back only on real implementer failure, and running independent consecutive slices concurrently in isolated git worktrees.

## 2. Operating Principles

What changes:

- Phase B becomes a routing dispatch instead of a hard-coded Sonnet subagent dispatch.
- Default implementer is Codex.
- Per-slice override via plan frontmatter `**Implementer:** sonnet | codex`. Honored unconditionally. No directive → Codex.
- Literal `**Implementer:** auto` is malformed.
- Routing logic = preferred → fallback → halt. Reactive real errors/timeouts only; no scoring rubric.
- The orchestrator never gets stuck on routing. It falls back when one agent is unavailable and halts only with `implementer-unavailable` when both fail.
- Parallel implementation is allowed for consecutive slices with non-overlapping `**Files:**` sets.
- Mixed Codex/Sonnet parallel batches are allowed.
- Each parallel slice runs in its own bootstrapped worktree at `<repo>/.git-worktrees/slice-<N>`.
- Commit conventions drop the required `Co-Authored-By: Claude` trailer.
- Provenance hook validation becomes subject-only.

What stays the same:

- Phase A, Phase C, Phase D, and Phase E stay unchanged.
- The persistent feature-review Codex thread remains for review phases.
- Implementation Codex dispatches use fresh threads, not the feature review thread.
- Sidecar is still the durable source of truth.
- Git reconciliation still verifies autopilot-owned commits before advancing state.
- `BLOCKED` and `NEEDS_CONTEXT` remain real blockers, not routing failures.

## 3. Routing Model

Preferred implementer resolution:

| Slice directive | Preferred implementer |
|---|---|
| `**Implementer:** codex` | Codex |
| `**Implementer:** sonnet` | Sonnet |
| no directive | Codex |
| `**Implementer:** auto` | malformed |
| any other value | malformed |

Routing sequence:

```text
preferred implementer
  -> fallback implementer on implementation-dispatch failure
  -> halt with implementer-unavailable only if both fail
```

Failure-triggers-fallback definition:

1. MCP error.
2. Subagent dispatch error.
3. 10-minute timeout.
4. Zero commits produced.
5. Non-conforming commits emitted.

NOT a fallback trigger:

- Agent reports `BLOCKED`.
- Agent reports `NEEDS_CONTEXT`.

Those are real blockers, such as "the spec is unclear." Autopilot halts with the existing blocker failure modes rather than trying the other implementer.

Codex-specific blocker halt reasons:

- `codex-blocked`
- `codex-needs-context`

Sonnet-specific blocker halt reasons stay existing:

- `subagent-blocked`
- `subagent-needs-context`

## 4. Implementer Dispatch: Codex Path

Codex implementation dispatch uses a fresh thread per dispatch.

Use `codex`, not `codex-reply`. Implementation threads are not the same as the persistent feature review thread.

Before Codex dispatch, the worktree must be bootstrapped.

Worktree bootstrap policy:

- v0.7.0 uses dependency symlinking, not install commands.
- The orchestrator links declared repo-local dependency artifact paths from the integration checkout into the worktree before dispatch.
- Default symlink candidates are:
  - `node_modules`
  - `.venv`
  - `venv`
- A project may declare additional symlink paths in `.codex-paired/project.json`:

```json
{
  "worktree_bootstrap": {
    "symlinks": ["node_modules", ".venv"]
  }
}
```

Rules:

- Symlink source must exist in the integration checkout.
- Symlink target must not already exist unless it is already the correct symlink.
- Symlink paths must be repo-relative and must not contain traversal.
- Bootstrap runs before implementer dispatch in every worktree.
- Bootstrap failure halts with `worktree-bootstrap-failed`.
- v0.7.0 does not run `npm install`, `pnpm install`, `pip install`, `cargo fetch`, or other dependency-install commands automatically.
- If a repo needs an install step instead of symlinked local artifacts, that repo is out of scope for v0.7.0 parallel worktree dispatch unless its dependencies are already available through declared symlinks.

Locked MCP call shape:

```json
{
  "tool": "mcp__plugin_codex-paired-superpowers_codex__codex",
  "arguments": {
    "prompt": "<implementation prompt>",
    "sandbox": "workspace-write",
    "approval-policy": "never",
    "cwd": "<worktree absolute path>",
    "model": "gpt-5.5",
    "config": {
      "model_reasoning_effort": "high"
    }
  }
}
```

Codex receives:

- Slice id.
- Full slice task list.
- Phase A test list.
- Phase A validation coverage.
- Commit Conventions.
- Slice `**Files:**` block, when present.
- Required test/verification commands from the plan.
- Instruction to commit logical changes using subject-only commit conventions.
- Instruction that no trailer is required.
- Instruction to report `DONE`, `DONE_WITH_CONCERNS`, `BLOCKED`, or `NEEDS_CONTEXT`.

Timeout:

- 10 minutes from successful MCP dispatch start.
- Timeout triggers fallback.
- The fallback starts fresh after worktree reset.

Thread handling:

- Record returned `thread_id` in sidecar when available.
- `thread_id` may be null if the MCP result does not expose one but commits reconcile successfully.
- Never continue a Codex implementation thread across fallback or later rounds.

## 5. Implementer Dispatch: Sonnet Path

Sonnet path is existing Phase B behavior, unchanged except for the updated commit convention and worktree bootstrap requirement.

Before Sonnet dispatch, the worktree must be bootstrapped using the same symlink policy from §4.

The Sonnet subagent receives:

- Slice id.
- Full slice task list.
- Phase A test list.
- Phase A validation coverage.
- Commit Conventions.
- Required test/verification commands from the plan.

It follows TDD, commits per logical task, and reports:

- `DONE`
- `DONE_WITH_CONCERNS`
- `BLOCKED`
- `NEEDS_CONTEXT`

Sonnet dispatch failure means subagent dispatch error, crash/no status, 10-minute timeout, zero commits, or non-conforming commits.

`BLOCKED` and `NEEDS_CONTEXT` are not dispatch failures.

## 6. Mid-Implementation Failure Recovery

Every Phase B attempt runs in a worktree rooted at `slice_start_sha`.

On failure that triggers fallback:

1. Record the failed dispatch in `dispatches[]`.
2. Mark outcome:
   - `failed-fallback-pending` if the other implementer remains.
   - `failed-halted` if no fallback remains.
3. Reset the worktree:

```bash
git reset --hard <slice_start_sha>
```

4. Re-run worktree bootstrap.
5. Dispatch the fallback agent fresh.
6. Reconcile only the fallback's commits.

Do not salvage partial work. Do not reuse a partial Codex thread. Do not ask the fallback to continue prior commits.

If reset fails, halt with `worktree-reset-failed` and leave the worktree for inspection.

If bootstrap fails after reset, halt with `worktree-bootstrap-failed`.

## 7. Parallel Slice Execution

Parallelism applies only to Phase B implementation.

Phase A must already be double-SHIP'd for every slice in the parallel candidate window. Phase C, Phase D, Phase E, and slice shipment still run in slice order after implementation commits are integrated.

Parallel candidate rule:

- When multiple consecutive slices are Phase B-ready and each has a valid `**Files:**` block, autopilot may dispatch them concurrently.
- Mixed-implementer parallel batches are explicitly allowed.
- Example: slice 3 using Codex and slice 4 using Sonnet may dispatch concurrently if their `**Files:**` sets do not overlap.
- Conflict detection runs on file sets only, not implementer choice.
- If candidate slices have overlapping file paths, force serial execution for those slices.
- If a slice selected for parallel dispatch is missing `**Files:**`, halt with `parallel-files-missing`.
- If a selected slice has malformed `**Files:**`, halt with `parallel-files-malformed`.

Conflict detection:

- Parse each slice's `**Files:**` block into exact repo-relative paths.
- Normalize to POSIX-style repo-relative paths.
- Reject absolute paths.
- Reject `.` and `..` traversal.
- Reject empty entries.
- Reject globs.
- Reject directory-only entries.
- Reject backslash separators.
- Reject duplicate entries within a slice.
- Intersect normalized path sets across candidate slices.
- Any intersection means no parallelism for those slices; execute serially.

Parallel setup behavior:

1. Create one worktree per slice at `<repo>/.git-worktrees/slice-<N>`.
2. Branch each worktree from the same `slice_start_sha`.
3. Branch name: `<slice-id>-impl`, for example `slice-3-impl`.
4. Bootstrap each worktree before dispatch using the symlink policy in §4.
5. If any bootstrap fails, halt the batch with `worktree-bootstrap-failed` before dispatching implementers.

Parallel dispatch behavior:

- Dispatch all candidate slices concurrently.
- Each dispatch uses its own worktree as `cwd`.
- Each dispatch reconciles commits on its own branch.
- The autopilot orchestrator achieves concurrency by issuing all parallel-batch dispatch tool calls in a single assistant turn using Claude's parallel-tool-call mechanism.
- Serial `await` across separate turns is not concurrent dispatch and is non-conforming.

Integration strategy:

- Pin ordered cherry-pick from the slice's worktree branch onto the integration branch.
- No merge commits.
- Rationale: merge commit subjects like `Merge branch 'slice-3-impl'` would violate the subject-only provenance hook.
- Integrate in ascending slice order.
- Cherry-pick commits from `slice_start_sha..slice-N-impl`.
- On cherry-pick conflict, abort cherry-pick and halt with `worktree-merge-conflict`.

## 8. Worktree Lifecycle

Create worktree at dispatch:

```bash
mkdir -p <repo>/.git-worktrees
git worktree add -b <slice-id>-impl <repo>/.git-worktrees/slice-<N> <slice_start_sha>
```

Then bootstrap the worktree:

```bash
ln -s <repo>/<artifact-path> <repo>/.git-worktrees/slice-<N>/<artifact-path>
```

The orchestrator owns lifecycle:

- Create at dispatch.
- Bootstrap before implementer runs.
- Reset and re-bootstrap on fallback.
- Cherry-pick successful commits to the integration branch.
- Remove clean worktrees after successful integration.
- Leave diagnostic worktrees on halt if they contain dirty state, conflicts, or unreconciled commits.

Success path:

1. Reconcile worktree commits.
2. Record `head_sha` and `commit_count`.
3. Ordered cherry-pick onto integration branch.
4. Update `last_commit_sha = HEAD`.
5. Mark implement phase shipped.
6. Remove worktree.
7. Delete temp branch only after commits are reachable from integration branch.

Halt path:

- Clean failed worktrees may be removed after diagnostics are persisted.
- Dirty/conflicted/unreconciled worktrees must be left in place.
- Halt summary must include absolute worktree path and branch name.

Worktree safety:

- If `<repo>/.git-worktrees/slice-<N>` already exists and is not a valid clean worktree for the same slice, halt with `worktree-path-conflict`.
- If `.git-worktrees/` is not gitignored, halt with `worktree-gitignore-missing` before creating worktrees.
- If dependency symlink bootstrap fails, halt with `worktree-bootstrap-failed`.

## 9. Plan Frontmatter Extensions

Implementer directive:

```markdown
**Implementer:** codex
```

Allowed values:

- `codex`
- `sonnet`

Rules:

- Trim surrounding whitespace.
- Values are lower-case exact matches.
- Missing directive means Codex.
- Literal `auto` is malformed.
- Unknown value halts with `implementer-directive-malformed`.

Files block:

```markdown
**Files:**
- lib/codex-bridge/implementer-routing.js
- tests/codex-bridge/implementer-routing.test.js
```

`**Files:**` is required for any slice intending parallel dispatch.

Parser rules:

- Block starts at a line equal to `**Files:**` after trimming.
- Block continues through consecutive `- <path>` bullet lines.
- Block ends at blank line, heading, or next bold directive.
- At least one file is required.
- Paths must be exact repo-relative file paths.
- Inline form like `**Files:** lib/foo.js` is malformed.
- Globs/directories/traversal/absolute paths are malformed.

New CLI helpers:

```bash
node lib/codex-bridge/cli.js parse-implementer-directive
node lib/codex-bridge/cli.js parse-files-block
node lib/codex-bridge/cli.js detect-parallel-slices
```

CLI behavior follows existing parser convention:

- Exit 0 with JSON on success.
- Exit 2 with `{ "defect": "<code>", "detail": "..." }` on parser defect.
- Exit 1 on infrastructure failure.

## 10. Sidecar Persistence

Add dispatch provenance under the implement phase:

```json
{
  "slice_reviews": {
    "slice-3": {
      "phases": {
        "implement": {
          "shipped": false,
          "preferred_implementer": "codex",
          "fallback_implementer": "sonnet",
          "parallel_group": "parallel-2026-05-08T12:00:00.000Z-3-5",
          "parallel_suppressed_reason": null,
          "worktree": "/repo/.git-worktrees/slice-3",
          "bootstrap": {
            "symlinks": ["node_modules"],
            "completed_at": "2026-05-08T12:00:02.000Z"
          },
          "dispatches": [
            {
              "slice_id": "slice-3",
              "agent": "codex",
              "thread_id": "019e...",
              "dispatched_at": "2026-05-08T12:00:00.000Z",
              "completed_at": "2026-05-08T12:04:00.000Z",
              "worktree": "/repo/.git-worktrees/slice-3",
              "head_sha": "abc123",
              "commit_count": 2,
              "outcome": "shipped"
            }
          ],
          "commits": ["abc123", "def456"],
          "concerns": []
        }
      }
    }
  }
}
```

Required dispatch fields:

- `slice_id`
- `agent`: `"codex"` or `"sonnet"`
- `thread_id?`
- `dispatched_at`
- `completed_at`
- `worktree`
- `head_sha`
- `commit_count`
- `outcome`: `"shipped" | "failed-fallback-pending" | "failed-halted"`

Dispatch entries are append-only.

If fallback ships, preserve both dispatches:

1. Failed preferred dispatch.
2. Shipped fallback dispatch.

Target size is lightweight; approximately 150 bytes per dispatch in normal compact form. Bootstrap metadata is allowed in addition to dispatch entries because dependency setup is part of the Phase B execution boundary.

## 11. Updated Commit Conventions

New convention is subject-only:

```text
(feat|test|fix|docs|refactor|chore)(slice:N): <description>
```

Examples:

```text
feat(slice:3): add implementer routing parser
test(slice:3): cover codex fallback after timeout
fix(slice:3): recover from empty commit range
docs(slice:3): document subject-only provenance hook
```

Trailer requirement is dropped entirely.

Rules:

- `Co-Authored-By: Claude` trailer is not required.
- Presence of trailer does not break compliance.
- Absence of trailer is fine.
- Existing repo commits with the old trailer still pass.
- Commit body is ignored by the provenance hook.
- Subject must match current slice number.

This applies to:

- Codex implementation dispatches.
- Sonnet implementation dispatches.
- Phase C fix subagents.
- Phase D docs commits.
- Phase E fix subagents.

Zero commits after Phase B implementation is a fallback trigger, unless the slice explicitly skips implementation before dispatch.

## 12. Updated Provenance Hook

`hooks/check-commit-provenance.sh` keeps active-anchor discovery but simplifies validation.

During active autopilot:

1. Read `<repo-root>/.codex-paired/active.json`.
2. Resolve the spec path.
3. Load the sidecar.
4. Read `autopilot.current_slice`.
5. Read most recent commit subject.
6. Check subject only:

```bash
^(feat|test|fix|docs|refactor|chore)\(slice:${CURRENT_SLICE}\):
```

7. Do not check trailers.
8. Do not check commit body.

Required hook tests:

- No anchor allows arbitrary commit.
- Conforming subject with old trailer passes.
- Conforming subject without trailer passes.
- Wrong slice subject blocks.
- Arbitrary manual subject blocks.
- Subdirectory commit resolves repo root correctly.
- Stdin filter still only evaluates Bash `git commit`.

## 13. Failure Modes

| Condition | Halt reason | Behavior |
|---|---|---|
| Bad `**Implementer:**` value | `implementer-directive-malformed` | Halt before dispatch. |
| Literal `**Implementer:** auto` | `implementer-directive-malformed` | Halt before dispatch. |
| Worktree dependency bootstrap fails | `worktree-bootstrap-failed` | Halt before implementer dispatch. |
| Codex MCP error | fallback trigger | Reset worktree, re-bootstrap, try Sonnet. |
| Sonnet dispatch error | fallback trigger | Reset worktree, re-bootstrap, try Codex. |
| 10-minute implementation timeout | fallback trigger | Reset worktree, re-bootstrap, try fallback. |
| Zero commits produced | fallback trigger | Reset worktree, re-bootstrap, try fallback. |
| Non-conforming commits emitted | fallback trigger | Cite SHA; reset worktree; re-bootstrap; try fallback. |
| Preferred and fallback both fail | `implementer-unavailable` | Halt; record both dispatch failures. |
| Codex returns `BLOCKED` | `codex-blocked` | Halt; no fallback. |
| Codex returns `NEEDS_CONTEXT` | `codex-needs-context` | Halt; no fallback. |
| Sonnet returns `BLOCKED` | `subagent-blocked` | Halt; no fallback. |
| Sonnet returns `NEEDS_CONTEXT` | `subagent-needs-context` | Halt; no fallback. |
| Missing Files block on selected parallel slice | `parallel-files-missing` | Halt before dispatch. |
| Malformed Files block | `parallel-files-malformed` | Halt before dispatch. |
| Overlapping Files sets | none | Force serial execution. |
| Worktree path conflict | `worktree-path-conflict` | Halt; do not delete unknown files. |
| Worktree not gitignored | `worktree-gitignore-missing` | Halt before worktree creation. |
| Worktree creation fails | `worktree-create-failed` | Halt with stderr and intended path. |
| Reset before fallback fails | `worktree-reset-failed` | Halt; leave worktree. |
| Ordered cherry-pick conflicts | `worktree-merge-conflict` | Abort cherry-pick; halt; leave branch/worktree. |
| Worktree cleanup fails | `worktree-cleanup-failed` | Record warning; include cleanup path. |
| Branch cleanup fails | `worktree-branch-cleanup-failed` | Record warning; continue if integrated. |
| Resume sees history divergence | `history-divergence` | Existing behavior. |
| Resume sees non-conforming integrated commit | `external-commit-detected` | Existing behavior, subject-only. |

## 14. Validation Rubric Coverage Applied To This Spec

- **tier:** critical
- **happy:** covered by default Codex dispatch, Sonnet override, preferred-fallback-halt routing, worktree bootstrap, sidecar dispatch persistence, ordered cherry-pick integration, and success criteria requiring empirical proof.
- **edge.zero-null-empty:** covered by missing implementer defaulting to Codex, literal `auto` rejected, empty Files block rejection, missing bootstrap symlink source halt, zero commits as fallback trigger, null `thread_id` allowance only after successful reconciliation, and no-commit implementation skip requiring explicit pre-dispatch declaration.
- **edge.boundary:** covered by 10-minute timeout, exactly one fallback implementer, one worktree per slice, bootstrap-before-dispatch, ordered integration, no merge commits, and wall-clock threshold for parallel smoke.
- **edge.large-input:** covered by exact file-set parser requirements, duplicate/path validation, stress-triggered tests for many slices/files, and lightweight sidecar dispatch entries.
- **edge.concurrent:** covered by isolated worktrees, isolated branches, bootstrapped dependencies before concurrent dispatch, mixed-implementer batch allowance, parallel-tool-call dispatch requirement, no concurrent writes to the integration branch, and deterministic ordered cherry-pick.
- **edge.adversarial:** covered by rejecting absolute paths, traversal, globs, directories, backslashes, malformed directives, non-conforming commit subjects, unknown existing worktree paths, invalid symlink bootstrap paths, and unreconciled dirty worktrees.
- **fail.dependency:** covered by Codex MCP error, Sonnet dispatch error, missing dependency artifacts, symlink creation failure, git worktree add/reset/cherry-pick/remove failure, sidecar persistence requirements, and provenance hook failure.
- **fail.malformed-input:** covered by parser defects for implementer directive and Files block, malformed bootstrap config, malformed parallel metadata halt, and subject regex enforcement.
- **fail.exception-path:** covered by named halt reasons for unrecoverable paths and explicit fallback behavior for dispatch failures.
- **integration.cross-module:** covered by required changes across autopilot orchestration, project config bootstrap loading, parser CLI helpers, sidecar schema, provenance hook, hook tests, worktree lifecycle, docs, and smoke tests.
- **stress.scale:** triggered because v0.7.0 parses plan sections, processes file sets, traverses commit ranges, symlinks dependency artifacts, and may dispatch multiple worktrees; implementation must include tests with many slices/files and deterministic overlap detection.
- **perf.slo:** not triggered; the 10-minute timeout and parallel-smoke wall-clock bound are operational failure boundaries, not product latency SLOs.
- **compat.breaking:** triggered because commit conventions change from trailer-required to subject-only; compatibility is covered by tests proving old trailer-bearing commits still pass and no-trailer commits now pass.
- **critical.residual-risk:** a paranoid senior would test crash recovery after some parallel slice branches have been cherry-picked and others have not; implementation must prove resume can avoid duplicate cherry-picks, identify already-integrated commits, and either continue safely or halt with actionable branch/worktree diagnostics.

## 15. Out Of Scope

Deferred to v0.7.1+:

- Usage-aware routing.
- Querying remaining quota.
- Cross-platform support beyond existing v0.6 constraints.
- Smart conflict resolution beyond halt-on-overlap / halt-on-cherry-pick-conflict.
- Auto-merge.
- Resuming partial Codex implementation work after failure.
- Continuing a Codex implementation thread across rounds.
- Reusing the persistent feature review thread for implementation.
- Multi-implementer coordination beyond worktree isolation.
- Parallel Phase A, Phase C, Phase D, or Phase E.
- Semantic dependency inference beyond consecutive slices and declared Files sets.
- Automatic dependency installation in worktrees.
- Build-cache coordination for write-heavy artifact directories such as Rust `target`.

## 16. Success Criteria

v0.7.0 is done when:

- No directive routes to Codex.
- `**Implementer:** codex` routes to Codex.
- `**Implementer:** sonnet` routes to Sonnet.
- Literal `**Implementer:** auto` halts with `implementer-directive-malformed`.
- Codex dispatch uses locked MCP parameters: `sandbox: workspace-write`, `approval-policy: never`, worktree `cwd`, `model: gpt-5.5`, `config: { model_reasoning_effort: high }`, and fresh `codex` thread.
- Every worktree is bootstrapped before implementer dispatch.
- Missing dependency artifact or failed symlink halts with `worktree-bootstrap-failed`.
- Preferred implementer failure resets the worktree to `slice_start_sha`.
- Fallback dispatch is fresh and does not reuse partial work.
- Halt with `implementer-unavailable` happens only after both implementers fail.
- `BLOCKED` and `NEEDS_CONTEXT` halt without fallback.
- Consecutive slices with non-overlapping `**Files:**` sets dispatch concurrently in separate worktrees.
- Mixed Codex/Sonnet parallel batches are allowed and tested.
- Overlapping `**Files:**` sets force serial execution.
- Missing or malformed Files metadata for selected parallel dispatch halts before dispatch.
- Successful parallel branches integrate through ordered cherry-pick onto the integration branch.
- Cherry-pick conflict halts with `worktree-merge-conflict`.
- Sidecar records every dispatch in `phases.implement.dispatches[]`.
- Provenance hook accepts conforming subject-only commits with or without the old trailer.
- Structural smoke covers Codex default, Sonnet override, malformed `auto`, fallback after zero commits, fallback after non-conforming commits, bootstrap failure, mixed-implementer parallel non-overlap, serial overlap, subject-only hook behavior, and cherry-pick conflict halt.
- Empirical parallelism smoke lives at `tests/smoke/implementer-routing-parallel.sh`.
- That smoke uses a 2-slice parallel batch where each slice performs a deterministic approximately 30-second implementation task, for example `sleep 25 && touch marker-N && git add marker-N && git commit -m "feat(slice:N): add marker N"`.
- The smoke measures wall-clock time from first dispatch to last reconciliation.
- Assertion: total wall-clock must be less than `1.5x` the measured single-slice baseline for the same deterministic task. This proves dispatch is closer to max-time than sum-time.
- The smoke fails if dispatches are serialized across separate turns.
