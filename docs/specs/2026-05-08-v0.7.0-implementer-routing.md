# v0.7.0 — Implementer Routing

**Status:** rev4 — codex dispatch via orchestrator-level background Bash (LocalShell migration)
**Date:** 2026-05-08
**Owner:** mkr
**Target version:** v0.7.2
**Validation:** critical
**Spec path:** `docs/specs/2026-05-08-v0.7.0-implementer-routing.md`

## 1. Goal

Extend autopilot's Phase B so implementation work can route to Sonnet subagents or Codex implementation dispatches, use domain-aware implementer policy, fall back only on real implementer failure, and run independent consecutive slices concurrently in isolated git worktrees.

## 2. Operating Principles

What changes:

- Phase B becomes a routing dispatch instead of a hard-coded Sonnet subagent dispatch.
- v0.7.1 adds domain-aware implementer selection through `agents/dispatchers.json`.
- Per-slice override via plan frontmatter `**Implementer:** sonnet | codex` is honored only when registry domain policy permits that implementer/domain pair.
- No directive uses the registry-preferred implementer for the resolved domain. If no preferred implementer exists for that domain, Claude falls back to Codex only if Codex is allowed for that domain.
- Literal `**Implementer:** auto` is malformed.
- Routing logic = preferred → fallback → halt. Reactive real errors/timeouts only; no scoring rubric.
- The orchestrator never gets stuck on routing. It falls back when one agent is unavailable and halts only with `implementer-unavailable` when both usable implementers fail or policy blocks the remaining fallback.
- Parallel implementation is allowed for consecutive slices with non-overlapping `**Files:**` sets.
- Mixed Codex/Sonnet parallel batches are allowed.
- Each parallel slice runs in its own bootstrapped worktree at `<repo>/.git-worktrees/slice-<N>`.
- Implementation dispatch uses plugin subagent definition files in `agents/`.
- Commit conventions drop the required `Co-Authored-By: Claude` trailer.
- Provenance hook validation becomes subject-only.

What stays the same:

- Phase A, Phase C, Phase D, and Phase E stay unchanged.
- The persistent feature-review Codex thread remains for review phases.
- Implementation dispatches do not use the persistent feature review thread.
- Sidecar is still the durable source of truth.
- Git reconciliation still verifies autopilot-owned commits before advancing state.
- `BLOCKED` and `NEEDS_CONTEXT` remain real blockers, not routing failures.
- Runtime safety checks remain mechanical Node code.

## 3. Decision Boundary

Claude, acting as the autopilot orchestrator, makes semantic routing and batching decisions by reading the implementation plan and following `skills/autopilot/SKILL.md` prose.

Node code handles mechanical state only:

- Dispatcher registry loading and consistency validation.
- Worktree creation.
- Symlink bootstrap.
- Bootstrap verification.
- Sidecar persistence.
- Git reconciliation.
- Ordered cherry-pick integration.
- Patch-id resume detection.
- Provenance hook regex.
- Worktree cleanup.

No parser CLI encodes implementer routing decisions. No Node router decides preferred → fallback → halt. No Node conflict detector decides parallel-vs-serial batches.

This boundary is intentional:

- Plan semantics are read by Claude.
- Domain inference is performed by Claude.
- Registry lookup enforces policy for a chosen `(implementer, domain)` pair.
- Mechanical state validity is verified by code.
- Subagent final JSON is advisory.
- Git state and reconciler output are authoritative.

## 3.5. Domain Semantics

v0.7.1 introduces slice domains.

Allowed domain values:

- `ui`
- `ai-harness`
- `backend`
- `general`

Optional slice directive:

```markdown
**Domain:** backend
```

Directive rules:

- Trim surrounding whitespace.
- Lower-case exact match only.
- Unknown value halts with a parser-defect-style halt, `domain-directive-malformed`.
- Missing directive triggers inference.

Precedence:

```text
explicit **Domain:** directive
  -> path-based inference from **Files:**
  -> general default
```

Path-based inference heuristics when `**Domain:**` is absent:

- `ui` if `**Files:**` includes paths such as:
  - `web/`
  - `app/`
  - `frontend/`
  - `*.tsx`
  - `*.css`
  - `*.html`
- `ai-harness` if `**Files:**` includes paths such as:
  - `skills/`
  - `agents/`
  - `hooks/`
  - `lib/codex-bridge/`
  - `*.skill.md`
- `backend` if `**Files:**` paths look like server/data/runtime code and do not match stronger `ui` or `ai-harness` signals.
- `general` when no confident inference exists.

Domain inference is performed by Claude following `SKILL.md` prose. The dispatcher registry does not implement heuristics.

Ambiguity rule:

- If inference cannot confidently choose between plausible domains, Claude may default to `general` only when the selected implementer is allowed for every plausible domain.
- If inference is ambiguous and the chosen implementer is forbidden in any plausible domain, halt `domain-policy-ambiguous`.
- The user must add an explicit `**Domain:**` directive to continue.

Examples:

- A slice touching `web/components/Button.tsx` infers `ui`.
- A slice touching `agents/slice-implementer-codex.md` infers `ai-harness`.
- A slice touching `lib/server/db.ts` infers `backend`.
- A slice touching only `README.md` infers `general`.

## 4. Routing Model

v0.7.1 changes v0.7.0 override semantics.

v0.7.0 rule:

```text
**Implementer:** directive honored unconditionally.
```

v0.7.1 rule:

```text
**Implementer:** directive honored only if the dispatcher registry permits
the (implementer, domain) pair.
```

If `**Implementer:** codex` is given for a slice with `**Domain:** ui`, the registry reports `forbidden` and autopilot halts with `domain-policy-violation`. The user must change either the directive or the slice's domain.

Migration note:

- v0.6.0 / v0.7.0 plans without `**Domain:**` directives infer to `general` or to a domain via path heuristics.
- Existing plans that explicitly set `**Implementer:** codex` for slices that infer to `ui` will halt under v0.7.1.
- Fix by adding `**Domain:** backend` if the slice is genuinely backend, or by removing/changing the implementer directive.

Preferred implementer resolution:

| Slice directive / state | Preferred implementer |
|---|---|
| `**Implementer:** codex` and registry allows/prefers Codex for domain | Codex |
| `**Implementer:** sonnet` and registry allows/prefers Sonnet for domain | Sonnet |
| no directive | registry `preferred` implementer for resolved domain |
| no directive and no preferred implementer | Codex if allowed for resolved domain; otherwise first allowed implementer |
| `**Implementer:** auto` | malformed |
| empty value | malformed |
| mixed case value | malformed |
| any other value | malformed |
| directive selects implementer forbidden for domain | halt `domain-policy-violation` |

Routing sequence:

```text
resolve domain
  -> resolve implementer via directive/default + registry policy
  -> preferred implementer dispatch
  -> fallback implementer on implementation-dispatch failure if fallback is allowed for domain
  -> halt with implementer-unavailable only if both usable implementers fail or policy blocks the remaining fallback
```

Failure-triggers-fallback definition:

1. Codex process error.
2. Subagent dispatch error.
3. 10-minute timeout.
4. Zero commits produced.
5. Non-conforming commits emitted.
6. Missing or malformed required final-message JSON block, unless the failure is already a clear `BLOCKED` or `NEEDS_CONTEXT` signal.

NOT a fallback trigger:

- Agent reports `BLOCKED`.
- Agent reports `NEEDS_CONTEXT`.

Those are real blockers, such as "the spec is unclear." Autopilot halts with the existing blocker failure modes rather than trying the other implementer.

Codex-specific blocker halt reasons:

- `codex-blocked`
- `codex-needs-context`

Sonnet-specific blocker halt reasons:

- `subagent-blocked`
- `subagent-needs-context`

Fallback policy distinction:

- `domain-policy-violation` means the user explicitly selected a forbidden implementer/domain pair.
- `implementer-unavailable` means the preferred implementer failed and policy blocked the only remaining fallback, or all allowed fallback attempts failed.

## 5. Pre-Dispatch Checklist

The checklist lives in `skills/autopilot/SKILL.md`; this spec defines its required behavior.

Before any Phase B dispatch, Claude must inspect the slice section directly.

Implementer directive rules:

- `**Implementer:**` absent → choose implementer using resolved domain + registry preference.
- Exactly `**Implementer:** codex` → Codex if registry permits Codex for domain.
- Exactly `**Implementer:** sonnet` → Sonnet if registry permits Sonnet for domain.
- `auto`, empty value, mixed case, or anything else → halt `implementer-directive-malformed`.
- Implementer forbidden for resolved domain → halt `domain-policy-violation`.

Domain directive rules:

- `**Domain:**` absent → infer from `**Files:**` using §3.5.
- Exactly `ui`, `ai-harness`, `backend`, or `general` → valid.
- Empty value, mixed case, or anything else → halt `domain-directive-malformed`.
- Ambiguous inferred domain where chosen implementer may be forbidden → halt `domain-policy-ambiguous`.

Parallel candidate `**Files:**` rules:

- Parallel candidate missing `**Files:**` → halt `parallel-files-missing`.
- Empty `**Files:**` block → halt `parallel-files-malformed`.
- Inline form like `**Files:** lib/foo.js` → halt `parallel-files-malformed`.
- Glob path → halt `parallel-files-malformed`.
- Traversal path using `.` or `..` segment → halt `parallel-files-malformed`.
- Absolute path → halt `parallel-files-malformed`.
- Backslash separator → halt `parallel-files-malformed`.
- Duplicate path inside a slice → halt `parallel-files-malformed`.
- Directory-only path with trailing `/` → halt `parallel-files-malformed`.

Candidate-window rule:

- Any halt in the current candidate window means no worktree setup.
- Halt before creating or bootstrapping any worktree for that window.
- Claude must surface the exact halt reason and offending slice/path/value.

Conflict rule:

- Claude compares exact normalized repo-relative paths across consecutive candidate slices.
- Any overlap forces serial execution for those slices.
- Conflict detection is based only on `**Files:**` sets, not implementer choice.
- Mixed Codex/Sonnet parallel batches are allowed when Files sets do not overlap.

## 6. Implementation Dispatch Model

v0.7.2 splits implementer dispatch into two transport-specific paths. Each implementer's transport determines whether it runs as a Claude Code subagent or as an orchestrator-spawned shell process.

### Sonnet path: Claude Code subagent

`agents/slice-implementer-sonnet.md` — required frontmatter:

```markdown
---
name: slice-implementer-sonnet
description: Implements one autopilot slice directly with Sonnet tools from an isolated worktree.
tools: Read, Edit, Write, Bash
model: sonnet
---
```

Behavior:

- Runs as a Claude Code subagent dispatched via the `Task` tool.
- Receives slice section text, worktree absolute path, `slice_start_sha`, and Commit Conventions.
- Implements directly using allowed tools.
- Follows TDD for the slice.
- Ends with the required final-message JSON block.

### Codex path: orchestrator-level background Bash (LocalShell pattern)

v0.7.2 changes the v0.7.1 model. Codex is no longer dispatched via a Claude Code subagent. The orchestrator (Claude in autopilot) invokes `codex exec` directly via the `Bash` tool with `run_in_background: true`. This pattern mirrors Claude Code's `LocalShellTask` (per `src/tasks/LocalShellTask/` in the runtime source).

Rationale:

- v0.7.1 wrapped codex dispatch in a Sonnet subagent whose only job was to shell out to `codex exec`. The wrapper added no behavior of its own and put a 10-minute synchronous Bash timeout cap on the codex run.
- Background Bash has no synchronous timeout — only a configurable runtime bound (see §6.6) — so long Codex implementations are supported.
- Skipping the subagent layer eliminates the redundant indirection and matches the runtime's documented long-running shell pattern.

The codex implementer contract lives at `docs/codex-implementer-contract.md`. It documents the locked invocation, prompt template, status file format, and JSON status block. The contract is referenced by the dispatcher registry (`contract` field) but is NOT a Claude Code subagent file — it lives in `docs/`, not `agents/`.

Locked codex invocation (orchestrator runs this via Bash with `run_in_background: true`):

```bash
scripts/codex-exec-with-status.sh \
  <status-file-path> \
  -- \
  codex exec \
    --skip-git-repo-check \
    -s workspace-write \
    -C <worktree-absolute-path> \
    -m gpt-5.5 \
    -c model_reasoning_effort=high \
    "<implementation prompt>" \
  </dev/null
```

The `scripts/codex-exec-with-status.sh` wrapper executes codex, captures the exit code, and writes a status file `{exit_code, completed_at, signal?, ...}` to a known path on exit. This is durable evidence that survives orchestrator session termination — Claude Code's Bash task registry is session-local; the status file is on disk.

Every flag is mandatory:

- `--skip-git-repo-check` because worktree git detection can be unreliable in unattended Codex runs.
- `-s workspace-write` for write access scoped to the worktree. The `workspace-write` sandbox is sufficient for unattended in-sandbox commit/file-edit operations without surfacing approval prompts; no separate `--approval-policy` flag is needed.
- `-C <worktree>` to pin cwd.
- `-m gpt-5.5` for the locked model.
- `-c model_reasoning_effort=high` for locked reasoning effort.
- `</dev/null` redirect prevents codex from inheriting the parent shell's stdin (otherwise it hangs reading from stdin under bash backgrounding).

Do not use `--dangerously-bypass-approvals-and-sandbox` — it bypasses the sandbox and would allow Codex to escape the worktree.

### Shared dispatch input

Every implementation dispatch (subagent or background Bash) includes:

### Shared dispatch input

Every implementation dispatch (subagent or background Bash) includes (continued):

- Slice id.
- Full slice section text from the plan.
- Resolved domain.
- Selected implementer.
- Worktree absolute path.
- `slice_start_sha`.
- Phase A validation coverage, when available.
- Commit Conventions.
- Required test/verification commands.
- Instruction that no trailer is required.
- Instruction to leave all changes committed before reporting `DONE`.

For the codex path, this input is composed into the implementation prompt passed as the final argument to `codex exec`. For the sonnet path, it is the user-message of the subagent dispatch.

### Defensive Base Check

Both subagents must verify before implementation:

```bash
git -C <worktree> rev-parse HEAD
```

If the result differs from `slice_start_sha`, return `BLOCKED` with both SHAs in concerns.

### Diff-Size Sanity Check

Both subagents must inspect diff size before reporting `DONE`:

```bash
git -C <worktree> diff --shortstat <slice_start_sha>..HEAD
```

If the diff exceeds 2000 lines or 10 files for a slice whose plan section does not justify it, return `NEEDS_CONTEXT`.

### Final-Message JSON Contract

Each implementation subagent must end its final response with a fenced JSON block:

```json
{
  "status": "DONE",
  "concerns": []
}
```

Allowed statuses:

- `DONE`
- `BLOCKED`
- `NEEDS_CONTEXT`

Optional fields may be included for diagnostics, but the orchestrator must not trust them for git truth.

Malformed or missing JSON is an implementation-dispatch failure and triggers fallback unless the final message is an unambiguous blocker that the orchestrator records as `BLOCKED` or `NEEDS_CONTEXT`.

## 6.5. Dispatcher Registry

Dispatcher registry location:

```text
agents/dispatchers.json
```

v0.7.2 schema is **transport-aware**: each entry's required fields depend on its `transport` value.

- `transport: claude-subagent` → entry requires `agent` (subagent frontmatter name); MUST NOT include `contract`.
- `transport: codex-background-bash` → entry requires `contract` (path to a docs file describing the codex contract); MUST NOT include `agent`.

Common fields for all transports:

- `transport`: one of `claude-subagent`, `codex-background-bash`.
- `tools`: exact tool allowlist for the transport. For subagent transports, must match the agent file's frontmatter `tools` exactly. For background-bash transports, must equal `["Bash"]` (the only tool the orchestrator uses to dispatch).
- `domains`: object mapping each of `ui` / `ai-harness` / `backend` / `general` → `forbidden | allowed | preferred`.

v0.7.2 registry:

```json
{
  "codex": {
    "transport": "codex-background-bash",
    "contract": "docs/codex-implementer-contract.md",
    "tools": ["Bash"],
    "domains": {
      "ui": "forbidden",
      "ai-harness": "forbidden",
      "backend": "preferred",
      "general": "allowed"
    }
  },
  "sonnet": {
    "transport": "claude-subagent",
    "agent": "slice-implementer-sonnet",
    "tools": ["Read", "Edit", "Write", "Bash"],
    "domains": {
      "ui": "preferred",
      "ai-harness": "preferred",
      "backend": "allowed",
      "general": "preferred"
    }
  }
}
```

Loader:

```text
lib/codex-bridge/dispatchers.js
```

Exports exactly:

```js
getDispatcher(implementer) -> entry | throws
enforceDomainPolicy(implementer, domain) -> "forbidden" | "allowed" | "preferred"
```

Hard constraints:

- Loader does not infer domain.
- Loader does not pick implementer.
- Loader does not dispatch.
- Loader does not inspect slice files.
- Loader does not implement routing order.
- Loader is pure config validation + lookup.

Load-time validation:

- Every registry entry has `transport`, `tools`, and `domains`.
- `transport` is exactly `claude-subagent` or `codex-background-bash`.
- For `transport: claude-subagent`: entry has `agent` (string), does NOT have `contract`.
- For `transport: codex-background-bash`: entry has `contract` (string), does NOT have `agent`.
- Every domain value is present for `ui`, `ai-harness`, `backend`, and `general`.
- Every domain policy is exactly `forbidden`, `allowed`, or `preferred`.
- For `transport: claude-subagent`: matching agent file exists at `agents/<agent>.md`; frontmatter `name` matches registry `agent`; frontmatter `tools` exactly matches registry `tools`.
- For `transport: codex-background-bash`: contract file exists at the path declared in `contract`.
- No registry entry points at the deprecated unified `agents/slice-implementer.md` or at `agents/slice-implementer-codex.md` (v0.7.2 moved codex contract out of `agents/`).
- Drift throws at load time.

## 6.6. Codex Implementer Contract

The codex contract is a markdown document at `docs/codex-implementer-contract.md`. It is NOT a Claude Code subagent file. It documents:

1. The locked `codex exec` invocation (with `scripts/codex-exec-with-status.sh` wrapper).
2. The implementation prompt template fields the orchestrator must include.
3. The Codex Commit Conventions (subject-only, no trailer).
4. The status file schema written by the wrapper:

```json
{
  "exit_code": 0,
  "completed_at": "2026-05-09T12:34:56.000Z",
  "started_at": "2026-05-09T12:30:01.000Z",
  "signal": null
}
```

5. The output file path convention: orchestrator generates a unique log path per dispatch (e.g., `~/Library/Application Support/Inkling/diagnostics/codex-dispatch/<slice-id>-<timestamp>.log`) and passes it through to Bash via the standard background output capture.
6. Maximum runtime semantics (see §6.7).

The contract document is read by Claude (orchestrator) as reference material. It is not invoked as a subagent.

## 6.7. Codex Background Dispatch — Runtime Bounds

Per-project configuration in `.codex-paired/project.json`:

```json
{
  "codex_dispatch": {
    "max_runtime_ms": 7200000,
    "log_max_bytes": 1048576
  }
}
```

Defaults:

- `max_runtime_ms`: 7200000 (2 hours).
- `log_max_bytes`: 1048576 (1 MB).

Behavior:

- Background codex tasks that exceed `max_runtime_ms` are killed by the orchestrator (best-effort SIGTERM, then SIGKILL after 5s grace). Halt with `codex-background-timeout`.
- Output logs exceeding `log_max_bytes` are tail-truncated in sidecar summaries (the full output file remains on disk for forensic inspection; only the sidecar reference is bounded).

Cleanup on user halt:

- Orchestrator must attempt to kill in-progress background codex tasks before exiting. Failed kills are recorded in sidecar with halt-reason annotation.

## 7. Reconciler Is Truth

Subagent JSON is advisory.

The authoritative implementation result comes from a shared Node reconciler module.

Required reconciler behavior:

```js
reconcileWorktree({
  worktreePath,
  sliceStartSha,
  sliceId
}) -> {
  commits,
  head_sha,
  commit_count,
  non_conforming_subjects
}
```

The reconciler reads git state from the worktree:

```bash
git -C <worktreePath> log <slice_start_sha>..HEAD
```

It verifies each commit subject against:

```text
^(feat|test|fix|docs|refactor|chore)\(slice:<N>\): <description>
```

Rules:

- `head_sha` comes from git.
- `commit_count` comes from git.
- `commits` come from git.
- `non_conforming_subjects` comes from git.
- The subagent cannot self-report commits as authoritative.
- `DONE` with zero commits triggers fallback.
- `DONE` with non-conforming commits triggers fallback.
- `DONE` with reconciler failure triggers fallback.
- `BLOCKED` / `NEEDS_CONTEXT` halts without fallback.
- Missing/malformed JSON triggers fallback only if there is no clear blocker signal.

## 8. Worktree Bootstrap

Before any implementation subagent dispatch, the worktree must be bootstrapped.

v0.7.1 uses dependency symlinking, not install commands.

Default symlink candidates:

- `node_modules`
- `.venv`
- `venv`

Default candidates are `required: false`:

- If present in the integration checkout, symlink them into the worktree.
- If absent, skip silently.

Configured symlinks are `required: true`:

```json
{
  "worktree_bootstrap": {
    "symlinks": ["node_modules", "custom_dir"]
  }
}
```

Rules:

- User-configured entries must exist.
- If user explicitly lists `node_modules`, it is required.
- Symlink paths must be repo-relative.
- Absolute paths are invalid.
- Traversal is invalid.
- Symlink target must not already exist unless it is already the correct symlink.
- Bootstrap failure halts with `worktree-bootstrap-failed`.

Two-tier bootstrap gate before dispatch:

1. Sidecar must have `phases.implement.bootstrap.completed_at`.
2. `verifyBootstrap(worktreePath, recordedSymlinks)` must verify:
   - each recorded path is a symlink;
   - `readlink` resolves to expected `<repoRoot>/<symlink>`;
   - missing/non-symlink/wrong-target paths halt.

Tier 1 missing:

- Halt `worktree-bootstrap-failed`.

Tier 2 failure:

- Halt `worktree-bootstrap-stale`.
- Diagnostics include `{symlink, expected, actual}`.

v0.7.1 does not run dependency install commands automatically.

Out of scope:

- `npm install`
- `pnpm install`
- `pip install`
- `cargo fetch`
- write-heavy shared build cache coordination such as Rust `target`

## 9. Mid-Implementation Failure Recovery

Every Phase B attempt runs in a worktree rooted at `slice_start_sha`.

On failure that triggers fallback:

1. Record the failed dispatch in sidecar `dispatches[]`.
2. Mark outcome:
   - `failed-fallback-pending` if the other implementer remains.
   - `failed-halted` if no fallback remains.
3. Reset the worktree:

```bash
git reset --hard <slice_start_sha>
```

4. Re-run worktree bootstrap.
5. Re-run bootstrap verification.
6. Dispatch the fallback subagent fresh if registry policy allows the fallback for the slice domain.
7. Reconcile only the fallback's commits.

Do not salvage partial work. Do not reuse a partial Codex implementation process. Do not ask fallback to continue prior commits.

If reset fails:

- Halt `worktree-reset-failed`.

If bootstrap after reset fails:

- Halt `worktree-bootstrap-failed`.

If bootstrap verification after reset fails:

- Halt `worktree-bootstrap-stale`.

If fallback is forbidden for the domain after preferred failure:

- Halt `implementer-unavailable`.

## 10. Parallel Slice Execution

Parallelism applies only to Phase B implementation.

Phase A must already be double-SHIP'd for every slice in the parallel candidate window. Phase C, Phase D, Phase E, and slice shipment still run in slice order after implementation commits are integrated.

Parallel candidate rule:

- Multiple consecutive Phase B-ready slices may dispatch concurrently when each has valid `**Files:**` metadata and file sets do not overlap.
- Mixed implementer batches are allowed.
- Example: slice 3 using Codex and slice 4 using Sonnet may dispatch concurrently if their `**Files:**` sets do not overlap and registry domain policy permits both.
- Conflict detection runs on file sets only, not implementer choice.
- Any Files malformed/missing halt in the candidate window stops all worktree setup for that window.
- Any domain-policy halt in the candidate window stops all worktree setup for that window.

Parallel setup behavior:

1. Claude reads the candidate slice sections.
2. Claude resolves each slice domain.
3. Claude validates implementer directives and Files blocks with the checklist in §5.
4. Claude enforces dispatcher registry domain policy.
5. Claude compares Files sets.
6. Claude creates one worktree per selected slice at `<repo>/.git-worktrees/slice-<N>`.
7. Claude branches each worktree from the same `slice_start_sha`.
8. Branch name: `<slice-id>-impl`, for example `slice-3-impl`.
9. Claude bootstraps each worktree.
10. Claude verifies bootstrap state.

Parallel dispatch behavior:

- Dispatch all candidate slices concurrently.
- Each dispatch uses its own worktree as `cwd`.
- The autopilot orchestrator achieves concurrency by issuing all parallel-batch subagent dispatch tool calls in a single assistant turn using Claude's parallel-tool-call mechanism.
- Serial `await` across separate turns is not concurrent dispatch and is non-conforming.
- Each dispatch reconciles commits on its own branch after the subagent returns.

Integration strategy:

- Ordered cherry-pick from the slice's worktree branch onto the integration branch.
- No merge commits.
- Rationale: merge commit subjects like `Merge branch 'slice-3-impl'` would violate the subject-only provenance hook.
- Integrate in ascending slice order.
- Cherry-pick commits from `slice_start_sha..slice-N-impl`.
- On cherry-pick conflict, abort cherry-pick and halt with `worktree-merge-conflict`.

## 11. Worktree Lifecycle

Create worktree at dispatch:

```bash
mkdir -p <repo>/.git-worktrees
git worktree add -b <slice-id>-impl <repo>/.git-worktrees/slice-<N> <slice_start_sha>
```

Then bootstrap the worktree:

```bash
ln -s <repo>/<artifact-path> <repo>/.git-worktrees/slice-<N>/<artifact-path>
```

The orchestrator owns lifecycle:

- Create at dispatch.
- Bootstrap before implementer runs.
- Verify bootstrap before implementer runs.
- Reset and re-bootstrap on fallback.
- Cherry-pick successful commits to the integration branch.
- Remove clean worktrees after successful integration.
- Leave diagnostic worktrees on halt if they contain dirty state, conflicts, or unreconciled commits.

Success path:

1. Reconcile worktree commits.
2. Record `head_sha` and `commit_count`.
3. Ordered cherry-pick onto integration branch.
4. Update `last_commit_sha = HEAD`.
5. Mark implement phase shipped.
6. Remove worktree.
7. Delete temp branch only after commits are reachable from integration branch.

Halt path:

- Clean failed worktrees may be removed after diagnostics are persisted.
- Dirty/conflicted/unreconciled worktrees must be left in place.
- Halt summary must include absolute worktree path and branch name.

Worktree safety:

- If `<repo>/.git-worktrees/slice-<N>` already exists and is not a valid clean worktree for the same slice, halt with `worktree-path-conflict`.
- If `.git-worktrees/` is not gitignored, halt with `worktree-gitignore-missing` before creating worktrees.
- If dependency symlink bootstrap fails, halt with `worktree-bootstrap-failed`.
- If bootstrap verification fails, halt with `worktree-bootstrap-stale`.

## 12. Patch-ID Resume And Integration Recovery

Ordered cherry-pick integration must be resume-safe.

For each source commit on a slice branch, compute:

```bash
git diff <sha>~..<sha> | git patch-id --stable
```

Resume detection scans the integration branch history backward for ordered `(patch_id, subject)` tuples.

Rules:

- All source commits match in order → slice branch already integrated; skip cherry-pick.
- None match → proceed with cherry-pick.
- Partial match → halt `worktree-resume-ambiguous`.
- Order broken → halt `worktree-resume-ambiguous`.
- Same subject but different patch-id → treat as not integrated; cherry-pick proceeds and may conflict.
- Merge commits in the candidate scan window are ignored for patch-id matching.
- If a merge commit makes equivalence ambiguous, halt `worktree-resume-ambiguous`.
- Empty source range after a supposedly shipped dispatch halts `worktree-integration-empty`.

`worktree-resume-ambiguous` diagnostics include:

```json
{
  "slice_id": "slice-3",
  "branch_name": "slice-3-impl",
  "integrated_subjects": [],
  "missing_subjects": [],
  "integration_branch_head": "abc123"
}
```

Cherry-pick conflict:

- Run `git cherry-pick --abort`.
- Halt `worktree-merge-conflict`.
- Leave branch/worktree for inspection.


Allowed values:

Rules:

- Trim surrounding whitespace.

```markdown

- lib/codex-bridge/worktree.js

```

`**Files:**` is required for any slice intending parallel dispatch.

- Block ends at blank line, heading, or next bold directive.

- Directories with trailing `/` are malformed.
- Absolute paths are malformed.
- Traversal paths are malformed.
- Backslash separators are malformed.

Required dispatch fields:

Dispatch entries are append-only.

If fallback ships, preserve both dispatches:

1. Failed preferred dispatch.
2. Shipped fallback dispatch.

Bootstrap metadata is persisted because dependency setup is part of the Phase B execution boundary.

Domain and implementer selection metadata are persisted because domain policy now affects replay and halt diagnostics.

## 15. Updated Commit Conventions

New convention is subject-only:

```text
(feat|test|fix|docs|refactor|chore)(slice:N): <description>
```

Examples:

```text
feat(slice:3): add implementer routing worktree support
test(slice:3): cover bootstrap stale symlink detection
fix(slice:3): recover from empty integration range
docs(slice:3): document subject-only provenance hook
```

Trailer requirement is dropped entirely.

Rules:

- `Co-Authored-By: Claude` trailer is not required.
- Presence of trailer does not break compliance.
- Absence of trailer is fine.
- Existing repo commits with the old trailer still pass.
- Commit body is ignored by the provenance hook.
- Subject must match current slice number.

This applies to:

- Codex implementation dispatches.
- Sonnet implementation dispatches.
- Phase C fix subagents.
- Phase D docs commits.
- Phase E fix subagents.

Zero commits after Phase B implementation is a fallback trigger unless the slice explicitly skips implementation before dispatch.

## 16. Updated Provenance Hook

`hooks/check-commit-provenance.sh` keeps active-anchor discovery but simplifies validation.

During active autopilot:

1. Read `<repo-root>/.codex-paired/active.json`.
2. Resolve the spec path.
3. Load the sidecar.
4. Read `autopilot.current_slice`.
5. Read most recent commit subject.
6. Check subject only:

```bash
^(feat|test|fix|docs|refactor|chore)\(slice:${CURRENT_SLICE}\):
```

7. Do not check trailers.
8. Do not check commit body.

Required hook tests:

- No anchor allows arbitrary commit.
- Conforming subject with old trailer passes.
- Conforming subject without trailer passes.
- Wrong slice subject blocks.
- Arbitrary manual subject blocks.
- Subdirectory commit resolves repo root correctly.
- Stdin filter still only evaluates Bash `git commit`.

## 17. Failure Modes

| Condition | Halt reason | Behavior |
|---|---|---|
| Bad `**Implementer:**` value | `implementer-directive-malformed` | Halt before dispatch. |
| Literal `**Implementer:** auto` | `implementer-directive-malformed` | Halt before dispatch. |
| Empty `**Implementer:**` value | `implementer-directive-malformed` | Halt before dispatch. |
| Mixed-case implementer value | `implementer-directive-malformed` | Halt before dispatch. |
| Bad `**Domain:**` value | `domain-directive-malformed` | Halt before dispatch. |
| Empty `**Domain:**` value | `domain-directive-malformed` | Halt before dispatch. |
| Mixed-case domain value | `domain-directive-malformed` | Halt before dispatch. |
| Directive selects implementer forbidden for slice domain | `domain-policy-violation` | Halt before worktree setup; user fixes plan. |
| Domain inference ambiguous and chosen implementer is forbidden in a plausible domain | `domain-policy-ambiguous` | Halt before worktree setup; user adds `**Domain:**`. |
| Registry missing implementer | `implementer-directive-malformed` | Halt before dispatch. |
| Registry load/consistency failure | `dispatcher-registry-malformed` | Halt before dispatch. |
| Worktree dependency bootstrap fails | `worktree-bootstrap-failed` | Halt before implementer dispatch. |
| Worktree bootstrap sidecar marker missing | `worktree-bootstrap-failed` | Halt before implementer dispatch. |
| Worktree bootstrap symlink stale/wrong/missing | `worktree-bootstrap-stale` | Halt before implementer dispatch. |
| Codex process error (nonzero exit per status_file) | fallback trigger | Reset worktree, re-bootstrap, try Sonnet if domain policy allows. |
| Codex background task lost (no Bash task entry AND no status_file with terminal evidence) | `codex-background-task-lost` | Halt with output_file path for forensics. Cannot infer whether codex completed; user must investigate. |
| Codex background task exceeds `max_runtime_ms` | `codex-background-timeout` | Orchestrator kills the task (SIGTERM then SIGKILL after 5s); records partial logs; treats as fallback trigger. |
| Sonnet dispatch error | fallback trigger | Reset worktree, re-bootstrap, try Codex if domain policy allows. |
| 10-minute implementation timeout | fallback trigger | Reset worktree, re-bootstrap, try fallback if policy allows. |
| Missing/malformed subagent JSON with no clear blocker | fallback trigger | Reset worktree, re-bootstrap, try fallback if policy allows. |
| Zero commits produced | fallback trigger | Reset worktree, re-bootstrap, try fallback if policy allows. |
| Non-conforming commits emitted | fallback trigger | Cite SHA; reset worktree; re-bootstrap; try fallback if policy allows. |
| Preferred and fallback both fail | `implementer-unavailable` | Halt; record both dispatch failures. |
| Preferred fails and only fallback is forbidden by policy | `implementer-unavailable` | Halt; policy blocked fallback after preferred failure. |
| Codex returns `BLOCKED` | `codex-blocked` | Halt; no fallback. |
| Codex returns `NEEDS_CONTEXT` | `codex-needs-context` | Halt; no fallback. |
| Sonnet returns `BLOCKED` | `subagent-blocked` | Halt; no fallback. |
| Sonnet returns `NEEDS_CONTEXT` | `subagent-needs-context` | Halt; no fallback. |
| Missing Files block on selected parallel slice | `parallel-files-missing` | Halt before worktree setup. |
| Malformed Files block | `parallel-files-malformed` | Halt before worktree setup. |
| Overlapping Files sets | none | Force serial execution. |
| Worktree path conflict | `worktree-path-conflict` | Halt; do not delete unknown files. |
| Worktree not gitignored | `worktree-gitignore-missing` | Halt before worktree creation. |
| Worktree creation fails | `worktree-create-failed` | Halt with stderr and intended path. |
| Reset before fallback fails | `worktree-reset-failed` | Halt; leave worktree. |
| Ordered cherry-pick conflicts | `worktree-merge-conflict` | Abort cherry-pick; halt; leave branch/worktree. |
| Resume detects partial/order-broken integration | `worktree-resume-ambiguous` | Halt with branch/worktree diagnostics. |
| Source range empty after shipped dispatch | `worktree-integration-empty` | Halt; broken invariant. |
| Worktree cleanup fails | `worktree-cleanup-failed` | Record warning; include cleanup path. |
| Branch cleanup fails | `worktree-branch-cleanup-failed` | Record warning; continue if integrated. |
| Resume sees history divergence | `history-divergence` | Existing behavior. |
| Resume sees non-conforming integrated commit | `external-commit-detected` | Existing behavior, subject-only. |

`domain-policy-violation` versus `implementer-unavailable`:

- `domain-policy-violation` means the user explicitly picked a forbidden implementer/domain pair.
- `implementer-unavailable` means the user did not pick the forbidden fallback; policy blocked fallback after the preferred implementer failed.
