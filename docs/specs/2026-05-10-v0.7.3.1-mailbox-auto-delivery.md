## 1. Goal

Close the auto-delivery gap left open by v0.7.3. v0.7.3 mailbox proved coordination works (orch ↔ slice, slice ↔ slice empirically validated). This spec eliminates the need for agents to remember to poll, while honoring the constraint that we don't own codex's runtime.

Two surgical additions to the v0.7.3 mailbox:

1. **Pre-injection at dispatch** (both transports) — orchestrator reads recipient's unread inbox before invoking codex/Task and prepends messages to the dispatch prompt as `<codex-paired-pending-messages>` blocks.
2. **PostToolUse hook for Sonnet path** — registered hook fires on every tool-use boundary inside Task subagents, reads the slice's mailbox, returns unread messages as `additionalContext`. Claude Code injects this into the next assistant turn. **This is genuine auto-injection for the Sonnet path** — equivalent in effect to Claude Code's native `<teammate-message>` delivery.

Codex path stays cooperative (subprocess opacity is genuine; no MCP push exists in protocol). Codex dispatch prompt instructs polling at five named semantic checkpoints — explicitly NOT pre-edit (avoids ritual polling).

## 2. Architectural Finding

Claude Code's agent-team auto-injection (`useMailboxBridge` + `queuePendingMessage` + `drainPendingMessages` + `<teammate-message>` XML in `src/utils/teammateMailbox.ts`) is a runtime feature. Plugins running on top of Claude Code's runtime do not get hooks between assistant turns.

However, **Claude Code's hook system reaches inside Task subagents AND exposes subagent identity in hook input.** Evidence:

- `services/tools/toolHooks.ts:39 runPostToolUseHooks` is the universal tool execution path.
- `tools/AgentTool/runAgent.ts` invokes `query()`, which routes tool calls through the same `toolExecution.ts` path as top-level tool calls.
- `utils/hooks.ts:2783-2788` shows hook output `additionalContext` becomes an attachment message injected into the next assistant turn (`additionalContexts: [result.additionalContext]`).
- `entrypoints/sdk/coreSchemas.ts:387 BaseHookInputSchema` shows hook stdin includes `cwd` and `agent_id` (the latter "present only when the hook fires from within a subagent").
- The plugin's existing `hooks/check-commit-provenance.sh` is designed to fire on subagent commits during autopilot — its functional design implies hooks fire inside subagents.

Therefore, a PostToolUse hook IS the auto-injection mechanism for Sonnet subagents, and `cwd` + `agent_id` from hook stdin provide reliable per-subagent identity (no env var needed).

For codex (`codex exec` subprocess), no equivalent hook surface exists. The MCP protocol has no spec-level support for unsolicited server-initiated push into a model's reasoning context. Mid-run injection for codex is genuinely architecturally impossible from outside the runtime; we accept this and use cooperative checkpoints.

## 3. Decision Boundary

Unchanged from v0.7.3. Claude orchestrates; Node verifies mechanical state.

New mechanical-state additions:

- Sidecar `injected_message_ids` tracking per dispatch entry (Node).
- Mark-read-batch CLI subcommand for hook efficiency (Node).
- PostToolUse hook script (Bash + Node for JSON formatting).

Claude still owns: dispatch sequencing, when to send messages, how to interpret received messages.

## 4. Pre-Injection at Dispatch

### 4.1 Flow (both transports)

Before orchestrator invokes a slice agent (codex or Task):

1. Read recipient's unread inbox:
   ```bash
   node lib/codex-bridge/cli.js mailbox-read \
     --for slice-N --actor orchestrator --unread
   ```
2. If non-empty, format as `<codex-paired-pending-messages>` block (see §4.2) and prepend to dispatch prompt body.
3. Capture array of message ids → write to sidecar dispatch entry as `injected_message_ids`.
4. Invoke codex/Task with the modified prompt.
5. **Do NOT mark read at this point.** Mark read only after terminal dispatch result (§4.4).

### 4.2 Injected block format

```xml
<codex-paired-pending-messages recipient="slice-N">
  <pending-message id="msg-2026-05-10T12:34:56.789Z-0001" from="orchestrator" timestamp="2026-05-10T12:34:56.789Z">
  user clarified that the auth token MUST be HMAC-SHA256, not SHA-1
  </pending-message>
  <pending-message id="msg-2026-05-10T12:35:01.123Z-0002" from="slice-3" timestamp="2026-05-10T12:35:01.123Z">
  my Files: list expanded to include lib/auth.js — please coordinate
  </pending-message>
</codex-paired-pending-messages>

(Messages above were queued for you in your inbox before this dispatch. They have NOT yet been marked read; if relevant, acknowledge them in your final response so the orchestrator knows you saw them.)
```

The block is prepended to the existing dispatch prompt body. Trailing prose in parens is part of the injected payload (instructs the agent how to handle the block).

Repo-specific tag (`codex-paired-pending-messages`, not bare `messages`) prevents collision with native Claude Code teammate auto-injection.

### 4.3 Sidecar schema delta

Each dispatch entry in `autopilot.dispatches` (or equivalent) gains an optional field:

```json
{
  "slice_id": "slice-5",
  "transport": "claude-subagent",
  "dispatched_at": "2026-05-10T12:34:56.789Z",
  "injected_message_ids": [
    "msg-2026-05-10T12:34:56.789Z-0001",
    "msg-2026-05-10T12:35:01.123Z-0002"
  ],
  "result": null
}
```

Field semantics:

- Empty array (`[]`) → no messages were pre-injected.
- Non-empty → those ids were prepended to the dispatch prompt and have been read by the agent (in the sense that the agent saw the prompt) but NOT yet marked read in the mailbox.

### 4.4 Mark-read on terminal result

When the orchestrator processes the slice's terminal result (`DONE`, `BLOCKED`, `NEEDS_CONTEXT`, or fallback-triggering parse failure where the prompt definitely ran), it marks every id in `injected_message_ids` as read:

```bash
node lib/codex-bridge/cli.js mailbox-mark-read-batch \
  --for slice-N --actor orchestrator \
  --message-ids "msg-...,msg-..."
```

If dispatch fails *before* the agent process started (e.g., codex binary not found, Task tool refused), the orchestrator does NOT mark read. Messages stay unread for next dispatch attempt.

If dispatch retries after agent crash, the unread messages get re-injected (duplicate context is preferred over loss).

### 4.5 New CLI subcommand: `mailbox-mark-read-batch` + shared helper

**Internal API (mailbox.js):** new exported helper `markManyAsRead(repoRoot, sliceId, messageIds)` — single lockfile acquisition for the whole batch. Used by both the CLI subcommand and the in-process hook (§5.2).

```js
export async function markManyAsRead(repoRoot, sliceId, messageIds) {
  // Acquires lockfile once; iterates ids; releases. Unknown ids are silently
  // skipped (idempotent). Returns { marked: string[], skipped: string[] }.
}
```

**CLI surface:** `mailbox-mark-read` already exists for single id. New `mailbox-mark-read-batch`:

```bash
node lib/codex-bridge/cli.js mailbox-mark-read-batch \
  --for <recipient> --actor <actor> --message-ids <id1,id2,...> \
  --repoRoot <abs-path>
```

Validation:

- `--actor` must equal `--for` OR `orchestrator` (same as single mark-read).
- `--message-ids` must be non-empty CSV; each id must match `msg-<ISO>-NNNN` format.
- Unknown ids in the list → silently skipped (idempotent).
- All-or-nothing within the lockfile: either all writes commit or none (lockfile rollback on JSON write failure).
- Both CLI and hook path through `markManyAsRead` — DRY, no risk of CLI/hook divergence.

## 5. PostToolUse Hook for Sonnet Auto-Injection

### 5.1 Identity mechanism — hook stdin metadata

Claude Code passes a JSON payload on stdin to every PostToolUse hook (`BaseHookInputSchema` in `entrypoints/sdk/coreSchemas.ts:387`):

```json
{
  "session_id": "...",
  "transcript_path": "...",
  "cwd": "/abs/path/to/agent/working/directory",
  "permission_mode": "...",
  "agent_id": "ag_xxx",
  "agent_type": "general-purpose",
  "hook_event_name": "PostToolUse",
  "tool_name": "Bash",
  "tool_input": {...},
  "tool_response": {...}
}
```

Note: `agent_id` is "present only when the hook fires from within a subagent" per the schema description.

Identity rules for the hook:

1. **Skip if `agent_id` is absent.** The orchestrator's main thread doesn't need hook injection — orchestrator handles its own mailbox via direct CLI calls in Phase B logic.
2. **Skip if `cwd` doesn't match the worktree pattern.** Slice subagents run in `<repo>/.git-worktrees/<slice-id>/...` per the v0.7.0 worktree convention. Pattern: `(?:^|/)\.git-worktrees/(slice-\d+)(?:/|$)`. If no match, the subagent is not a slice agent (e.g., a code-reviewer subagent) — exit silently.
3. **Extract `slice-N` from `cwd`.** That is the actor identity. No env var, no leakage between concurrent slices (each runs in its own worktree path).
4. **Repo root inference (path-segment parse + validation).** Split `cwd` into path segments. Scan from left to right for the LAST occurrence of `.git-worktrees` immediately followed by `slice-\d+`. Take the prefix up to (but not including) `.git-worktrees` as the candidate repo root. **Validate the candidate**: it must contain a `.codex-paired/` subdirectory (the mailbox root). If validation fails, exit 0 (not a slice context). This handles the edge case where the repo itself is mounted under a path containing `.git-worktrees/slice-N` — the inner-most match wins, and validation rejects spurious matches.

### 5.2 Hook implementation — Node module + thin bash wrapper

Per Codex critique (round 1), the hook logic is a Node module with unit tests; the bash wrapper is a one-liner.

`lib/codex-bridge/hook-mailbox-inject.js`:

```js
// PostToolUse hook injection logic. Reads hook stdin, identifies the slice
// actor from cwd, reads the actor's unread mailbox, emits hookSpecificOutput
// JSON on stdout.
//
// Exit codes:
//   0 — success, quiet no-op, or non-fatal failure with breadcrumb (always 0
//       so Claude Code does not inject hook errors into the subagent context).
//
// Why always exit 0: Claude Code wraps exit code 2 as `hook_blocking_error`
// AND non-zero/non-2 exits as `hook_non_blocking_error` attachments. Both
// inject into the subagent's tool-result stream. Mailbox infrastructure
// errors (corruption, lock contention) are not actionable by the worker —
// surfacing them just pollutes the prompt. We exit 0 and write a breadcrumb
// to <repoRoot>/.codex-paired/diagnostics/hook-failures.jsonl so the
// maintainer (or B.6.5 polling) can audit failures without disturbing the
// worker.
//
// Failure semantics (pinned per spec §5.4):
//   - Hook stdin missing/malformed → exit 0 with no output (treat as no-op).
//   - cwd doesn't match slice worktree → exit 0 with no output (not a slice).
//   - mailbox-read fails → exit 0 + append breadcrumb to diagnostics file.
//     Orchestrator's B.6.5 polls separately and is authoritative for halt.
//   - mailbox-mark-read fails AFTER successful emit → continue (per-message
//     try/catch swallows; message stays unread → duplicate-delivered on next
//     fire; acceptable).
//
// Order: read → emit additionalContext → AWAIT stdout flush → mark read.
// Stdout flush is explicit (callback-based write) so mark-read only fires
// after Claude Code has the bytes.

import { readUnreadMessages, markManyAsRead } from './mailbox.js';
import { realpathSync, existsSync, appendFileSync, mkdirSync } from 'node:fs';
import { join } from 'node:path';

const WORKTREE_SEGMENT = '.git-worktrees';
const SLICE_RX = /^slice-\d+$/;

function canonicalizePath(p) {
  if (!p) return null;
  try { return realpathSync.native(p); }
  catch { return p; }  // realpath failed (e.g., path doesn't exist) → trust literal
}

function inferActorAndRepoRoot(cwd) {
  const canonical = canonicalizePath(cwd);
  if (!canonical || typeof canonical !== 'string') return null;
  const segments = canonical.split('/');
  // Scan right-to-left for the inner-most .git-worktrees/slice-N pair.
  for (let i = segments.length - 2; i >= 0; i--) {
    if (segments[i] === WORKTREE_SEGMENT && SLICE_RX.test(segments[i + 1])) {
      const candidateRoot = segments.slice(0, i).join('/');
      if (!candidateRoot) continue;
      if (!existsSync(join(candidateRoot, '.codex-paired'))) continue;
      return { actor: segments[i + 1], repoRoot: candidateRoot };
    }
  }
  return null;
}

function writeBreadcrumb(repoRoot, sliceId, errMsg) {
  try {
    const dir = join(repoRoot, '.codex-paired', 'diagnostics');
    mkdirSync(dir, { recursive: true });
    const entry = JSON.stringify({
      ts: new Date().toISOString(),
      slice: sliceId,
      error: errMsg,
    }) + '\n';
    appendFileSync(join(dir, 'hook-failures.jsonl'), entry);
  } catch { /* breadcrumb is best-effort */ }
}

function flushStdout(payload) {
  return new Promise((resolve) => {
    process.stdout.write(payload, () => resolve());
  });
}

export async function main(stdinJson) {
  let input;
  try { input = JSON.parse(stdinJson); } catch { return { exit: 0 }; }
  if (!input.agent_id) return { exit: 0 };
  const ident = inferActorAndRepoRoot(input.cwd);
  if (!ident) return { exit: 0 };
  const { actor, repoRoot } = ident;

  let unread;
  try {
    unread = await readUnreadMessages(repoRoot, actor);
  } catch (err) {
    writeBreadcrumb(repoRoot, actor, `read failed: ${err.code || err.message}`);
    return { exit: 0 };
  }
  if (!Array.isArray(unread) || unread.length === 0) return { exit: 0 };

  const context = formatContext(actor, unread);
  const output = { hookSpecificOutput: { hookEventName: 'PostToolUse', additionalContext: context } };
  await flushStdout(JSON.stringify(output));

  // Stdout has flushed. Mark-read can now safely fire.
  try {
    await markManyAsRead(repoRoot, actor, unread.map(m => m.id));
  } catch (err) {
    writeBreadcrumb(repoRoot, actor, `mark-read failed: ${err.code || err.message}`);
    // Acceptable: messages remain unread → duplicate-delivered next fire.
  }
  return { exit: 0 };
}

function formatContext(recipient, msgs) {
  const esc = (t) => String(t).replace(/&/g, '&amp;').replace(/</g, '&lt;').replace(/>/g, '&gt;');
  const tags = msgs.map(m =>
    `  <pending-message id="${esc(m.id)}" from="${esc(m.from)}" timestamp="${esc(m.timestamp)}">\n` +
    `  ${esc(m.text)}\n` +
    `  </pending-message>`
  ).join('\n');
  return `<codex-paired-pending-messages recipient="${esc(recipient)}">\n${tags}\n</codex-paired-pending-messages>\n\n` +
         `(Messages above were just delivered to your mailbox. They have been marked read.)`;
}
```

### 5.4 Failure semantics (pinned)

| Failure | Exit | Side effect | Rationale |
|---|---|---|---|
| Hook stdin missing/malformed | 0 | none | Defensive; not our context |
| `agent_id` absent (main thread) | 0 | none | Orchestrator handles its own mailbox |
| `cwd` doesn't match slice worktree | 0 | none | Subagent not a slice (e.g., code-reviewer) |
| Repo-root candidate validation fails | 0 | none | Spurious `.git-worktrees` match; not real slice context |
| `mailbox-read` fails | 0 | breadcrumb append | Non-blocking; written to `<repo>/.codex-paired/diagnostics/hook-failures.jsonl`. NOT injected into worker prompt. |
| No unread messages | 0 | none | Fast path |
| Read OK, emit + mark-read OK | 0 | message marked read | Happy path |
| Read OK, emit OK, mark-read fails | 0 | breadcrumb append | Message stays unread → duplicate-delivered next fire. Acceptable. |
| Read OK, emit fails (process crash before flush) | >0 (crash) | none | Hook process died; message remains unread |

**Why always exit 0 on logic failures:** Claude Code wraps non-zero hook exits as attachments (`hook_blocking_error` for code 2, `hook_non_blocking_error` for other non-zero codes) injected into the worker's tool-result stream. Mailbox infrastructure errors (corruption, lock contention) are not actionable by the worker — surfacing them only pollutes the prompt. We exit 0 and write a breadcrumb to `<repo>/.codex-paired/diagnostics/hook-failures.jsonl` for maintainer audit.

**Key contract:** the orchestrator's existing Phase B.6.5 between-turns inbox poll remains the authoritative halt path. The hook is opportunistic auto-injection within a slice's run; orchestrator polling is what catches crashed/missed slices at terminal-result time. The diagnostics breadcrumb supports post-mortem without disturbing live work.

### 5.5 Concurrency

- Concurrent slices run in their own worktree paths (`<repo>/.git-worktrees/slice-3/`, `<repo>/.git-worktrees/slice-5/`). Hook reads `cwd` from stdin → no env var collision.
- Hook invocations from concurrent subagents both write to the same mailbox files. `proper-lockfile` (already in v0.7.3) handles cross-process atomicity.
- `mailbox-mark-read-batch` acquires the lockfile once for the whole batch, releases on completion or failure.
- Codex slices: hook does not fire for codex's internal tool calls (codex is a Bash subprocess from Claude Code's perspective, not a Task subagent). Codex stays on cooperative polling (§6).

## 6. Codex Cooperative Polling

For the codex transport, no auto-injection is achievable. Dispatch prompt instructs polling at semantic checkpoints. Specific checkpoints (in the codex agent's prompt body):

```
Mailbox checkpoints — call `mailbox-read --unread` at these points:

1. At the start of your work (after reading this prompt).
2. Before running each test (before `npm test`, `pytest`, etc.).
3. Before each `git commit`.
4. After any command that takes >30s.
5. Before composing your final response JSON.

Do NOT poll before every file edit — this trains ritual polling rather than thoughtful integration.
```

Pre-injection (§4) covers the most common case (orchestrator queues context before dispatch). Mid-run injection for codex remains a known limitation; documented as an architectural constraint, not a bug.

## 7. Halt-Now Signaling

Mailbox is for coordination/diagnostics only. To stop a slice immediately:

- Background Bash task: orchestrator calls `kill -TERM <pid>` on the codex subprocess. The status_file wrapper records `signal: SIGTERM`.
- Task subagent: orchestrator cancels the Task tool invocation (Claude Code's built-in mechanism).

Mailbox messages are NOT a halt signal. Slices may queue urgent messages but should not assume immediate consumption.

## 8. SKILL.md Phase B updates

Phase B.4 (dispatch slice agent) gains a pre-injection step before invocation. **No env var manipulation** — slice identity flows through hook stdin via `cwd` + `agent_id` (per §5.1).

```markdown
### Phase B.4 — Dispatch slice agent

For each slice in the current batch:

1. **Pre-inject pending mailbox messages.** Read recipient's unread inbox:
   ```bash

2. **Dispatch** via codex exec OR Task tool with the prepared prompt. The Sonnet path's PostToolUse hook (lib/codex-bridge/hook-mailbox-inject.js) handles all subsequent in-flight mailbox injection automatically based on the subagent's worktree cwd. The codex path relies on cooperative checkpoints in the dispatch prompt body (§6).

3. **On terminal result**, mark pre-injected messages read:
   ```bash
   node lib/codex-bridge/cli.js mailbox-mark-read-batch \
     --for slice-N --actor orchestrator \
     --message-ids "<csv>" \
     --repoRoot "$REPO_ROOT"
   ```
   If dispatch failed before the agent process started, SKIP the mark-read (messages stay unread for next dispatch attempt).
```

Phase B.4.5 (between-turns inbox polling) is preserved as a fallback for codex slices and documents the semantic checkpoint cadence (§6). For Sonnet slices, the hook handles in-flight delivery; B.4.5 polling for Sonnet becomes a redundant safety net but is retained for crash-recovery cases (e.g., hook itself failed).

2. **`--repoRoot` flag** added explicitly to `mailbox-read`, `mailbox-mark-read`, and `mailbox-mark-read-batch`. Replaces implicit cwd-based repo detection for CLI callers. The autopilot orchestrator's CLI invocations should pass `--repoRoot` for explicitness. The PostToolUse hook does NOT use the CLI — it imports `mailbox.js` directly and passes the repo root inferred from `cwd` (per §5.1 step 4).

Existing `mailbox-read` and `mailbox-mark-read` continue to work without `--repoRoot` (back-compat: falls back to cwd-derived repo root if not passed).

## 10. Risks and Non-Goals

### 10.1 Architectural limits (documented, not bugs)

- Mid-run injection for codex is impossible without runtime hooks. Cooperative polling is the only path for codex.
- `<codex-paired-pending-messages>` injection requires the agent to actually read its prompt. If the agent ignores the block, messages are lost (in the sense that the orchestrator believes they were delivered). Mitigation: the injected block ends with a parenthetical instructing the agent to acknowledge in its final response.

### 10.2 Concurrency risks

- Concurrent Task subagents each run in their own worktree (`<repo>/.git-worktrees/slice-N/`). Identity is derived per-call from hook stdin `cwd`, so concurrent subagents have isolated actor identity with no cross-contamination. (No env-var sharing; no parent-shell state.)
- Hook invocations from concurrent subagents both write to the same mailbox files. `proper-lockfile` (v0.7.3) handles cross-process atomicity; `mark-read-batch` acquires once for the whole batch.
- Mailbox archive rotation (per v0.7.3 spec) still happens once per Phase B turn; pre-injection happens between turns within Phase B.
- Race between concurrent reads (two slices reading the same orchestrator inbox): both readers will see the same unread set; both may emit `additionalContext` referencing it. Only the actor-matching slice can mark its own as read. For cross-slice messaging (slice A → slice B), slice B's hook fires when slice B is the agent, so only slice B's reads/marks apply. Risk: low.

### 10.3 Performance

- Hook fires on every Bash/Edit/Write/Read call. Empty-inbox fast path is a single file read + JSON parse. Estimated <10ms per fire on warm fs cache.
- Heavy slices (50+ tool calls) → 50+ hook fires. Acceptable, not a hot path.

### 10.4 Non-goals

- True mid-run push for codex (out of scope; documented).
- MCP server wrapping codex (deferred to v0.8.0 spike, gated on this work proving auto-injection ergonomics).
- Cross-machine mailbox (mailbox is filesystem-local; remote teaming is out of scope).
- Programmatic halt via mailbox (use signal-kill).

## 11. Migration & Compatibility

- v0.7.3 plans without `injected_message_ids` in their sidecar continue to work (field is optional, defaults to empty array).
- v0.7.3 mailboxes are forward-compatible (no schema change to message format).
- Hook is plugin-scoped; users without the plugin loaded are unaffected.
- Existing v0.7.3 cooperative polling instructions in dispatch prompts are downgraded to "fallback" for Sonnet, retained as primary for codex.

## 12. Out of Scope

- Auto-injection for codex (architectural limit).
- Long-poll mailbox semantics (would require codex MCP server; v0.8.0).
- Visual mailbox debugger (CLI subcommands sufficient for now).
