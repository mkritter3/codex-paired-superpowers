## 1. Goal

Three coupled features unlock "as many parallel agents as the dependency graph allows, without overlap":

1. **File-based mailbox** for orchestrator ↔ in-flight slice agents and slice ↔ slice messaging during execution. Persistent, atomic via `proper-lockfile`, archive-on-rotate.
2. **`**DependsOn:**` directive** in plan frontmatter — explicit DAG of slice dependencies.
3. **Dependency-graph batching** replaces v0.7.0's consecutive-slice batching. Phase B.2 computes the *ready-set* (pending slices whose deps are shipped) and dispatches the maximal non-overlapping subset of it. Non-consecutive slices can now parallelize when deps + Files allow.

## 2. Decision Boundary

Unchanged from rev4. Claude orchestrates; Node verifies mechanical state.

New additions to the mechanical-state side (Node):

- DAG construction + cycle detection (DependsOn parser).
- DAG digest computation + revalidation across resume.
- Mailbox file I/O (atomic write under `proper-lockfile`).
- Maximal first-fit non-overlap batching algorithm (deterministic).
- Archive rotation + retention.

## 4. Mailbox Architecture

### 4.1 File layout

```
<repo>/.codex-paired/
  └── mailboxes/
      ├── orchestrator.json          # orchestrator's inbox; agents write status/blockers here
      ├── slice-3.json               # slice-3's inbox
      ├── slice-5.json
      ├── *.json.lock                # lockfiles managed by proper-lockfile
      └── archive/
          ├── slice-3-2026-05-10T12:34:56.000Z.json   # rotated archives
          └── slice-3-corrupt-2026-05-10T13:00:00.000Z.json   # quarantined corrupt files
```

### 4.2 Message schema

Every message is an object with this shape:

```json
{
  "id": "msg-2026-05-10T12-34-56-000Z-0001",
  "from": "slice-3",
  "to": "orchestrator",
  "text": "Progress: 30% — finished test list",
  "timestamp": "2026-05-10T12:34:56.000Z",
  "summary": null,
  "color": null,
  "read_at": null
}
```

Field rules:

- `id`: monotonic, locally unique. Format `msg-<timestamp>-<seq>` where seq is a 4-digit zero-padded counter for that timestamp. Generation happens under lock so collisions are impossible.
- `from`: sender slice id or literal `"orchestrator"`. Must match `^orchestrator$|^slice-\d+$/`. Reject anything else BEFORE path construction (prevents path traversal).
- `to`: recipient. Same validation as `from`.
- `text`: arbitrary string. Multiline allowed; the JSON encoding handles escaping.
- `timestamp`: ISO-8601 UTC, written under lock.
- `summary`: optional 5-10 word preview shown in orchestrator diagnostics. Null if absent.
- `color`: optional sender's color hint for UI display. Null if absent.
- `read_at`: ISO-8601 timestamp when the message was marked read; null while unread.

### 4.3 Mailbox API (`lib/codex-bridge/mailbox.js`)

```js
writeToMailbox(repoRoot, recipient, message) -> Promise<{id}>
  // Acquires lock with retry policy below. Reads existing inbox, appends with
  // generated `id`, writes back, releases. Throws on validation failure
  // (recipient malformed, message JSON invalid, etc.) BEFORE any file I/O.
  // After successful write, checks size; if exceeded, calls archiveAndReset.
  // Returns `{id}` of the appended message.

readMailbox(repoRoot, sliceId) -> Promise<TeammateMessage[]>
  // Returns full array. ENOENT → []. Malformed JSON → BEST-EFFORT moves file
  // to archive/<slice>-corrupt-<ts>.json (failure to move is logged but does
  // not mask the primary halt) AND throws DispatcherRegistryError-style
  // error with code "mailbox-corrupt" so orchestrator can halt cleanly.
  // Diagnostic includes both the corruption reason AND any archive-move
  // failure detail.

readUnreadMessages(repoRoot, sliceId) -> Promise<TeammateMessage[]>
  // Filters readMailbox() to entries where read_at === null.

markAsRead(repoRoot, sliceId, messageId) -> Promise<void>
  // Acquires lock, finds message by id (NOT by index), sets read_at to now,
  // writes back. Throws if id not found.

archiveAndReset(repoRoot, sliceId) -> Promise<{archivedPath, carriedForward: TeammateMessage[]}>
  // Internal helper. Splits inbox into read (archive) and unread (carry forward).
  // Edge case: ALL messages unread AND inbox exceeds max_bytes → throws
  // "mailbox-overflow-unread" instead. Orchestrator halts; user must intervene
  // (read messages, raise max_bytes, or drop manually).

cleanupArchives(repoRoot, retentionConfig) -> Promise<{deleted: number}>
  // Deletes archive files older than retention_days OR beyond retention_count
  // per recipient. Run periodically by orchestrator.
```

### 4.4 Lockfile policy

- Library: `proper-lockfile` (existing battle-tested cross-process locking).
- Retry config: **50 retries**, jittered exponential backoff up to 250ms per attempt, total wall-clock ~10s. After exhaustion → halt `mailbox-lock-timeout`.
- **Stale-lock recovery**: `proper-lockfile` `stale: 60000` (60 seconds). If a process crashed while holding a lock and >60s elapsed, the lock is considered stale and replaced. 60s is ~6× the retry budget, giving healthy contention plenty of room before stale-detection triggers.
- **Update interval**: `update: 10000` (10 seconds). Lock-holders refresh their lock every 10s to prove liveness. Tests must verify that a held lock past the stale threshold gets reclaimed.
- Filesystem note: lockfile relies on directory creation atomicity. Works on local FS, NFS, APFS, ext4. Network filesystems with broken atomic create may not be safe; flag in README.

### 4.5 CLI subcommands

```
node lib/codex-bridge/cli.js mailbox-write \
  --to <recipient> \
  --from <sender> \
  [--text "..." | --text-stdin | --message-json-stdin] \
  [--summary "..."] \
  [--color "..."]

node lib/codex-bridge/cli.js mailbox-read \
  --for <recipient> \
  --actor <orchestrator|slice-N> \
  [--unread] \
  [--json]

node lib/codex-bridge/cli.js mailbox-mark-read \
  --for <recipient> \
  --actor <orchestrator|slice-N> \
  --id <message-id>
```

`--actor` is **required** for `mailbox-read` and `mailbox-mark-read`. The CLI validates the actor against the inbox owner per the permissions table in §4.6. Halt `mailbox-permission-denied` if an actor reads/marks an inbox they don't own (and is not orchestrator).

`mailbox-write` does not require `--actor` because writing is permissive (any actor can write to any recipient, subject to recipient-format validation).

Stdin variants `--text-stdin` and `--message-json-stdin` are required for multiline messages, JSON snippets, or anything with shell-unsafe characters. CLI exit semantics:

- 0 = success
- 1 = infrastructure failure (file I/O, lock timeout)
- 2 = parser/validation defect (recipient malformed, message JSON invalid, message-id not found)

### 4.6 Permissions model

| Action | Slice agent (e.g. slice-3) | Orchestrator |
|---|---|---|
| Write to `orchestrator.json` | ✅ allowed | ✅ allowed |
| Write to any other `slice-N.json` | ✅ allowed (slice-to-slice signaling) | ✅ allowed |
| Read own inbox (`slice-3.json` for slice-3 agent) | ✅ allowed | ✅ allowed |
| Read another slice's inbox | ❌ forbidden | ✅ allowed (orchestrator polls all in-flight) |
| Mark messages read in own inbox | ✅ allowed | ✅ allowed |
| Mark messages read in another inbox | ❌ forbidden | ✅ allowed |

Enforcement: the CLI accepts a `--actor <orchestrator|slice-N>` parameter; all read/mark-read operations validate that actor matches the inbox owner OR is `orchestrator`. `mailbox-write` is permissive (anyone can write to anyone).

### 4.7 Per-project config

`.codex-paired/project.json`:

```json
{
  "mailbox": {
    "max_bytes": 262144,
    "archive_policy": "rotate",
    "archive_retention_days": 30,
    "archive_retention_count": 100
  }
}
```

Defaults:
- `max_bytes`: 262144 (256 KB) — triggers archive rotation
- `archive_policy`: `"rotate"` (alternative: `"drop"` — see semantics below)
- `archive_retention_days`: 30 — delete archives older than this
- `archive_retention_count`: 100 — keep at most this many archives per recipient
- Cleanup applies the smaller of the two limits

**`archive_policy` semantics:**

- `"rotate"` (default): on size-exceedance, read messages move to a timestamped archive file under `archive/`, unread messages carry forward into the fresh inbox. All-unread overflow → halt `mailbox-overflow-unread`.
- `"drop"`: on size-exceedance, read messages are **dropped** (no archive file). Unread messages still carry forward. All-unread overflow still halts `mailbox-overflow-unread` — drop never silently loses unread visibility. Use this in memory-constrained envs where forensic archive accumulation is too costly.

Both modes preserve the invariant: unread messages are never silently lost. The only difference is whether read messages persist on disk after rotation.

## 5. DependsOn Directive

### 5.1 Plan frontmatter

```markdown
## Slice 5: Foo

**Implementer:** codex
**Domain:** backend

**DependsOn:**
- slice-3
- slice-4

**Files:**
- lib/foo.js
- tests/foo.test.js

### Tasks
- [ ] ...
```

### 5.2 Parser rules

Block parser reuses the same path as `**Files:**` block parsing (same battle-tested function from v0.7.0 slice 1). Specific rules:

- `**DependsOn:**` is optional. Absent → empty deps.
- Block format only — `**DependsOn:** slice-3` (inline) → halt `dep-block-malformed`.
- Empty block (no bullets) → halt `dep-block-malformed`.
- Each bullet must be `- slice-<N>` exact; reject anything else (no slugs, no descriptions).
- Self-reference (slice-3 depends on slice-3) → halt `dep-self-reference`.
- Unknown slice id (`slice-99` when plan only has slice-1..5) → halt `dep-unknown-slice` with offending id.
- Cyclic dependency (slice-3 → slice-5 → slice-3) → halt `dep-cycle` with cycle path in diagnostic.

### 5.3 DAG construction

`lib/codex-bridge/dependency-graph.js`:

```js
buildDAG(planPath) -> { ok: true, dag } | { ok: false, halt }
  // Parses every slice's DependsOn block. Validates: no cycles, no unknown ids,
  // no self-refs. Returns DAG: { nodes: { sliceId: { dependsOn: string[] } } }.
  // On any validation failure, returns { ok: false, halt: { reason, detail } }.

computeReadySet(dag, sliceStates) -> string[]
  // sliceStates: { sliceId: "pending" | "in-progress" | "shipped" | "failed" }
  // Returns slice IDs that are pending AND every dep is shipped.

maximalFirstFitNonOverlap(readySet, filesIndex) -> string[]
  // Deterministic first-fit by numeric slice order:
  //   for each slice in numeric order:
  //     if Files set is disjoint from already-picked Files: include
  // Returns the picked subset. Algorithm is deterministic and explainable.
  // For empty readySet: returns []. For single-slice readySet: returns [first].

computeDigest(dag) -> string
  // SHA-256 hex digest over canonical JSON of:
  //   { nodes: sortedKeys.map(id => [id, sortedDependsOn]) }
  // Stable across reload as long as plan content is unchanged.
```

### 5.4 DAG persistence + revalidation

- At autopilot session start: `buildDAG()` → persist `{digest, dag}` to sidecar `autopilot.dependency_graph` block.
- At every Phase B turn AND on every resume: rebuild DAG from current plan, compare digest with persisted. Mismatch → halt `plan-changed-during-autopilot` with old/new digests in diagnostic.
- This catches plan edits made between autopilot turns or sessions.

## 6. Phase B Integration

### 6.1 Phase B.0 (domain resolution + DAG build) — extended

Add steps before domain resolution:

1. **Build DAG.** `buildDAG(planPath)`. On halt, surface and exit (graph errors are pre-dispatch fatal).
2. **Persist + verify digest.** First call: write digest to sidecar. Subsequent calls: compare; halt `plan-changed-during-autopilot` on mismatch.
3. **Compute slice states.** From sidecar's per-slice `phases.implement.shipped` flag and `dispatches[].outcome`. States: `pending`, `in-progress` (active background task), `shipped`, `failed`.

Then proceed with domain resolution (rev3 §3.5) per slice.

### 6.2 Phase B.2 (batching) — REPLACED

v0.7.0 logic ("consecutive non-overlapping slices") is replaced with:

```
1. ready_set = computeReadySet(dag, slice_states)
2. if ready_set is empty: nothing to do this Phase B turn (wait for in-progress completions)
3. batch = maximalFirstFitNonOverlap(ready_set, files_index)
4. dispatch batch
```

Mixed Codex/Sonnet within batch still allowed (transport-aware dispatch from rev4 §6).

### 6.3 Phase B.4 (dispatch) — extended for mailbox protocol

Every dispatch's prompt now includes the agent's mailbox protocol:

```
Your inbox: <repo>/.codex-paired/mailboxes/<your-slice-id>.json

To send a status/progress/blocker message to the orchestrator:
  node <plugin>/lib/codex-bridge/cli.js mailbox-write \
    --to orchestrator \
    --from <your-slice-id> \
    --text-stdin
  (then write your message to stdin)

To send a message to another in-flight slice (e.g. slice-3):
  node <plugin>/lib/codex-bridge/cli.js mailbox-write \
    --to slice-3 \
    --from <your-slice-id> \
    --text-stdin

For long-running slices: send a progress message every 5-10 minutes so the
orchestrator has visibility. Progress messages are best-effort logging;
failure to deliver does not block your work.

EXCEPTION: BLOCKER messages (text starting with "BLOCKER:") MUST be
delivered. If your mailbox-write call for a BLOCKER message fails (lock
timeout, recipient malformed, mailbox corrupt) — DO NOT silently continue.
Set your final-message JSON status to "BLOCKED" or "NEEDS_CONTEXT" with
the failed-delivery reason in concerns. The orchestrator's failure-mode
table treats failed-blocker-delivery as a real blocker, not a logging hiccup.
```

**Failure boundary distinction (mailbox operations):**

| Operation | Caller | Failure handling |
|---|---|---|
| Orchestrator reading any inbox | orchestrator | Halt on `mailbox-corrupt`, `mailbox-lock-timeout`, `mailbox-permission-denied` (programming error). |
| Orchestrator writing to any inbox | orchestrator | Halt on lock timeout / corruption. Orchestrator-side write failures are uncommon and indicate filesystem trouble. |
| Agent writing a non-BLOCKER message | slice agent | Best-effort. Lock timeout / write failure → log to stderr, continue work. Does NOT escalate. |
| Agent writing a BLOCKER message | slice agent | Mandatory delivery. On failure → set final status to BLOCKED/NEEDS_CONTEXT with delivery-failure noted in concerns. Orchestrator treats as real blocker. |
| Agent reading own inbox | slice agent | Best-effort. Failure to read own inbox does not block agent's work. |

### 6.4 Phase B between-turns — orchestrator polls

Between every assistant turn during Phase B (after dispatching, before reading task notifications):

1. Read `orchestrator.json` inbox (unread). Surface each message in diagnostics.
2. For each in-flight slice (outcome=in-progress): read that slice's inbox (unread). Surface in diagnostics.
3. Mark all messages read after surfacing.
4. If any message is a blocker (text starts with `BLOCKER:` convention), pause integration for that slice and surface to user.

### 6.5 Failure cascade

When ANY slice in the autopilot run reaches a `failed-halted` outcome (preferred + fallback both failed, or halt without fallback), the orchestrator:

1. Computes the descendant set: all slices transitively depending on the failed slice (BFS on DAG).
2. Halts the autopilot run with `dependency-cascade-halt` diagnostic listing failed slice + blocked descendants.
3. Leaves shipped slices intact (they're already on integration branch).
4. User must investigate the failed slice; on resume, autopilot re-checks DAG digest and resumes from `pending` state.

## 7. Failure Modes (additions to rev4 §17)

| Condition | Halt reason | Behavior |
|---|---|---|
| `**DependsOn:**` block malformed | `dep-block-malformed` | Halt before any worktree work; diagnostic cites slice + line. |
| `**DependsOn:**` references unknown slice id | `dep-unknown-slice` | Halt; diagnostic cites both slice ids. |
| Slice depends on itself | `dep-self-reference` | Halt; diagnostic cites slice. |
| Cycle in dependency graph | `dep-cycle` | Halt; diagnostic includes cycle path. |
| Plan changed mid-autopilot (digest mismatch) | `plan-changed-during-autopilot` | Halt; diagnostic includes old + new digests. |
| Mailbox JSON unparseable | `mailbox-corrupt` | Move corrupt file to `archive/<slice>-corrupt-<ts>.json` BEFORE halt; user creates fresh inbox manually. |
| All inbox messages unread + inbox exceeds max_bytes | `mailbox-overflow-unread` | Halt; user must read messages or raise `max_bytes`. |
| `mailbox-write` recipient/from doesn't match `orchestrator|slice-\d+` | `mailbox-recipient-malformed` | Halt before path construction (path-traversal guard). |
| `mailbox-read` or `mailbox-mark-read` actor doesn't own inbox AND isn't orchestrator | `mailbox-permission-denied` | Halt; programming error in caller. |
| `proper-lockfile` retry budget exhausted | `mailbox-lock-timeout` | Halt; diagnostic includes file path + last error. (Orchestrator only — agent BLOCKER writes escalate via final-status JSON; agent progress writes log + continue.) |
| Mailbox write for `BLOCKER:` message fails (lock timeout / corruption / recipient malformed) and agent must report a real blocker | (no halt at mailbox layer) | Per §6.3 boundary table, agent's final status JSON sets BLOCKED/NEEDS_CONTEXT with delivery-failure noted in concerns. Orchestrator treats as real blocker. |
| Slice fails (preferred + fallback both fail or halt-no-fallback) | `dependency-cascade-halt` | Compute blocked descendants; halt run; preserve shipped slices. |

## 8. Validation Rubric Coverage

- **tier:** critical
- **happy:** linear deps (A→B→C runs serial); diamond (A,B → C; D→E); fan-out (1 slice, 5 fanned out); mailbox roundtrip (orchestrator↔agent↔orchestrator); message id uniqueness; archive rotation preserves unread.
- **edge.zero-null-empty:** absent DependsOn block; empty inbox; missing inbox file (lazy creation); unread-but-empty after mark-as-read on all.
- **edge.boundary:** inbox at exactly max_bytes; archive count at exactly retention_count; retention_days at boundary.
- **edge.large-input:** plan with 50 slices in complex DAG; 200-message inbox before rotation.
- **edge.concurrent:** 5 agents writing to orchestrator inbox simultaneously (50-retry policy holds); concurrent archive rotation + write race.
- **edge.adversarial:** path traversal in recipient name (`../etc/passwd`); cycle in DAG; corrupt JSON; lockfile stuck (stale lock recovery).
- **fail.dependency:** `proper-lockfile` not installed; `node_modules` missing in worktree; lockfile retry exhausted.
- **fail.malformed-input:** dep block edge cases; mailbox JSON edge cases; recipient/from format violations.
- **fail.exception-path:** all new halt reasons surfaced cleanly with structured diagnostics.
- **integration.cross-module:** changes touch mailbox.js, dependency-graph.js, cli.js (3 new subcommands), project-config.js (mailbox block), sidecar.js (dependency_graph digest), SKILL.md Phase B.0/B.2/B.4 prose.
- **stress.scale:** plan with 50 slices + 100-message archives at retention boundary.
- **perf.slo:** lockfile total budget ~10s; mailbox-write latency under contention should stay under 1s p99.
- **compat.breaking:** existing v0.7.2 plans without `**DependsOn:**` directive default to no deps (empty array); zero-deps invariant ENDS with proper-lockfile addition (called out in README).
- **critical.residual-risk:** crash recovery mid-batch when inboxes have unread messages — orchestrator on resume must restore visibility into in-flight slices, NOT lose mailbox state. Tested by simulating SIGKILL of orchestrator with unread blocker in `orchestrator.json` and confirming next session reads them on B.0.

## 9. Out of Scope (deferred to v0.7.4+)

- Mailbox UI / live-tailing in a separate pane.
- Inter-slice RPC (synchronous request/response across slices).
- Mailbox sharding for very-high-contention orchestrator inbox (deferred until empirical contention exceeds 50-retry budget).
- Cross-repo mailbox (e.g., a slice in one repo signaling a slice in another).
- Mailbox encryption (current messages are in plaintext under repo-relative path).
