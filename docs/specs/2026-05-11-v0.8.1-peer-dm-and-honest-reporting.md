# v0.8.1 — Peer-DM Enqueue + Honest-Reporting Hook

**Codex thread:** `019e1536-0211-7d50-a307-d16d092d44b7` (continuation of v0.8.0 follow-up thread)
**Design rounds:** 4 rounds (REVISE → REVISE → REVISE → SHIP) on 2026-05-11. Claude SHIP round 4 → double-SHIP.

## Why

The v0.8.0 release surfaced two real gaps:

1. **Peer-DM round-trip is broken end-to-end.** Machine Result `peer_messages_sent` was captured but never written to recipient mailboxes. `expert-dm-scheduler.drainPeerDMs` could only fire for experts seeded by the orchestrator. The "peer-negotiated coordination" story was unwired.

2. **Orchestrator made unverified claims.** During the v0.8.0 release-gate write-up, Claude (this orchestrator) reported PASS for things it hadn't actually verified end-to-end. The user surfaced the gap. Mechanical guardrails are needed.

## Gap 1 — Peer-DM Enqueue

### Schema (Machine Result side)

- New canonical field: `peer_messages_requested` (array of items).
- Legacy alias: `peer_messages_sent` accepted for v0.8.1 only, removed v0.9.0.
- If both present → preferred wins, legacy ignored with parser warning.
- Each item: `{to: string, body: string, summary?: string}` with `body ?? summary` fallback so v0.8.0 outputs (which used `summary` only) keep working.

### Parser scope (`lib/codex-bridge/expert-output-parser.js`)

- Validates **array-ness** of whichever field is present.
- Does NOT validate per-item shape — runtime owns classification.
- Repair prompt asks for `peer_messages_requested`.
- A v0.8.0 expert returning `peer_messages_sent` requires NO repair — parser normalizes.

### Runtime pipeline (`lib/codex-bridge/expert-turn.js runTurnWithDeps`)

After parse success, before mark-read:

1. Resolve `requests = parsed.peer_messages_requested ?? parsed.peer_messages_sent ?? []`.
2. For each item, classify:
   - `valid` — passes recipient check, not self-DM, has body or summary.
   - `invalid-recipient` — recipient fails `RECIPIENT_RE` validation.
   - `self-dm` — `entry.to === identity.id`.
   - `empty-body` — neither `body` nor `summary` is a non-empty string.
   - `malformed-item` — not an object, or `to` not a string.
3. Enqueue valid via `writeToMailbox(repoRoot, to, {from: identity.id, text: body ?? summary, summary: summary ?? null})`.
4. Catch each per-item MailboxError; record `{to, reason, code}` in `peer_messages_failed`.
5. Mark inbound messages read (existing behavior).
6. Append sidecar turn with new fields:
   - `peer_messages_enqueued: [{to, message_id, summary?}]`
   - `peer_messages_failed: [{to, reason, code?}]`
7. Return `{ok: true, result, peer_dm_summary: {enqueued: N, failed: M}}`.

### Scheduler ownership (`lib/codex-bridge/expert-dm-scheduler.js`)

`drainPeerDMs` checks `turnResult.peer_dm_summary.failed > 0` immediately after each `runTurn` returns. If >0 → halt with `expert-peer-dm-enqueue-failed`. Primary detector. Sidecar scanning in B.5.5 is the secondary safety net for cases where `appendExpertTurn` failed and the scheduler is the only path that observed the failure.

### Halt-reason policy (`lib/codex-bridge/expert-archive.js`)

Add `expert-peer-dm-enqueue-failed` to `HALT_REASONS_PRESERVE`.

### Invariant

Peer-DM enqueue failure does NOT roll back the inbound mark-read or fail the turn itself — the expert's own work succeeded. But the scheduler halts the drain so the orchestrator can surface to user / debug. Mailboxes are preserved.

### Spawn prompt & repair prompt updates

`assembleSpawnPrompt` now says:

> `peer_messages_requested: array of {to, body, summary?}` — messages you want delivered to other active experts via the mailbox. `body` is the full text; `summary` is an optional one-liner for the sidecar audit. The orchestrator's runtime enqueues these on parse success.

Repair prompt mirrors that and asks for `peer_messages_requested` only.

## Gap 2 — Honest-Reporting Hook

### Activation marker (strict scope)

Hook activates ONLY when `<repo-root>/.codex-paired/honest-reporting-active.json` exists AND its `expiresAt` is in the future. Existence of `.codex-paired/` alone is insufficient.

Marker shape:

```json
{
  "skillName": "autopilot",
  "sessionStartedAt": "2026-05-11T14:30:00.000Z",
  "expiresAt": "2026-05-11T22:30:00.000Z",
  "specPath": "/abs/path/to/spec.md"
}
```

Default TTL 8 hours. Hook treats missing OR malformed OR expired marker as inactive (exit 0). No skill-cleanup step needed — TTL is the contract.

Skills that write the marker on entry (v0.8.1):

- `codex-paired-superpowers:brainstorming`
- `codex-paired-superpowers:writing-plans`
- `codex-paired-superpowers:subagent-driven-development`
- `codex-paired-superpowers:autopilot`
- `codex-paired-superpowers:test-driven-development`
- `codex-paired-superpowers:systematic-debugging`

Each skill's entry block adds one line invoking the bridge CLI helper.

### Stop hook scanner (`lib/codex-bridge/hook-honest-reporting.js`)

1. Check marker presence and TTL — exit 0 if absent/expired/malformed.
2. Read last assistant turn from `$CLAUDE_TRANSCRIPT_PATH`.
3. Strip excluded regions: fenced code blocks, inline backticks, blockquotes (`>`), `<<<VERDICT>>>...<<<END>>>` blocks, `## Machine Result` sections, user-quoted lines.
4. Scan remaining text for high-precision claim vocabulary:
   - Caps-sensitive: `\b(VERIFIED|PASS|PASSED|SHIPPED|TAGGED|PUSHED|DEPLOYED|RELEASED|INSTALLED|CONFIRMED)\b`
   - Case-insensitive phrases: `verified`, `tests? pass(?:es|ed)?`, `smoke pass(?:es|ed)?`, `confirmed`, `deployed`, `released`, `installed`, `shipped`
5. For each match, look for **evidence proximity** within ±200 chars AND same paragraph:
   - inline backticks
   - tool-call references: `Bash`, `Edit`, `Read`, `gh `, `git `, `node `, `npm `
   - file paths matching `(?:[\w.-]+/){1,}[\w.-]+`
   - explicit caveat markers: `ASSUMED`, `UNTESTED`, `not verified`, `haven't checked`, `based on prior session`
   - proximity to a stripped code-block region (we still know where they were)
6. Decision:
   - all matches have proximity evidence → exit 0.
   - any match lacks evidence → exit 2 with rewrite instructions.
   - no matches → exit 0.

### PreToolUse guards

Active-marker required. Bash command matchers:

- `^git tag\b`
- `^git push\b`
- `^gh release create\b`
- `^npm publish\b`

When matched + active: scan last 2 assistant turns for vocabulary; if any unsourced claim → exit 2 with: "About to <command>. Show the verification tool calls in this/prior turn, or reclassify each claim as ASSUMED/UNTESTED."

### Exit-code-2 message template

```
Honest-reporting hook: found N claim(s) without nearby evidence:
  - "VERIFIED" at offset 1234: no tool reference or caveat within ±200 chars
  - "shipped" at offset 5678: no tool reference or caveat within ±200 chars

Required action: rewrite the response so each claim is one of:
  (a) cited: name the tool command and quote the output that established it, OR
  (b) marked ASSUMED, UNTESTED, or "based on prior session" if you didn't verify this turn.

Hook is active because <.codex-paired/honest-reporting-active.json> is present
(skill: <skillName>, expires: <expiresAt>).
```

### Tests (scanner)

- verdict block ignored.
- fenced code block ignored.
- inline backtick ignored.
- match + adjacent file path = pass.
- match + adjacent caveat marker = pass.
- match + adjacent fenced output block proximity = pass.
- match alone = fail (exit 2).
- caps-sensitivity: `pass` mid-sentence does not match; `PASS` does.
- marker-file-absent → noop (exit 0).
- marker expired → noop.
- malformed transcript path → exit 0 (fail-open on env weirdness).
