# v0.4.1 — Parser-as-Code + Sidecar Relocation

**Status:** draft for review (round 2)
**Date:** 2026-05-08
**Owner:** mkr
**Target version:** v0.4.1
**Validation:** critical

## Goal

Close two implementation-fidelity gaps surfaced by Codex's residual concern after v0.4 SHIP'd:

1. The validation-coverage parser is currently prose in `skills/autopilot/SKILL.md`, not code. An orchestrator following the SKILL.md prose could interpret "validate keys" loosely; the rubric's enforcement is only as strong as the orchestrator's discipline. Make it code.
2. Sidecar JSON files (`<spec>.codex.json`) sit next to the specs they review, polluting the visible directory and conflating transient runtime state with permanent docs. Move to a hidden per-repo directory while preserving the spec-to-sidecar mapping.

## Non-goals
- Real e2e autopilot smoke (v0.5.0).
- Concurrent-autopilot protection (deferred).
- Codex MCP timeout handling (deferred).
- Sophisticated MCP retry policies (deferred).

## Part 1: Validation-coverage parser as code

### New module: `lib/codex-bridge/validation-coverage.js`

**Scope:** Phase A coverage parsing ONLY. Phase C verifies Phase A's commitments via different keys (`rubric.diff-vs-plan`, `rubric.test-results`, `rubric.uncovered-paths`, `rubric.new-triggers`); a Phase C verifier is out of scope for v0.4.1 and will use a separate function in a future release.

Single exported function:

```js
parseValidationCoverage(critique: string[], opts?: { tier?: string }) =>
  | { ok: true, tier: 'light' | 'standard' | 'critical', coverage: { [key: string]: string } }
  | { ok: false, defect: string, detail?: string }
```

Where:
- `critique`: the array of bullet strings from a verdict block.
- `opts.tier`: optional explicit tier override (the slice's declared tier from plan frontmatter); if absent, the function reads the `tier:` bullet from `critique`.

Defect codes (the parser is the gate; exhaustive list):
- `tier-missing` — no `tier:` bullet found in critique.
- `tier-invalid:<value>` — `tier` bullet has a value not in `{light, standard, critical}`.
- `tier-mismatch` — caller passed `opts.tier` and it differs from the bullet's value (e.g., plan declares `critical` but verdict says `standard`).
- `missing-key:<key>` — required Tier-1 or Tier-2 key absent.
- `missing-key:critical.residual-risk` — `tier: critical` but no `critical.residual-risk:` bullet.
- `duplicate-key:<key>` — same key appears in two bullets.
- `unknown-key:<key>` — bullet whose key isn't in the allowed set.
- `malformed-bullet:<index>` — bullet at zero-indexed position N has no `:` separator OR the part before `:` is not a valid-shaped key (empty, whitespace-only, or invalid characters). The presence of a colon with whitespace after it is NOT malformed at this layer; that's caught by `empty-value` once the key is recognized.
- `empty-value:<key>` — bullet has a recognized key + colon, but the value after `:` is empty or whitespace-only. (This is a separate defect from `malformed-bullet` because the bullet structure was valid enough to extract the key.)
- `not-array` — input was provided but isn't a JSON array (CLI-mode only; Node-mode passes typed args).
- `non-string-element:<index>` — array contained a non-string element at index N.
- `invalid-json-input` — CLI-mode received stdin that doesn't parse as JSON.

These defect codes are parser-domain only. Infrastructure failures (Node crash, missing CLI, fs error) are NOT defects; the CLI exits with a different code (see CLI subcommand below).

Allowed-keys set is hard-coded in the module (the rubric's source of truth):
- Metadata: `tier`
- Tier 1: `happy`, `edge.zero-null-empty`, `edge.boundary`, `edge.large-input`, `edge.concurrent`, `edge.adversarial`, `fail.dependency`, `fail.malformed-input`, `fail.exception-path`, `integration.cross-module`
- Tier 2: `stress.scale`, `perf.slo`, `compat.breaking`
- Tier 3 (only when `tier: critical`): `critical.residual-risk`

### CLI subcommand: `validation-parse`

```
node ${CLAUDE_PLUGIN_ROOT}/lib/codex-bridge/cli.js validation-parse [--tier <t>] < critique.json
```

- Reads JSON array of strings from stdin.
- Calls `parseValidationCoverage(critique, { tier })`.

**Exit code semantics (autopilot reads these to dispatch):**
- `0`: success. Stdout is the coverage map JSON.
- `2`: parser defect. Stderr is `{ defect: "<code>", detail: "<text>" }` JSON. Autopilot halts with `halt_reason: "validation-coverage-malformed:<defect>"`.
- `1`: CLI infrastructure failure (uncaught exception, fs error, malformed argv). Stderr is the raw error. Autopilot halts with `halt_reason: "cli-infrastructure-error"` — distinct from a parser defect because retry/fix paths differ.

This three-way split lets the orchestrator distinguish "Codex emitted bad rubric coverage" (re-prompt Codex with the defect) from "the parser itself is broken / Node crashed" (escalate to user, don't retry blindly).

### Autopilot SKILL.md update

Phase A's step 6 ("On double-SHIP, parse and validate") changes from prose to:

```bash
echo "<JSON array of critique bullets>" | \
  node ${CLAUDE_PLUGIN_ROOT}/lib/codex-bridge/cli.js validation-parse --tier <slice-tier>
```

Dispatch on exit code (matching the CLI's three-way semantics):
- **Exit 0**: success. Stdout is the coverage map JSON. Write directly to `slice_reviews[slice-N].phases.plan-slice.validation_coverage`.
- **Exit 2**: parser defect. Stderr has `{ defect, detail }` JSON. Halt with `halt_reason: "validation-coverage-malformed:<defect>"`. Codex needs to re-emit a conforming verdict; this is a re-prompt path.
- **Exit 1**: CLI infrastructure failure (uncaught exception, fs error, malformed argv). Halt with `halt_reason: "cli-infrastructure-error"`. This is NOT a Codex problem — escalate to the user; do NOT retry blindly.

The orchestrator no longer interprets the rubric — it shells out to a deterministic parser whose behavior is unit-tested. The exit-code split lets the orchestrator distinguish "Codex emitted bad rubric coverage" from "the parser itself crashed."

## Part 2: Sidecar relocation

### New layout

- **Anchor** (unchanged): `<repo-root>/.codex-paired/active.json` — used by the hook for fast lookup.
- **Sidecar** (NEW): `<repo-root>/.superpowers-codex-paired/<spec-relative-path>.json`.

Where `<spec-relative-path>` is the spec's path relative to the repo root.

Examples:
- `<repo>/docs/specs/foo.md` → `<repo>/.superpowers-codex-paired/docs/specs/foo.md.json`
- `<repo>/spec/bar/baz.md` → `<repo>/.superpowers-codex-paired/spec/bar/baz.md.json`

If the spec path is not under the repo root (e.g., `/tmp/...`), fall back to the OLD layout (`<spec>.codex.json` next to the spec) — this keeps tests using `mkdtemp` paths working without complication.

### Implementation in `sidecar.js`

The function auto-discovers the repo root via git from the spec's directory. This means **no caller needs to pass `repoRoot`** — the change is transparent to existing CLI subcommands, the hook, and SKILL.md callers.

```js
function sidecarPathFor(specPath) {
  // 1. Resolve specPath to absolute, normalized form.
  // 2. Try to find a git repo root by walking up from dirname(specPath).
  //    Use `git rev-parse --show-toplevel` (synchronous spawn).
  // 3. If found AND specPath is contained under repoRoot (use realpath
  //    on both sides to defeat symlinks; check normalized prefix with
  //    a path separator boundary so /repo and /repo-old don't match):
  //      hidden = path.join(repoRoot, '.superpowers-codex-paired',
  //                         path.relative(repoRoot, specPath) + '.json')
  //      legacy = specPath + '.codex.json'
  //
  //      Discovery rule: if `legacy` exists AND `hidden` does not, return legacy
  //                       and emit a deprecation warning to stderr (one-time
  //                       per process via a module-scoped Set). This prevents
  //                       silent shadowing during transition.
  //                     Otherwise return `hidden`.
  // 4. If no repo (mkdtemp test contexts, /tmp paths): return legacy.
}
```

The single-argument signature is preserved; behavior is determined by environment, not by extra args. This is critical so `cli.js`, the hook, and skills don't need updating beyond their sidecar-content references.

**First-write parent-directory creation:** `saveSidecar` (and any other writer) MUST `mkdirSync(dirname(target), { recursive: true })` before writing the temp file. The hidden-dir layout introduces nested paths like `.superpowers-codex-paired/docs/specs/foo.md.json`; without `mkdir -p`, `sidecar-init` fails with ENOENT on the first call for any spec whose ancestor directories don't already exist. This applies to BOTH the temp-file path AND the final renamed location (they share a parent).

**Stale-shadow prevention:** the discovery rule above explicitly handles the case where a developer has a stale `<spec>.codex.json` left over from before migration. If `hidden` exists, it wins; if only `legacy` exists, warn and use it (so we don't lose data). The migration script eliminates legacy files in the same operation that creates hidden ones.

**Symlink + sibling-prefix safety:** path containment uses `realpath` on both sides + `path.sep`-anchored prefix match (`a.startsWith(b + path.sep)` not `a.startsWith(b)`). This prevents `/repo-old/spec.md` being treated as inside `/repo`.

### Migration of existing sidecars (tracked AND ignored)

**Inventory is dynamic, not fixed.** The migration script discovers sidecars at runtime via `find . -name '*.codex.json' -not -path './node_modules/*' -not -path './.superpowers-codex-paired/*'`. Examples observed during spec authorship:
- `lib/codex-bridge/prompts/validation-rubric.md.codex.json` (tracked; 4-round v0.4 audit trail)
- `docs/specs/2026-05-08-autopilot-design.md.codex.json` (gitignored; 12-round v0.3 spec+plan review)
- Possibly `docs/specs/2026-05-08-v0.4.1-parser-and-sidecar-relocation.md.codex.json` (gitignored; this v0.4.1 review thread, may or may not exist depending on whether this current spec review created one).

The exact set on any given checkout depends on review history. The script processes whatever `find` returns; it does not hardcode the list.

**Policy: migrate ALL sidecars found on disk regardless of git-tracked status.** The audit trails are NOT reliably preserved in spec markdown or commit messages — commits record summaries, not the full round-by-round critique text. Losing them would lose review provenance. Migrating them keeps the data in the new structured location even if they remain gitignored.

Migration: a one-shot script `scripts/migrate-sidecars-to-hidden-dir.sh`. For each `*.codex.json` found via `find` (excluding `.superpowers-codex-paired/` itself and `node_modules/`), state machine:

| Source (legacy) | Destination (hidden) | Action |
|---|---|---|
| exists, git-tracked | absent | `git mv` source → dest (preserves git history) |
| exists, git-tracked | exists | error: ambiguous state, abort with helpful message |
| exists, untracked | absent | `mkdir -p dirname(dest) && mv source dest` (plain mv; untracked stays untracked) |
| exists, untracked | exists | error |
| absent | exists | already migrated; no-op |
| absent | absent | nothing to do; no-op |

The script supports `--dry-run` to print intended actions without executing.

Update markdown references that mention the literal `<spec>.codex.json` path: search for the string `.codex.json` across the repo and replace where it refers to a path the user might compute manually. Fenced code blocks showing `node cli.js sidecar-show --specPath <path>` don't need to change because the CLI auto-discovers the new layout.

Run once during v0.4.1 release. Never run again. The script halts on any ambiguous state rather than guessing.

## Failure modes / edge cases
- Existing tests pass `specPath` only (no `repoRoot`): falls back to old layout; tests pass unchanged.
- Migration script run twice: idempotent (`git mv` of already-moved file is a no-op error; check first).
- Spec moved to a new path: sidecar at the OLD path is orphaned. Out of scope for v0.4.1.
- `cli.js validation-parse` called with empty stdin: emits `tier-missing` defect.
- Critique includes prose lines that don't have `:` (e.g., a header): treated as `malformed-bullet:N`. That's correct — the parser is strict.

- All existing 31 unit tests + 10 hook tests + 1 structural smoke pass after the change.
- New `validation-coverage.test.js` has at least 13 cases covering all defect codes + happy paths.

- Existing sidecars migrated to new locations; git history preserved via `git mv` where applicable (untracked sidecars use plain `mv` and have no git history to preserve, by definition).

## This spec's own validation coverage (since `validation: critical`)

- **tier**: critical
- **happy**: parser correctly accepts a valid Phase A critique at each tier; sidecar discovery returns the hidden-dir path inside a git repo; first-write into a hidden-dir path that doesn't yet exist creates the parent directories and succeeds. Covered by `validation-coverage.test.js` happy-path cases + `sidecar.test.js` repo-context cases + first-write-mkdir test.
- **edge.zero-null-empty**: empty critique array, empty stdin, empty specPath, empty repoRoot. Covered by `not-array` / `tier-missing` / sidecar fallback tests.
- **edge.boundary**: repo root containment (sibling-prefix `/repo` vs `/repo-old`), nested specs many-levels-deep, specs exactly at repo root, specs outside any repo. Covered by sidecar tests.
- **edge.large-input**: critique array of 100+ bullets (some valid, some malformed), 1000+ migrated sidecars. Covered by stress test.
- **edge.concurrent**: reader during migration mid-flight. Mitigation: migration script is single-threaded and atomic per-file (`git mv` is one operation); concurrent reads either see legacy or hidden, not both. Documented; not tested directly.
- **edge.adversarial**: path traversal in specPath (`../../../etc/passwd`), symlinks resolving outside the repo, malformed JSON in stdin, key bullets with embedded newlines or null bytes. Covered by sidecar realpath tests + parser malformed-input tests.
- **fail.dependency**: `git rev-parse` fails (no git, corrupt repo); `node` subprocess fails; `mkdirSync` fails (perm denied) when creating the hidden-dir parents. Covered by sidecar fallback tests + CLI exit-1 path + a parent-dir-creation-failure test.
- **fail.malformed-input**: invalid JSON, non-array, non-string elements, empty values. Covered by parser tests.
- **fail.exception-path**: parser throws → CLI catches → exit 1 with stderr. Covered by CLI infrastructure-failure test.
- **integration.cross-module**: parser + CLI + autopilot SKILL.md; sidecar.js + CLI + hook + skills; migration + git history; first-write through the full sidecar-init → saveSidecar → mkdir-p chain across nested hidden paths. Covered by integration tests + autopilot structural smoke (extended to use the new layout) + first-write integration test that runs `sidecar-init --specPath docs/specs/foo.md` in a fresh git repo and asserts the hidden-dir file gets created without pre-creating parents.
- **stress.scale**: triggered (parser handles arrays; sidecar walks paths). Covered by large-input edge case above.
- **perf.slo**: not triggered (no SLO stated for parser or sidecar lookup; both are local operations expected sub-millisecond).
- **compat.breaking**: triggered. On-disk sidecar location changes; the migration script + auto-discovery preserve compatibility. Documented in CHANGELOG.
- **critical.residual-risk**: stale legacy `<spec>.codex.json` sitting on a developer's filesystem after migration could shadow a hidden-dir sidecar. Mitigation: discovery rule prefers hidden-dir; if both exist, hidden wins; if only legacy, deprecation warning to stderr. Test: create both, verify hidden is returned and warning emitted.
