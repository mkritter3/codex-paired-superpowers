# v0.6.0 — Live Verification

**Status:** double-SHIP'd (2 Codex review rounds)
**Date:** 2026-05-08
**Owner:** mkr
**Target version:** v0.6.0
**Validation:** critical
**Thread:** `019e0931-b140-7ce2-bf11-393d7510307f`

## Goal

Add a default Phase E live-verification gate to autopilot so every behavior-changing slice launches the real app, safely drives Codex-generated user-visible scenarios through Claude Code `/computer-use`, captures logs and screenshots, and fixes verified failures before the slice can ship.

## Operating Principles

Phase E is inserted after `docs-update` and before `shipped`:

```text
plan-slice -> implement -> review-slice -> docs-update -> live-verification -> shipped
```

What changes:

- Every behavior-changing slice runs live verification by default.
- A slice may opt out only with plan frontmatter `live-verification: skip` plus justification.
- Pure-library repositories may opt out at project level in `.codex-paired/project.yaml`.
- Phase E has its own 7-round budget.
- Phase E uses the same Codex, fix-subagent, reconciliation, sidecar, and double-SHIP discipline as other phases.
- `/computer-use` screen takeover is guarded by explicit user-safety policy before any UI action.
- Every scenario precondition must be enforceable by config, setup command, reset command, setup steps, or a declared blocking reason.
- After any Phase E fix commit, all scenarios for the slice are re-run.
- Same-SHA failure is retried once before fix-loop entry to distinguish likely flake from deterministic bug.
- Evidence is persisted per slice under `.superpowers-codex-paired/evidence/<slice-id>/`.

What stays the same:

- One persistent Codex thread per feature.
- One `codex-reply` in flight per feature thread.
- Sidecar remains the durable source of truth.
- Git reconciliation walks `last_commit_sha..HEAD`.
- Autopilot clears the active anchor on halt/completion.
- Out-of-scope issues go under `## Deferred`.

## Phase E Architecture

Phase E owns these sub-phases:

1. `safety-gate`
2. `scenario-gen`
3. `launch`
4. `precondition-setup`
5. `execute`
6. `flake-check`
7. `fix-loop`
8. `evidence-verdict`
9. `cleanup`

Sidecar phase name:

```json
"live-verification"
```

Phase E starts only after `docs-update.shipped == true`.

### Safety Gate

Before launching or using `/computer-use`, autopilot must evaluate the project's takeover policy.

Default policy is `confirm_each_phase_e`.

Claude must show a concise confirmation before any screen control:

```text
Phase E live verification is about to take screen control via /computer-use for ~<estimate>.
It may move the mouse, click, type, and switch focus.
Continue now?
```

Allowed responses:

- Continue: proceed.
- Cancel/postpone: halt with `live-verification-user-postponed`, leaving the anchor cleared.

Config may allow scheduled unattended windows:

```yaml
live_verification:
  takeover:
    mode: "confirm_each_phase_e | scheduled_window"
    scheduled_windows:
      - days: ["mon", "tue", "wed", "thu", "fri"]
        start: "02:00"
        end: "06:00"
        timezone: "America/Los_Angeles"
```

Rules:

- `confirm_each_phase_e` is the default.
- `scheduled_window` may skip the prompt only if current local time is inside a configured window.
- Outside the window, halt with `live-verification-outside-scheduled-window`.
- Ralph-loop resume must re-check takeover policy on every tick before `/computer-use`.
- A prior confirmation does not carry across Claude Code sessions.
- The spec does not claim Claude can detect whether the user is typing, in a call, or using another app; it prevents surprise takeover through explicit confirmation or scheduled-window policy.

### Scenario Generation

Codex receives:

- Slice id.
- Slice task list from Phase A.
- Slice diff from `slice_start_sha..HEAD`.
- Plan frontmatter UI surface.
- Project app description.
- Relevant `ui_globs`.
- Phase A `validation_coverage`.
- Precondition capabilities from project config.

Codex returns strict JSON:

```json
{
  "scenarios": [
    {
      "id": "lv-001",
      "title": "Save settings change",
      "risk": "happy-path",
      "why": "Catches user-visible save regressions",
      "preconditions": [
        { "type": "route", "value": "/settings", "enforcement": "navigate" },
        { "type": "auth", "value": "seeded_test_user", "enforcement": "project.setup.login_profile" }
      ],
      "steps": [
        { "action": "click", "target": "Settings nav item" },
        { "action": "type", "target": "Display name input", "value": "Avery" }
      ],
      "assertions": [
        "Saved display name is visible without page reload",
        "No error toast appears"
      ],
      "diagnostic_expectations": [
        "No uncaught exception in app logs",
        "No 5xx response in server logs"
      ],
      "timeout_ms": 60000
    }
  ],
  "deferred": []
}
```

Invalid scenario output (any of: missing/duplicate ids, unsupported actions, preconditions without enforceable `enforcement`, zero scenarios for a behavior-changing slice, assertions that cannot be checked from screenshot/log evidence) → REVISE within Phase E. If unresolved after 7 rounds, halt with `live-verification-scenario-deadlock`.

### Precondition Enforcement

Scenario preconditions are not comments. They are executable requirements.

Project config declares setup capabilities:

```yaml
live_verification:
  setup:
    reset_command: "npm run test:reset-db"
    seed_command: "npm run test:seed"
    login_profiles:
      seeded_test_user:
        username: "test@example.com"
        password_env: "LIVE_TEST_PASSWORD"
        login_route: "/login"
    setup_timeout_ms: 60000
```

Supported enforcement types:

- `navigate`: open configured start URL plus route.
- `reset_command`: run project reset command.
- `seed_command`: run project seed command.
- `login_profile`: use configured credentials through `/computer-use`.
- `setup_steps`: explicit UI setup actions before scenario.
- `manual_blocked`: scenario cannot run and must be deferred or rejected.

Runner behavior:

1. Before every scenario attempt, execute project reset if configured.
2. Apply scenario preconditions in listed order.
3. Capture setup logs separately from scenario logs.
4. If setup fails, mark scenario `blocked-precondition`.
5. `blocked-precondition` cannot ship unless both Claude and Codex agree the scenario is out of scope and record it under `deferred`.

This prevents Codex from diagnosing fixture/setup failures as product bugs.

### App Launch And Ready Signal

Autopilot launches the app from project config, records PID/process group, starts logs before readiness probing, and polls readiness.

Launch metadata persists:

```json
{
  "launch": {
    "command": "npm run dev",
    "pid": 12345,
    "ready": true,
    "ready_signal": "http://127.0.0.1:3000/healthz -> 200"
  }
}
```

Launch failure halts with `live-verification-launch-failure`.

### Computer-Use Integration

`/computer-use` is a Claude Code native mode, not an MCP tool or Node API.

Mechanism:

- Phase E is a foreground Claude Code operation inside the autopilot skill.
- Claude prepares scenario runbooks and evidence paths.
- Claude drives the actual app through native `/computer-use`.
- Codex never performs UI actions.
- Subagents do not own `/computer-use`.
- Sidecar checkpoints are written before and after each scenario.

Unsupported environments halt with `live-verification-computer-use-unavailable`:

- non-macOS
- Claude Code below `2.1.85`
- plan/account without `/computer-use`
- ralph tick session where `/computer-use` is unavailable

Ralph-loop requirement:

- Every ralph tick that reaches or resumes Phase E must verify `/computer-use` availability in that Claude Code session.
- Availability from the initial `/autopilot` invocation is not sufficient.
- Implementation must include an empirical smoke proving Phase E cannot silently continue when a resumed ralph session lacks `/computer-use`.

### Log Tailing

Autopilot uses bounded buffers:

```yaml
logs:
  paths:
    - "logs/dev.log"
  include_process_output: true
  max_bytes_per_source: 262144
  max_excerpt_bytes_per_scenario: 32768
  error_patterns:
    - "ERROR"
    - "Unhandled"
    - "TypeError"
    - "500"
```

Rules:

- Start tailing before launch.
- Include stdout/stderr when configured.
- Store capped per-scenario excerpts.
- Missing all diagnostic sources halts with `live-verification-log-unavailable`.
- Log paths must resolve safely under repo root unless explicitly allowed.

### Scenario Execution

For each scenario attempt:

1. Record current SHA.
2. Enforce preconditions.
3. Capture `before.png`.
4. Execute steps via `/computer-use`.
5. Capture `after.png`.
6. Capture logs.
7. Evaluate assertions.
8. Persist `result.json`.

Evidence layout:

```text
.superpowers-codex-paired/evidence/<slice-id>/
  scenarios.json
  launch.json
  summary.json
  <scenario-id>/
    attempt-1/
      before.png
      after.png
      logs.txt
      setup-logs.txt
      result.json
```

Statuses: `passed`, `failed`, `flaky`, `blocked-precondition`, `blocked-environment`, `fixed`.

### Flake Handling

A first scenario failure does not immediately enter the fix loop.

Same-SHA flake check:

1. If a scenario fails, do not modify files.
2. Re-run that same scenario once at the same `HEAD`.
3. Re-apply preconditions from scratch before retry.
4. If it fails again with materially same assertion/log failure, treat as deterministic and enter fix loop.
5. If it passes on retry, mark `flaky`.

Flaky result policy:

- `flaky` does not trigger fix-subagent automatically.
- `flaky` cannot be silently treated as pass.
- Claude and Codex must decide whether flake evidence is acceptable for this slice.
- If flake affects the changed behavior, Phase E is REVISE unless the flake is clearly environment/tool variance and documented in `deferred`.
- Repeated flakes across two scenarios halt with `live-verification-flaky-runner` because the runner is no longer trustworthy.

### Fix Loop

Failure flow:

```text
deterministic scenario failure
  -> collect screenshots/logs/diff/setup evidence
  -> Codex diagnosis
  -> Claude verifies diagnosis
  -> fix-subagent
  -> git reconciliation
  -> relaunch/reset if needed
  -> re-run all scenarios in the slice
  -> evidence verdict
```

After any Phase E fix-subagent commit:

- Re-run all scenarios in the slice.
- Do not rely on Codex to identify coupled scenarios.
- This is intentionally conservative because scenario count is bounded and user-visible regressions are the target risk.

Round budget: max 7. After round 7 without double-SHIP, halt with `live-verification-deadlock`.

### Cleanup And Halt Mode

Project config controls process cleanup on halt:

```yaml
live_verification:
  cleanup:
    on_success: "kill"
    on_halt: "kill | leave_running"
    leave_running_note: "Keep dev server alive for manual inspection"
    shutdown_command: null
```

Defaults: `on_success: kill`, `on_halt: kill`.

Rules:

- `leave_running` is opt-in.
- If `leave_running` is used, sidecar must record PID, command, port/start URL, and a suggested cleanup command.
- Autopilot must never leave a process running without making that visible in the halt summary.
- Evidence write failure always uses `kill`, because the run is not inspectable through persisted evidence.

### Evidence Verdict

After scenarios complete, Codex reviews:

- scenario list
- setup/precondition results
- pass/fail/flake matrix
- screenshots
- log excerpts
- fix commits
- final app readiness metadata
- deferred scenarios
- cleanup status

Phase E ships only on same-round double-SHIP.

## Per-Project Config

Path: `<repo>/.codex-paired/project.yaml`. JSON also allowed at `<repo>/.codex-paired/project.json`. If both, YAML wins.

Schema:

```yaml
version: 1

app:
  name: "Example Web"
  description: "User-facing web app"
  type: "web | desktop | mobile | cli | library"

live_verification:
  default: "run | skip"
  skip_reason: null

  takeover:
    mode: "confirm_each_phase_e | scheduled_window"
    scheduled_windows: []

  launch:
    command: "npm run dev"
    cwd: "."
    env: {}
    timeout_ms: 120000
    shutdown:
      signal: "SIGTERM"
      grace_ms: 5000

  ready:
    strategy: "http | stdout_regex | log_regex | fixed_wait"
    url: "http://127.0.0.1:3000"
    expected_status: 200
    timeout_ms: 120000
    poll_interval_ms: 1000

  setup:
    reset_command: null
    seed_command: null
    login_profiles: {}
    setup_timeout_ms: 60000

  logs:
    paths: []
    include_process_output: true
    max_bytes_per_source: 262144
    max_excerpt_bytes_per_scenario: 32768
    error_patterns: ["ERROR", "Unhandled", "TypeError", "500"]

  computer_use:
    required: true
    start_url: "http://127.0.0.1:3000"
    viewport:
      width: 1440
      height: 900
    scenario_timeout_ms: 60000
    max_action_retries: 2

  cleanup:
    on_success: "kill"
    on_halt: "kill"
    shutdown_command: null

  evidence:
    ui_globs:
      - "app/**/*.{tsx,jsx,css}"
      - "components/**/*.{tsx,jsx,css}"
    screenshot_format: "png"
    preserve_failure_evidence: true
    prune_pass_evidence_on_ship: true
```

Rules:

- `app.type: library` requires `live_verification.default: skip` and a non-empty `skip_reason`.
- Slice-level `live-verification: skip` overrides project `run` only with justification in plan frontmatter.
- Missing config on a behavior-changing slice halts with `live-verification-config-missing`.
- Malformed config halts with `live-verification-config-malformed`.

## Failure Modes

| Failure | Halt reason | Behavior |
|---|---|---|
| User declines takeover | `live-verification-user-postponed` | Halt before launch/computer-use; clear anchor. |
| Outside scheduled window | `live-verification-outside-scheduled-window` | Halt before screen control. |
| `/computer-use` unavailable on ralph resume | `live-verification-computer-use-unavailable` | Halt; do not assume previous session capability. |
| Missing config | `live-verification-config-missing` | Halt before launch. |
| Malformed config | `live-verification-config-malformed` | Halt with schema error. |
| Slice skip lacks justification | `live-verification-skip-malformed` | Halt before Phase E. |
| Scenario precondition cannot be enforced | `live-verification-precondition-unenforceable` | REVISE scenario or defer with evidence. |
| Setup/reset/login fails | `live-verification-precondition-failed` | Mark blocked; cannot ship unless deferred. |
| Malformed scenario JSON | `live-verification-scenario-malformed` | REVISE within Phase E. |
| Launch exits or never ready | `live-verification-launch-failure` | Capture logs; cleanup per halt mode. |
| Logs unavailable | `live-verification-log-unavailable` | Halt unless process output is available. |
| Scenario timeout | `live-verification-scenario-timeout` | Same-SHA retry once, then deterministic failure or flake. |
| Repeated action failure | `computer-use-action-failure` | Retry action, then same-SHA scenario retry. |
| App crash mid-scenario | `live-verification-app-crash` | Capture logs, cleanup/relaunch, same-SHA retry. |
| Multiple flakes make runner untrustworthy | `live-verification-flaky-runner` | Halt with evidence. |
| 7-round fix deadlock | `live-verification-deadlock` | Halt with unresolved evidence. |
| Fix commit violates conventions | `subagent-broke-commit-conventions` | Halt; cite SHA. |
| Evidence write failure | `live-verification-evidence-write-failure` | Halt and kill launched process. |

Preferred Phase E fix commit:

## Out Of Scope

Deferred from v0.6.0:

- Linux/Windows support.
- CI integration.
- Playwright fallback.
- MCP browser fallback.
- Multi-app orchestration.
- Video recording.

## Success Criteria

v0.6.0 is done when:

- Autopilot has `live-verification` between `docs-update` and `shipped`.
- Behavior-changing slices run Phase E by default.
- Takeover safety prevents surprise screen control.
- Ralph resume re-checks `/computer-use` availability per session.
- Preconditions are executable or explicitly blocked/deferred.
- Scenario failure gets one same-SHA retry before fix-loop.
- Any Phase E fix commit causes all slice scenarios to re-run.
- Cleanup policy is explicit and reflected in halt summaries.
- Evidence is persisted and pruned according to policy.
- Phase E verdict uses `live.*` keys without weakening Phase A validation parsing.
- Structural smoke proves a slice cannot reach `shipped` until Phase E ships or is validly skipped.
- A fixture app with an intentional user-visible bug is caught, fixed, rerun across all scenarios, and shipped only after evidence double-SHIP.

## This Spec's Validation Coverage (`validation: critical`)

- **tier:** critical
- **happy:** covered by the Phase E lifecycle from safety-gate through evidence-verdict and success criteria proving a real user-visible bug is caught and fixed.
- **edge.zero-null-empty:** covered by missing config, zero scenarios, missing logs, missing skip justification, missing setup capability, and unavailable computer-use handling.
- **edge.boundary:** covered by 7-round cap, launch/ready/setup/scenario timeouts, action retry max, one same-SHA flake retry, log caps, and scheduled-window boundaries.
- **edge.large-input:** covered by bounded log buffers, capped excerpts, ui_globs filtering, and sidecar summaries instead of raw evidence blobs.
- **edge.concurrent:** covered by one codex-reply mutex, foreground computer-use ownership, per-session ralph capability checks, sidecar checkpoints, and explicit process cleanup policy.
- **edge.adversarial:** covered by malformed config/scenario handling, safe log path requirements, takeover protection, precondition enforcement, and unsupported computer-use halt.
- **fail.dependency:** covered by launch failure, ready timeout, log unavailable, setup failure, computer-use unavailable, app crash, evidence write failure, and subagent convention failure modes.
- **fail.malformed-input:** covered by strict project config, strict scenario JSON, strict live-validation keys, and malformed skip/precondition halt paths.
- **fail.exception-path:** covered by named halt reasons and cleanup behavior for each unrecoverable path.
- **integration.cross-module:** covered by explicit integration across autopilot phase order, project config, sidecar schema, evidence files, Codex scenario generation, Claude computer-use execution, logs, fix-subagents, git reconciliation, ralph-loop, and validation parsing.
- **stress.scale:** triggered; covered by bounded buffers, capped evidence, and all-scenario reruns after fix commits.
- **perf.slo:** not triggered; timeouts are operational safety bounds, not product latency guarantees.
- **compat.breaking:** triggered; covered by additive `live-verification` schema and explicit skip paths rather than replacing existing phases.
- **critical.residual-risk:** strongest remaining risk is that native `/computer-use` reliability and availability may vary across Claude Code sessions; the spec requires per-ralph-tick availability checks, same-SHA flake triage, takeover gating, and an empirical smoke, but implementation still must prove this foreground-mode workflow is operable in real Claude Code sessions.
