# v0.10.0 Ecosystem Notes (operator-facing)

## 1. Namespace claim list

v0.10.0 adds the following namespaces to the plugin tree:

- `.superpowers-codex-paired/` — hidden sidecar directory (per-repo, gitignored). Contains per-spec sidecar JSON files and a `responses/` subdirectory for overflow response files (content-addressed).
- `.codex-paired/` — shared runtime directory. Contains:
  - `mailboxes/` — per-recipient JSON inboxes for inter-agent messaging
  - `diagnostics/hook-failures.jsonl` — breadcrumb log for hook infrastructure errors (redacted)
  - `active-anchor.json` — autopilot active-run state (written by autopilot, NOT read by ralph-loop)
  - `*.lock` files — proper-lockfile artifacts for sidecar + mailbox + merger-agent locking
- `.git-worktrees/` — per-slice worktrees created by the autopilot implementer fan-out

All three directories should be added to `.gitignore` for operator repos.

## 2. Sidecar reader API deferred status

The `implementer_experts` block is documented in the sidecar schema (v0.10.0). The block is written by `startImplementerRun`, `appendImplementerEventLocked`, and `completeImplementerRun`.

An external reader API (for tooling outside this plugin) is deferred to v0.11+. Current consumers use `readImplementerRun(specPath, sliceId)` from `lib/codex-bridge/sidecar.js`.

`replayImplementerEvents(sidecar, opts)` in `lib/codex-bridge/replay.js` is the canonical way to extract and sort events from a loaded sidecar object.

## 3. Ralph-loop one-sided coupling

The autopilot writes to `.codex-paired/active-anchor.json` to record the active run state. Ralph-loop is independent and does NOT read this file — it only observes the `/autopilot` command's halt envelope output.

This is intentional: ralph-loop is a thin restart wrapper that reads halt envelopes (transient vs terminal) and decides whether to re-fire. The active-anchor is an autopilot-internal mechanism for cross-session continuity.

## 4. Feature-dev coexistence

v0.10.0 is opt-in via the `**Implementers:**` frontmatter block in plan slices. Slices without this block follow the single-implementer path unchanged.

The feature-dev plugin's flow remains unchanged: it doesn't interact with `implementer_experts` blocks, mailboxes for implementer recipients, or worktree fan-out logic.

Operators using both plugins simultaneously: the `.codex-paired/` directory is shared, but mailbox recipients are namespaced (`impl-*` for implementers, `slice-N` for the autopilot orchestrator, `orchestrator` for the outer loop). No collisions.

## 5. Commit/PR attribution

Per-implementer blame surfaces in `git log` and in sidecar audit events:
- Each implementer commits to their own worktree branch with their `member_id` in the commit trailer (`Implementer-Member-Id: ...`)
- Sidecar `started`, `checkpoint`, `completed` events carry `member_id` + `runtime_kind`

What does NOT surface in the PR UI:
- GitHub PRs show only the final merge author (the integration commit author). Per-implementer attribution is visible only via `git log` on the integration branch or by reading the sidecar.

## 6. Future-grep policy

Every v0.11+ feature spec MUST include an installed-plugin grep result to catch feature drift early. See `docs/integration/future-grep-policy.md` for the canonical commands.

### Diagnostic writer limitation note

The `writeBreadcrumb` function in `lib/codex-bridge/hook-mailbox-inject.js` is the only JS-level diagnostic writer. It now redacts secret patterns (via `redactSecretFields`) before persisting to `hook-failures.jsonl`.

Shell scripts that might write to `.codex-paired/diagnostics/` directly (if any) are outside the scope of JS-level redaction. As of v0.10.0, no shell scripts write directly to `hook-failures.jsonl` — the only writer is `writeBreadcrumb` in the JS module. If shell-level writers are added in future versions, they must implement their own redaction or pipe through a JS wrapper.
