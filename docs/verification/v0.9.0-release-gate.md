`composeExperts({phase, signals, repoRoot})` must select at minimum two distinct
role IDs given a typical feature spec's signals. This validates that role composition
actually fans out across multiple advisors rather than collapsing to a single pass.

**How to verify:**  
1. Run the gate script: `./scripts/v0.9.0-release-gate.sh`  
   The script invokes `composeExperts({phase: 'spec-review', signals: {specHas: [...], domains: [...]}, repoRoot})` and logs `result.selected[].id`.  
2. Manually: load a spec via `brainstorming` or `writing-plans` and observe the
   expert-turn log (or sidecar `role_sessions` entries). Count distinct `requested_role`
   values across the top-level dispatch.

   ```bash
   node -e "import('./lib/codex-bridge/role-composer.js').then(m=>console.log(m.composeExperts({phase:'spec-review', signals:{specHas:['auth','credential'], domains:['security']}, repoRoot: process.cwd()}).selected.map(r=>r.id)))"
   ```  

   Expect array with length ≥ 2.

**Description:**

roles must dispatch to different CLIs (e.g., `codex` for one role and `claude` or
`ollama` for another). The sidecar `resolved_cli` field must differ across role turns.
This confirms the routing ladder works — not that every role picks a unique CLI, but
that the system is not trivially pinned to a single adapter.

   as "available" so the routing ladder has candidates to choose from.

**Evidence:**

## Criterion 4 — Peer DM round-tripped across adapters

**Description:**  
At least one peer direct-message must be enqueued by one expert role and consumed
by a different role. The sidecar `mailbox_message_ids` for the receiving turn must
reference the sent message ID. This proves the mailbox injection hook fires correctly
in worktree subagent contexts and that cross-adapter DM delivery is not silently
dropped.

**How to verify:**  
1. Run `./scripts/v0.9.0-release-gate.sh` — the runner inspects every turn in the
   discovered sidecar, filters for turns with `mailbox_message_ids[]` non-empty,

   size ≥ 2 (cross-adapter round-trip proof). If turns have DMs but only one

## Criterion 5 — Double-SHIP convergence within 7-round budget on a real spec

**Description:**  
Running `autopilot` (or the equivalent `writing-plans` → `subagent-driven-development`
cycle) on the bundled test spec must reach a SHIP verdict from both Claude and the
routed expert CLI within at most 7 rounds. Both advisors must emit `SHIP` (or the
equivalent approval verdict) before the round budget is exhausted. The sidecar must
record the final `verdict: "SHIP"` per-round entry.

**How to verify:**  
1. Run `./scripts/v0.9.0-release-gate.sh` — the runner does NOT spawn its own
   autopilot pass (that would require live CLIs + an interactive driver). It
   inspects the discovered sidecar's `rounds[]` and looks for the first round
   where both `claude` and `codex` verdicts start with `SHIP`. PASS if such a
   round is found within the first 7 entries; FAIL if 7+ rounds exist without
   convergence; PENDING if the sidecar has no rounds yet.  
2. To produce the sidecar evidence: run `brainstorming` or `autopilot` on a
   small spec first; then re-run the gate runner (it picks up the most-recent
   sidecar automatically).  
3. Manually verify convergence detail:  
   ```bash
   cat .superpowers-codex-paired/path/to/spec.md.json | \
     jq '[.rounds[] | {round, claude, codex}]'
   ```  
   The first round where both verdicts start with `SHIP` is the convergence
   round; its index must be ≤ 6 (0-indexed) for PASS.  
4. If convergence fails within 7 rounds: the gate FAILS (do not adjust the budget
   without a documented justification reviewed on the Codex thread).

**Result:** PASS

**Evidence:**

## Criterion 6 — Replay reconstructs identical inputs_hash from sidecar

**Description:**  
Given any completed expert-turn sidecar entry, `replayTurn(turn, deps)` must
reconstruct the assembled prompt and produce an `inputs_hash` that is byte-identical
to the recorded value. "Byte-identical" means exact string equality, not just hash
collision resistance. This validates the stateless-replay invariant that makes
sidecar entries the canonical source of truth for audit.

**How to verify:**  
1. Run `./scripts/v0.9.0-release-gate.sh` — after completing a round, it calls
   `replayTurn` against the first expert turn and asserts `result.inputsHashMatches === true`
   AND that `result.assembledPrompt` contains the same role ID, role-prompt content,
   and spec snippet that were recorded.  
2. Manually: run `node --test tests/replay/replay-from-sidecar.test.js` — all
   replay tests must pass including the byte-identity assertion.  
3. Cross-CLI variant: record a turn under the `codex` adapter, then replay under
   `claude-task`. The `inputs_hash` must still match (adapter is not in the hash
   domain); only a warning about adapter mismatch should appear, not a FAIL.

**Result:** PASS

**Evidence:**
