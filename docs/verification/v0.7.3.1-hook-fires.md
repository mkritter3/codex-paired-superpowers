# v0.7.3.1 — PostToolUse hook fires in Task subagent (release gate)

## Status

**INCONCLUSIVE (2026-05-10)** — the Task tool in Claude Code 2.1.138 does not expose a `cwd` parameter, so the load-bearing claim (PostToolUse hook stdin's `cwd` reflects the subagent's working directory) cannot be empirically validated from this dispatch surface. The hook works correctly when invoked directly; the architecture remains plausibly correct in production autopilot use, where Claude Code internally drives Task dispatch with subagent-specific contexts not exposed to user-level Task calls. Re-run this verification once a Claude Code version exposes a Task `cwd` parameter, or once an autopilot run can be observed empirically with `CPS_HOOK_DEBUG=1`.

### Empirical findings (2026-05-10)

**Direct hook invocation works.** Piping a fixture stdin with `cwd: <slice-99-worktree>` directly to `hooks/mailbox-inject.sh` produces the spec'd `additionalContext` JSON and marks the seeded messages read in the mailbox. Doctor checks all 8 pass. So the hook plumbing itself is correct.

**Task-subagent dispatch did not deliver.** Dispatched a `general-purpose` Task subagent with a Bash command `cd <slice-99-worktree> && echo verify`. After the dispatch:
- Subagent reported no `<codex-paired-pending-messages>` in its context.
- Mailbox messages remained unread.
- No breadcrumbs written (so the hook either didn't fire at all, OR it fired but `cwd` resolved to the orchestrator's cwd and identity inference correctly no-op'd).

**Most likely cause.** The Task tool exposed in Claude Code 2.1.138 doesn't have a `cwd` parameter. A prose `cd <worktree>` inside the subagent prompt only changes the subagent's bash subshell cwd — Claude Code's PostToolUse hook stdin reports the parent/orchestrator cwd, not the subagent's bash cwd. The hook's identity inference correctly determined "not in a slice worktree" and silently no-op'd.

**What this means for v0.7.3.1.** The hook implementation is correct and tested (35 unit tests + 3 smoke suites green on both macOS and Linux). The plumbing through Claude Code's runtime cannot be empirically confirmed from this session because the dispatch surface doesn't expose the load-bearing parameter. Production autopilot use (which drives Task dispatch internally) may exercise a different codepath that DOES set the subagent's cwd correctly — that would manifest empirically when the autopilot runs against a real slice.

**Recommended next test.** During the next autopilot run against a slice-bearing plan, set `CPS_HOOK_DEBUG=1` in the Claude Code environment. The hook debug output will appear in Claude Code's hook diagnostics stream and reveal whether the autopilot's internal Task dispatch passes the subagent's cwd. If yes → mark this gate PASS. If no → file an upstream issue / consider pre-injection-only fallback per spec §9.3.

## Why this gate exists

Spec §5.2 architecture relies on a runtime claim: that Claude Code fires
PostToolUse hooks INSIDE Task-tool subagents and that the hook stdin's
`cwd` field reflects the subagent's worktree-rooted working directory
(not the orchestrator's cwd). Per spec §9.3 + Codex round-1 critique, this
claim must be empirically verified end-to-end before the v0.7.3.1 hook
path can be released.

If the claim is wrong:
- Revert spec §5 (the hook architecture).
- Ship pre-injection (§4) only as v0.7.3.1.
- Document the hook approach as a v0.7.3.2 R&D item.

## Procedure

Run the scaffolding script and follow its printed steps:

```bash
plugins/codex-paired-superpowers/tests/smoke/hook-fires-in-task-subagent.sh
```

The script creates a temp repo with a fake worktree at
`<tmp>/.git-worktrees/slice-99/`, pre-seeds two messages into the
slice-99 mailbox, and prints the exact Task tool invocation to issue
from a Claude Code session running this plugin.

The release gate is met when:

1. The dispatched subagent's transcript contains a system-injected
   `<codex-paired-pending-messages recipient="slice-99">` block listing
   both pre-seeded message ids.
2. After the subagent returns, both messages are marked read in the
   mailbox (`mailbox-read --unread` returns `[]`; `mailbox-read` returns
   both messages with non-null `read_at`).
3. A second Task subagent dispatched into the same worktree with a
   trivial Bash command receives NO `<codex-paired-pending-messages>`
   block (already-read messages must not re-deliver).
