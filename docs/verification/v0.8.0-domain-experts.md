## Why this gate exists

The v0.8.0 domain-experts feature ships an executable runtime pipeline
(`assembleSpawnPrompt` → Agent dispatch → `parseExpertOutput` →
`markManyAsRead` → `appendExpertTurn`). The release gate verifies this
pipeline in **two complementary parts**:

1. **Automated runtime smoke** (`tests/smoke/domain-experts-end-to-end.sh`)
   exercises every step EXCEPT the Agent dispatch, which is mocked via
   the DI seam returning a canned valid `## Machine Result` block. This
   smoke is run during CI / pre-tag verification and currently PASSES.
   It proves the runtime pipeline is correct end-to-end given a working
   Agent response.

2. **Manual live-Agent verification** (this document) demonstrates that
   the real Claude Code Agent tool, dispatched by an orchestrator
   following the two-step pattern documented in
   `skills/autopilot/SKILL.md` §B.1.5 / §B.5.5, produces a valid
   `Machine Result` response from a Sonnet subagent given a spawn
   prompt assembled by the production runtime.

Why two parts: Claude Code's Agent / Task tool is exposed only inside
a live Claude Code session, not from a Node module API. The runtime's
`defaultAgentDispatch` is therefore intentionally unwired — see
`lib/codex-bridge/expert-turn.js` (the throw message explicitly directs
callers to the two-step pattern). The automated smoke proves the
pipeline's correctness; this manual procedure proves the integration
with the live Agent surface.

**Combined acceptance:** both parts must PASS before tagging v0.8.0.
The automated smoke must run cleanly; the manual procedure below must
produce a transcript that meets the acceptance criteria.

## Acceptance criteria

The release gate is met when **all four** hold after running the smoke:

1. **Mailbox delivery.** The 2 pre-seeded unread messages in
   `expert-ui`'s inbox are visible to the dispatched subagent (either
   via spawn-prompt rehydration or via the PostToolUse mailbox-inject
   hook — `runTurn()` rehydrates explicitly through the spawn prompt,
   so the subagent must repeat back both ids in its transcript).
2. **Machine Result parses.** The subagent emits a `## Machine Result`
   block matching the parser contract (`verdict`, `scope`,
   `blocking_findings[]`, `nonblocking_findings[]`,
   `peer_messages_sent[]`, `questions_for_orchestrator[]`). For the
   release-gate fixture, `verdict` is `SHIP` and all arrays are empty.
3. **Mark-read.** After the dispatch returns, `mailbox-read --unread`
   for `expert-ui` returns `[]` and `mailbox-read` returns both
   messages with non-null `read_at`. (In a production autopilot run,
   `runTurn()`'s terminal-result handler owns mark-read via
   `markManyAsRead`; for the fixture the subagent invokes
   `mailbox-mark-read-batch` directly.)
4. **Sidecar turn record.** A turn record is appended to
   `expert_teammates.turns[]` for `expert-ui` with non-null
   `verdict` (`SHIP` or `REVISE`) and the 2 pre-seeded ids present in
   `mailbox_message_ids_injected` (the slice 4 sidecar field name —
   NOT `injected_message_ids`, which is the v0.7.3.1 dispatch-record
   field). Validated by `appendExpertTurn` per slice 4. Executable
   assertion:

   ```bash
   # After the Task subagent returns, inspect the sidecar:
   SIDECAR=$(node "$PLUGIN_ROOT/lib/codex-bridge/cli.js" sidecar-path --specPath "$SPEC_PATH")
   node -e "
     const sc = JSON.parse(require('fs').readFileSync('$SIDECAR','utf8'));
     const turn = sc.expert_teammates?.turns?.[0];
     if (!turn) { console.error('FAIL: no turn record'); process.exit(1); }
     if (turn.expert_id !== 'expert-ui') { console.error('FAIL: wrong expert_id'); process.exit(1); }
     if (!['SHIP','REVISE'].includes(turn.verdict)) { console.error('FAIL: bad verdict ' + turn.verdict); process.exit(1); }
     const inj = turn.mailbox_message_ids_injected;
     if (!Array.isArray(inj) || inj.length !== 2) { console.error('FAIL: bad mailbox_message_ids_injected: ' + JSON.stringify(inj)); process.exit(1); }
     console.log('PASS: sidecar turn record valid');
   "
   ```

### Part 2 — Manual live-Agent verification (must PASS before tagging)

From a Claude Code session loaded with this plugin:

1. **Set up a fresh fixture** (a tmp repo with `.codex-paired/`, a spec
   stub, and 2 pre-seeded unread messages for `expert-ui`). You can
   adapt the fixture-setup portion of `domain-experts-end-to-end.sh`,
   or hand-roll it via the production CLI:

   ```bash
   TMP_ROOT=$(mktemp -d)
   mkdir -p "$TMP_ROOT/.codex-paired"
   echo "# spec" > "$TMP_ROOT/spec.md"
   PLUGIN=$(claude plugin marketplace list ... )  # adapt as needed
   node "$PLUGIN/lib/codex-bridge/cli.js" sidecar-init --specPath "$TMP_ROOT/spec.md" --feature live --threadId stub
   ID1=$(node "$PLUGIN/lib/codex-bridge/cli.js" mailbox-write --to expert-ui --from orchestrator --text "live-Agent verification msg 1" --repoRoot "$TMP_ROOT" | node -pe "JSON.parse(require('fs').readFileSync(0,'utf8')).id")
   ID2=$(node "$PLUGIN/lib/codex-bridge/cli.js" mailbox-write --to expert-ui --from expert-ux --text "live-Agent verification msg 2" --repoRoot "$TMP_ROOT" | node -pe "JSON.parse(require('fs').readFileSync(0,'utf8')).id")
   ```

2. **Drive the two-step orchestration pattern** documented in
   `skills/autopilot/SKILL.md` §Phase B.1.5 / §B.5.5:
   - Resolve `expert-ui` identity via `expert-resolver.resolveIdentity('ui', TMP_ROOT)`.
   - Read unread via `mailbox.readUnreadMessages`.
   - Assemble the spawn prompt via `expert-turn.assembleSpawnPrompt(request)`.
   - **Dispatch the Task tool yourself** with that prompt as the
     subagent prompt. Capture the subagent's final response text.
   - Call `runTurnWithDeps(request, { agentDispatch: async () => taskResponseText })`.

3. **Verify acceptance criteria 1-4** against the resulting state
   (unread mailbox empty, sidecar turn record has both ids in
   `mailbox_message_ids_injected`, etc.). The Bash assertion snippet
   under acceptance criterion 4 above is executable.

## Fallback (if FAIL)

Per slice 7 plan + spec §9.3:

- If the Task surface fundamentally precludes the dispatch in the
  current Claude Code version (e.g., subagent context cannot be
  addressed from a user-driven Task call), document the limitation
  here and **HOLD** the v0.8.0 release pending remediation.
- Remediation options to evaluate before re-running:
  1. Wire a native Agent-tool adapter once the Claude Code SDK exposes
     a programmatic dispatch surface (currently in
     [Deferred](../plans/2026-05-10-domain-experts.md#deferred-out-of-v080-scope)).
  2. Document the runtime as Claude-driven only (orchestrator dispatches
     experts from inside Claude Code, never from Node scripts) and
     re-run the gate from inside a live autopilot session against the
     fixture.
  3. Roll back the runtime facade's `runTurn` export to a stub and
     ship v0.8.0 as a "definition-only" preview release (NOT
     recommended — the load-bearing claim is the whole point of v0.8.0).
- If FAIL, capture the failure mode in detail under "Notes / anomalies"
  below so the next maintainer has enough context to triage.
