## Why this gate exists

Spec §9.3 and slice 7 of the v0.8.0 domain-experts implementation plan
both make the same release-blocking claim: that
`lib/codex-bridge/expert-runtime.js`'s `runTurn(...)` correctly drives a
real Agent subagent end-to-end. Per round-1 critique on the plan,
**INCONCLUSIVE is not acceptable for v0.8.0 ship** — the load-bearing
claim must be empirically demonstrated, not deduced from unit tests
alone.

The Agent / Task tool surface is exposed only inside a live Claude Code
session, not from a plain Node script. So this gate is a manual user
procedure (analogous to v0.7.3.1's `hook-fires-in-task-subagent.sh`):
the smoke script sets up the fixture and prints the exact Task tool
invocation the maintainer must issue from a Claude Code session.

## Acceptance criteria

The release gate is met when **all four** hold after running the smoke:

1. **Mailbox delivery.** The 2 pre-seeded unread messages in
   `expert-ui`'s inbox are visible to the dispatched subagent (either
   via spawn-prompt rehydration or via the PostToolUse mailbox-inject
   hook — `runTurn()` rehydrates explicitly through the spawn prompt,
   so the subagent must repeat back both ids in its transcript).
2. **Machine Result parses.** The subagent emits a `## Machine Result`
   block matching the parser contract (`verdict`, `scope`,
   `blocking_findings[]`, `nonblocking_findings[]`,
   `peer_messages_sent[]`, `questions_for_orchestrator[]`). For the
   release-gate fixture, `verdict` is `SHIP` and all arrays are empty.
3. **Mark-read.** After the dispatch returns, `mailbox-read --unread`
   for `expert-ui` returns `[]` and `mailbox-read` returns both
   messages with non-null `read_at`. (In a production autopilot run,
   `runTurn()`'s terminal-result handler owns mark-read via
   `markManyAsRead`; for the fixture the subagent invokes
   `mailbox-mark-read-batch` directly.)
4. **Sidecar turn record.** A turn record is appended to
   `expert_teammates.turns[]` for `expert-ui` with non-null
   `verdict` and the 2 pre-seeded ids present in `injected_message_ids`
   (validated by `appendExpertTurn` per slice 4). The fixture spec
   path lives at `<tmp>/spec.md`; the sidecar file lives next to it
   per `sidecarPathFor()`.

## Procedure

Run the scaffolding script and follow its printed steps:

```bash
plugins/codex-paired-superpowers/tests/smoke/domain-experts-end-to-end.sh
```

The script:

- Creates a temp repo with `.codex-paired/` and a minimal fixture spec.
- Pre-seeds 2 unread messages into `expert-ui`'s mailbox via the
  production CLI (`mailbox-write --to expert-ui --from orchestrator`).
- Writes a Node entry script (`run-expert-turn.mjs`) that imports
  `expert-runtime.runTurn(...)` for the fixture identity, so a
  maintainer who prefers to dispatch the Agent indirectly (via Task →
  subagent invokes Node entry) has that path available.
- Prints the exact Task tool invocation to issue, plus the
  post-dispatch assertion commands.

## Fallback (if FAIL)

Per slice 7 plan + spec §9.3:

- If the Task surface fundamentally precludes the dispatch in the
  current Claude Code version (e.g., subagent context cannot be
  addressed from a user-driven Task call), document the limitation
  here and **HOLD** the v0.8.0 release pending remediation.
- Remediation options to evaluate before re-running:
  1. Wire a native Agent-tool adapter once the Claude Code SDK exposes
     a programmatic dispatch surface (currently in
     [Deferred](../plans/2026-05-10-domain-experts.md#deferred-out-of-v080-scope)).
  2. Document the runtime as Claude-driven only (orchestrator dispatches
     experts from inside Claude Code, never from Node scripts) and
     re-run the gate from inside a live autopilot session against the
     fixture.
  3. Roll back the runtime facade's `runTurn` export to a stub and
     ship v0.8.0 as a "definition-only" preview release (NOT
     recommended — the load-bearing claim is the whole point of v0.8.0).
- If FAIL, capture the failure mode in detail under "Notes / anomalies"
  below so the next maintainer has enough context to triage.
