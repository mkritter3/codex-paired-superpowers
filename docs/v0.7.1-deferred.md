## 1. Migrate Sonnet dispatch to agent teams

**Problem:** v0.7.0's Sonnet path uses the `Task` tool with `subagent_type: general-purpose` (or `slice-implementer-sonnet`). Empirical measurement during release showed `Task` subagent dispatches serialize at the runtime layer â€” two parallel calls in a single assistant turn yielded ~1.7Ă— wall-clock vs single-call baseline, not the 1Ă— the architecture's parallel claim assumes.

**Solution:** Claude Code's experimental "agent teams" feature (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`) spawns separate Claude Code sessions per teammate, achieving true wall-clock parallelism. Each teammate has its own context window and makes independent Claude API calls.

**Migration path:**
- SKILL.md Phase B: when dispatching multiple Sonnet slices in parallel, spawn them as agent team members instead of `Task` tool calls.
- Reference existing `agents/slice-implementer-sonnet.md` as a teammate template.
- Coordination via shared task list + mailbox (built into agent teams).

**Out of scope for v0.7.1 first cut:** session resumption with in-process teammates is not yet supported; some task-status lag is documented. Agent teams remains experimental â€” v0.7.1 should add this as an opt-in path, not replace the Task-based path entirely.

**References:**
- https://code.claude.com/docs/en/agent-teams
- Claude Code v2.1.32+ required.

## 2. Domain-aware routing policy

**Problem:** v0.7.0's routing only considers `**Implementer:**` directive (codex / sonnet). It does not enforce a domain-based safety policy.

**User policy:**
- UI / UX work â†’ ALWAYS Claude (Codex has weak UI/UX judgment)
- AI harness work (autopilot orchestrator, hooks, MCP servers, Codex itself) â†’ ALWAYS Claude (Codex shouldn't write the systems that govern it)
- Backend / pure logic / data plumbing â†’ Codex acceptable
- Mixed / unclear â†’ Claude (default safe)

**Solution:** SKILL.md pre-dispatch checklist gains a domain check:
- Plan frontmatter optionally declares `**Domain:** ui | ai-harness | backend | general` per slice.
- If `Domain: ui` or `Domain: ai-harness`, force Sonnet/Claude regardless of `**Implementer:**` value.
- If `Domain` is absent, Claude inspects the slice's `**Files:**` paths and infers: paths under `web/`, `app/`, `src/components/`, `*.tsx`, `*.css`, `*.html`, `frontend/` â†’ ui. Paths under `skills/`, `agents/`, `hooks/`, `lib/codex-bridge/`, MCP server code â†’ ai-harness. Otherwise general/backend.
- The orchestrator halts with `domain-policy-violation` if the explicit `**Implementer:** codex` clashes with an inferred or declared `ui` / `ai-harness` domain. User must fix the plan.

**Default flip:** consider whether to flip the default from Codex to Claude. v0.7.0 defaults to Codex on absent directive. Given the policy ("Codex okay for backend, Claude for everything else"), Claude is the safer default. Plan author opts into Codex per slice for backend work.

## 4. Real-time slice_start_sha capture

v0.7.0 added a defensive check inside `slice-implementer-codex.md` that worktree HEAD matches the supplied `slice_start_sha`. This catches the bug where the orchestrator passes a stale SHA. Better fix: the orchestrator (SKILL.md) should capture `slice_start_sha = git rev-parse HEAD` IMMEDIATELY before each `git worktree add`, never reusing a stale value from earlier in the autopilot run. Document this requirement explicitly in SKILL.md Phase B prose. Defensive subagent check stays as belt-and-suspenders.
