# v0.10.0 Implementer Experts Design

## Headline

v0.10.0 adds opt-in, write-capable implementer experts to the `codex-paired-superpowers` v0.9.x dispatcher so one slice can run multiple isolated implementation agents in parallel. The design keeps reviewer-only panels unchanged, gives each implementer its own git worktree, routes peer communication through the existing mailbox primitive, and merges only through a controlled integration branch. The goal is faster implementation without weakening sidecar audit, secret handling, halt envelopes, replay, or human-reviewed merge gates.

## Background: v0.9.x Architecture State

The plugin already has the primitives needed for a conservative implementation:

- `lib/codex-bridge/cli-harness/adapters/codex.js` spawns Codex for reviewer-style CLI dispatch and includes v0.9.1 process hardening.
- `lib/codex-bridge/cli-harness/adapters/ollama.js` provides the existing Ollama Cloud CLI harness adapter shape.
- `lib/codex-bridge/cli-harness/harness.js` owns normalized CLI dispatch.
- `lib/codex-bridge/panel/dispatcher.js` fans out reviewer panel members in parallel and aggregates through panel contracts.
- `lib/codex-bridge/sidecar.js` owns sidecar persistence, migrations, append validators, response storage, `inputs_hash`, and cross-process locking.
- `lib/codex-bridge/mailbox.js` owns the file-based peer DM primitive under `.codex-paired/mailboxes/`.
- `lib/codex-bridge/expert-turn.js` owns `runTurnWithDeps`, mailbox injection, parsing, and expert turn persistence.
- `lib/codex-bridge/halt-envelope.js` owns terminal halt known-set behavior.
- `lib/codex-bridge/replay.js` reconstructs replay prompts from sidecar fields.
- `skills/autopilot/SKILL.md` defines the autopilot phases.
- `skills/writing-plans/SKILL.md` defines plan/frontmatter conventions.
- `tests/codex-bridge/**`, `tests/replay/replay-from-sidecar.test.js`, and `tests/installed-smoke/**` provide the current proof surfaces.

The gap is that these are mainly reviewer and single-implementer paths. Existing implement metadata in `sidecar.js` (`phases.implement`, `appendImplementDispatch`) remains the legacy single-implementer path and should not be overloaded with concurrent implementer event streams.

## The Gap: Implementer Experts

v0.10.0 must add a distinct, opt-in execution mode:

- Implementers receive workspace-write authority.
- Each implementer edits in an isolated git worktree, never directly in the operator's current checkout.
- Multiple instances of the same model must be callable in one slice.
- Mixed providers must share one dispatch surface.
- Agents must communicate while a slice is still executing, not only after all children finish.
- Read-only reviewers and reviewer panels must not regress.
- The merged result receives one post-merge paired review from Codex and Claude.
- Merge conflicts may be attempted by an AI merger, but the merger output cannot land unless Claude orchestrator and Codex independently return SHIP.

This mode activates only when a slice declares an explicit `Implementers` block. Slices without that block keep v0.9.x behavior.

## Architecture: CLI Adapter Unification

Do not copy the Codex adapter into a Claude adapter wholesale. Codex and Claude have different protocols, sandbox flags, stream formats, auth env, and timeout surfaces. The right seam is a shared implementer subprocess contract with adapter-specific codecs.

```js
 * @property {string} memberId

 * @property {number|null} exitCode
 * @property {string|null} headSha
 * @property {string|null} diffHash
 * @property {string[]} changedFiles
 * @property {object|null} haltEnvelope

```

`codex-cli`:

- Uses `codex exec --sandbox workspace-write -C <worktree>`.
- Must preserve the v0.9.1 timeout, SIGKILL escalation, process-group reap, and SIGINT propagation already present in `adapters/codex.js`.
- Should be implemented as an implementer mode in the Codex adapter, not as a reviewer-path mutation.

`claude-cli`:

- Uses `claude --model <model>` with `cwd` set to the implementer worktree.
- Receives only sanitized env.
- Must produce a stable machine-readable result. If the installed CLI cannot provide a supported stream/result shape, halt with `claude-cli-protocol-unsupported`.
- Is distinct from any Claude Task/subagent path. Runtime kind is `claude-cli`, not `claude-task`.

The shared dispatcher should normalize results, but adapters own protocol parsing.

## Member-ID Disambiguation: N-of-Same-Model

Member IDs use:

```text
<role_id>@<cli_kind>:<model_id>#<ordinal>
```

Examples:

```text
expert-implementer@claude:kimi-k2.6:cloud#0
expert-implementer@claude:kimi-k2.6:cloud#1
expert-implementer@claude:kimi-k2.6:cloud#2
expert-implementer@codex:gpt-5.5#0
expert-implementer@claude:glm-4.7:cloud#0
```

Parsing rules:

- Split at the last `#` for ordinal.
- Split the pre-ordinal portion at the first `@`.
- Split runtime/model at the first `:`.
- Treat the remaining model string as opaque; model IDs may contain `:`.

Worktree paths must not use raw member IDs directly. Use a slug plus hash:

```text
.codex-paired/worktrees/v0.10.0-implementer-experts/expert-implementer-claude-kimi-k2-6-cloud-0-<hash>/
```

## Plan Frontmatter: Implementers, Files, high_cost

Implementer mode is enabled only by an explicit `Implementers` block.

Top frontmatter:

```yaml
---
slice_id: v0.10.0-implementer-experts
phase: implementation
high_cost: false
high_cost_rationale: ""
---
```

Machine-readable implementer block:

```markdown
**Implementers:**
- member_id: expert-implementer@claude:kimi-k2.6:cloud#0
  adapter: claude-cli
  model: kimi-k2.6:cloud
  required: true
  files:
    - lib/codex-bridge/cli-harness/adapters/claude-cli.js
    - tests/codex-bridge/cli-harness/adapters/claude-cli.test.js
- member_id: expert-implementer@codex:gpt-5.5#0
  adapter: codex-cli
  model: gpt-5.5
  required: true
  files:
    - lib/codex-bridge/implementer/worktree-fanout.js
    - tests/codex-bridge/implementer/worktree-fanout.test.js
- member_id: expert-implementer@claude:glm-4.7:cloud#0
  adapter: claude-cli
  model: glm-4.7:cloud
  required: false
  files:
    - lib/codex-bridge/implementer/merge-coordinator.js
    - tests/codex-bridge/implementer/merge-coordinator.test.js
```

Rules:

- Default cap is 3 implementers.
- 4 or 5 implementers require `high_cost: true` and a concrete non-empty `high_cost_rationale`.
- More than 5 implementers halts with the existing fan-out protection extended to implementers: `role-composer-fan-out-unjustified`.
- Every implementer needs at least one claimed file or directory glob.
- Claimed file partitions may overlap only when each overlapping member has `overlap_rationale`.
- Diff validation rejects edits outside claimed files unless the path is in a slice-level `shared_files` allowlist.

## Worktree Fan-Out and Parallel Dispatch

Autopilot Phase B becomes an integration coordinator for implementer-expert slices:

1. Record `base_sha`.
2. Generate a new `implementer_run_id` UUID for this slice execution.
3. Validate `Implementers`, cap, member IDs, claimed files, overlap rationale, and `high_cost`.
4. Create one branch and worktree per implementer:

```text
implementer/<slice_id>/<member_slug>
```

5. Dispatch all implementers concurrently.
6. Keep mailbox storage centralized under `.codex-paired/mailboxes/`; do not copy inbox state into worktrees.
7. On completion, capture `head_sha`, `diff_hash`, `changed_files`, claimed-file violations, transcript handle, and halt envelope.
8. If any required implementer halts, cancel active siblings and halt the slice.
9. If all required implementers complete, merge branches into an integration worktree in deterministic member-ID order.
10. Run post-merge paired review.
11. Expose the merged diff as the candidate only after both reviewers SHIP.

Merge strategy: `git merge --no-ff --no-commit` per branch in an integration worktree.

Rejected alternatives:

- `git apply` loses branch ancestry and handles renames/modes worse.
- Rebase rewrites implementer history and weakens failure forensics.
- Direct edits in the main worktree break isolation and dirty-worktree discipline.

## Mailbox During Execution

Use the existing mailbox primitive in `lib/codex-bridge/mailbox.js`. Do not introduce a second chat bus.

Hard truth: Codex CLI and Claude CLI cannot be safely interrupted mid-token to inject a peer DM. v0.10.0 defines "during execution" as delivery at adapter-observable checkpoints while sibling implementer processes are still alive.

Delivery mechanism:

- Each implementer receives mailbox instructions in its prompt.
- `lib/codex-bridge/implementer/mailbox-poller.js` wraps nonblocking `readUnreadMessages` and `markAsRead` calls.
- Supervisor polls:
  - at process start
  - after adapter-observable tool/edit/test/checkpoint events
  - before declaring completion
  - on a timed cadence while the process is alive
- Polling never blocks waiting for a message.
- New messages are injected through the adapter-native continuation/checkpoint channel where supported.
- If adapter-native mid-run injection is unavailable, messages are surfaced at the next continuation/checkpoint prompt.
- Agents must poll before completion and after handling any peer request.

Cadence:

- Default timed cadence: 45 seconds.
- Jitter: +/- 10 seconds.
- Urgent messages are delivered at the next checkpoint.
- Empty inbox polls are cheap and nonblocking.

Mailbox schema extension is intentionally minimal. Existing message records already carry `id`, `from`, `to`, `text`, `timestamp`, `summary`, `color`, `read_at`. v0.10.0 adds optional fields accepted by `mailbox.js` validation:

```js
{
  kind: "message" | "question" | "answer" | "handoff" | "merge_note" | "shutdown_request",
  priority: "normal" | "urgent",
  implementer_run_id: "<uuid>",
  slice_id: "<slice-id>",
  body_hash: "sha256:<64-hex>"
}
```

Back-compat: old mailbox messages without these fields remain valid.

## Merger Agent and Double-SHIP Gate

Normal path:

- Integration worktree merges implementer branches in member-ID order.
- If no conflict occurs, commit the merge result in the integration branch.
- Run one paired review of the merged diff: Codex + Claude.
- If both return SHIP, the candidate is ready.
- If either returns REVISE, halt with `post-merge-review-revise`.

Conflict path:

1. Stop at the first conflicting merge.
2. Preserve conflict markers in the integration worktree.
3. Spawn one merger agent in that integration worktree.
4. The merger agent may edit only conflicted files plus explicitly allowed generated lockfiles.
5. The merger agent does not land anything by itself.
6. Run double review:
   - Claude orchestrator review
   - Codex review
7. If both return SHIP, continue the merge sequence.
8. If either returns REVISE, halt with `merge-conflict-double-ship-failed`.

Merger prompt includes:

- slice ID and original plan
- `base_sha`
- `implementer_run_id`
- merge order
- current merge command and failing branch
- conflict files with markers
- each implementer's diffstat
- full diffs for conflicted files
- relevant neighboring diffs when imports/types are involved
- last known test output from each implementer
- relevant mailbox `merge_note` messages
- instruction: resolve conflicts only; no unrelated refactor

Do not hydrate full transcripts by default. Include transcript handles and hydrate excerpts only when a conflict explicitly depends on a prior decision.

## Auth Token Handling

Token source order:

1. OS credential store or plugin-managed secret reference when available.
2. Explicit developer env:
   - `OLLAMA_CLOUD_API_KEY`
   - `ANTHROPIC_AUTH_TOKEN`
   - `ANTHROPIC_API_KEY` only for real Anthropic routes
3. No plaintext project config fallback in v0.10.0.

Subprocess env is constructed per member and never serialized into sidecar events.

Sidecar may store safe metadata only:

```json
{
  "provider": "ollama-cloud",
  "base_url_hash": "sha256:<64-hex>",
  "model": "kimi-k2.6:cloud",
  "secret_presence": "present"
}
```

Sidecar, mailbox, diagnostics, breadcrumbs, transcript handles, and replay artifacts must not store:

- raw tokens
- bearer headers
- full child env
- prompts containing copied secrets
- provider credential payloads

`lib/codex-bridge/implementer/secret-redaction.js` owns redaction. It must redact by field ownership and env-key denylist, not only token regex.

Protected env keys:

```js
[
  "OLLAMA_CLOUD_API_KEY",
  "ANTHROPIC_AUTH_TOKEN",
  "ANTHROPIC_API_KEY",
  "ANTHROPIC_BASE_URL",
  "ANTHROPIC_MODEL",
  "OPENAI_API_KEY"
]
```

## Sidecar Audit and Replay

Concurrent implementers produce nondeterministic wall-clock ordering. Replay must not depend on wall-clock determinism.

Use a new `implementer_run_id`, not `team_run_id`. The plugin does not currently have a team-run schema. `implementer_run_id` is generated once at autopilot Phase B start for a single implementer-experts slice execution.

## Halt Classification

Required implementer halt:

- Cancel all active implementers.
- Preserve completed branches.
- Preserve active worktrees for forensics.
- Emit terminal slice halt.

Optional implementer halt:

- Mark the member failed.
- Continue only if no required file partition depends on it.
- Exclude that branch from merge.
- Record omission in post-merge review prompt.

New halt codes added to `lib/codex-bridge/halt-envelope.js`:

```text
implementer-cap-exceeded
implementer-high-cost-rationale-missing
implementer-member-id-invalid
implementer-claimed-files-missing
implementer-claimed-file-violation
implementer-required-child-failed
codex-cli-blocked
claude-cli-protocol-unsupported
claude-cli-auth-missing
claude-cli-auth-rejected
ollama-cloud-route-invalid
mailbox-delivery-failed
worktree-create-failed
worktree-dirty-before-dispatch
merge-conflict-double-ship-failed
post-merge-review-revise
sidecar-replay-concurrent-order-invalid
```

All are terminal in v0.10.0. Unknown halt behavior remains fail-closed through `isTerminalHalt`.

## Migration and Back Compatibility

Unchanged:

- reviewer-only panels
- single-implementer slices without `Implementers`
- existing `phases.implement` records
- existing `appendImplementDispatch`
- v0.9.1 process hardening
- terminal halt envelope known-set behavior
- proper-lockfile sidecar append behavior

- a valid `Implementers` block exists

### Hook Firing Under N-Way Parallelism

The `honest-reporting` hook is an orchestrator-session hook. It fires from Claude Code around the orchestrating session's tool use and stop lifecycle.

Implementer subprocesses do not fire Claude Code hooks:

- `codex exec` runs as its own process with Codex's sandbox/approval model.
- `claude` CLI runs as its own process with Claude CLI's permission/tool configuration.
- v0.10.0 does not make either subprocess share Claude Code's orchestrator hook lifecycle.

Consequences:

- The orchestrator's Stop hook fires once at orchestrator-stop time, not once per implementer.
- There is no shared-state race between orchestrator hooks and N implementer subprocesses.
- v0.10.0 does not need per-implementer locking for the honest-reporting marker file.
- The marker only protects orchestrator claims.
- Any implementer-specific hook/permission behavior belongs to that CLI's own configuration, not to `codex-paired-superpowers`.

The installed-plugin grep found zero external references to `honest-reporting`, so no other plugin currently depends on this hook state.

### Ralph-Loop Coupling

A grep across `claude-plugins-official/ralph-loop` found zero literal references to `isTerminalHalt` or the explicit halt-envelope helper. This is a real coupling boundary: `codex-paired-superpowers` returns halt envelopes, but `ralph-loop` does not directly import or inspect `lib/codex-bridge/halt-envelope.js`.

v0.10.0 policy:

- Keep returning structured halt envelopes from `wrapAsHaltEnvelope`.
- Treat exit code plus textual halt-envelope payload in skill output as the actual cross-plugin coupling.
- Document that `ralph-loop` consumes the envelope through conventional skill output, not direct helper invocation.
- Do not assume `ralph-loop` enforces `isTerminalHalt`.
- If stronger coupling is needed later, propose a formal envelope contract upstream to `ralph-loop`.

This does not block v0.10.0 because the current contract is one-sided and downstream-compatible.

### Commit and PR Attribution

`commit-commands` has no special merge-commit or co-author behavior. It commits with normal `git commit -m ...` semantics.

`pr-review-toolkit` has no explicit per-author attribution logic. `gh pr create` produces a normal PR from the branch HEAD.

v0.10.0 implication:

- The integration branch contains the implementer branch ancestry in git history.
- The final PR usually shows one operator-authored branch/merge result.
- Per-implementer contribution is discoverable through git history, branch names, sidecar `phases.implementer_experts`, and merge metadata.
- Per-implementer blame is not surfaced in the PR UI by default.

This is acceptable for v0.10.0. Rich per-implementer PR attribution and blame UI are deferred to v0.10.1+.

## Decisions on Open Questions

1. `claude-cli` is a distinct runtime kind with a shared subprocess contract, not a forked Codex adapter and not `claude-task`.

2. Merge via deterministic `git merge --no-ff --no-commit` into an integration worktree. Do not use rebase or plain `git apply`.

3. Mailbox delivery is nonblocking pull at checkpoints plus timed cadence. This is honest about CLI limits and still delivers while siblings are alive.

4. Tokens come from keychain/plugin-managed references or explicit developer env, then pass only through sanitized per-child env.

5. Poll cadence is 45 seconds with jitter, plus immediate checkpoint polls. Empty inbox polls never block.

6. Required implementer failure halts the slice and cancels siblings. Optional implementer failure can continue only when file/dependency claims allow it.

7. Replay determinism comes from locked `event_seq` plus causal IDs, not deterministic scheduling.

8. Merger prompt includes plan, base SHA, implementer run ID, merge order, conflict files, relevant diffs, test deltas, and mailbox merge notes. It does not hydrate full transcripts unless directly needed.

## Acceptance Criteria

v0.10.0 is done when:

- 3x same-model implementers dispatch with distinct member IDs.
- 5x Codex dispatch only with `high_cost: true` and rationale.
- Mixed Codex + Kimi + GLM dispatch through one surface.
- Reviewer-only flows still pass unchanged.
- Each implementer edits only its own worktree.
- Implementers can exchange mailbox DMs while sibling processes are still active.
- Required implementer failure halts the slice and cancels siblings.
- Clean branches merge deterministically.
- Conflict branches invoke merger-agent path and require double SHIP.
- One paired post-merge review gates the merged diff.
- Sidecar replay reconstructs interleaved implementer events.
- Secret canaries do not appear in sidecar, mailbox, diagnostics, logs, transcripts, adapter metadata, or replay artifacts.
- The plugin ecosystem integration notes are reflected in docs so users know this surface belongs to `codex-paired-superpowers`, not `feature-dev` or adjacent domain plugins.
