# codex-paired-superpowers v0.9.0+ — Destination

**Status:** Double-SHIP'd 2026-05-11 by Claude (Opus 4.7) and Codex (GPT-5.5, thread `019e1536-…`) across 2 architecture-consult rounds (REVISE → SHIP).

**Why this doc exists.** Across v0.6 → v0.8.1.1 the plugin drifted from the user's stated goal — building a "Codex-paired" tool when the user has been describing a **model-routed dev team** since v0.6. The v0.7.0 implementer-routing was the design signal; v0.8.0's Claude-Task-subagent experts missed it. This doc fixes the drift and pins the destination so subsequent releases land against an explicit target.

## 0. Practical end goal (plain language)

**What you experience when this works:** You sit in Claude Code and invoke one of the plugin's skills (`/brainstorming`, `/writing-plans`, `/test-driven-development`, `/subagent-driven-development`, `/autopilot`, `/systematic-debugging`). The plugin orchestrates a team of advisors — each running on the LLM best-suited to its domain via a separate CLI subprocess. The L11 discipline (anti-yes-man critique, structured Machine Result verdicts, double-SHIP rounds, mailbox-mediated peer coordination, audit trails) is enforced across all advisors regardless of which model serves each role.

**Concretely on a single spec-review round:**

1. You describe what you want to build. Claude Code orchestrates.
2. Claude (orchestrator) drafts the spec.
3. The plugin's `composeExperts` picks 2–4 advisors from the spec's signals.
4. For each picked advisor, the plugin's dispatcher walks the role's preference ladder, finds the first installed CLI, and spawns it as a subprocess with the role's system prompt.
5. Each advisor returns a `## Machine Result` block with SHIP/REVISE verdict + blocking/nonblocking findings + peer DMs.
6. The orchestrator aggregates, surfaces blockers to you, routes peer DMs to recipient advisor mailboxes for the next round.
7. Sidecar records every dispatch: which CLI ran, which role prompt version, what was inputted, what came back, why the ladder resolved to that CLI.
8. Repeat until double-SHIP across all advisors (or terminate on a blocker that needs your judgment).

**What the user gets:** cross-model diversity (codex's anti-yes-man rigor + claude's UI judgment + gemini's long-context + kimi/glm via Ollama Cloud), structured verdicts, full audit, graceful degradation when CLIs aren't installed. The plugin's value is the **discipline layer** that makes this fleet act as one coherent dev team rather than a stochastic chorus.

**What the user does NOT get:** a model-as-a-service. No model is hosted by the plugin. Each CLI handles its own auth, retries, rate limits, sandboxing. The plugin only orchestrates.

## 1. Vision

**codex-paired-superpowers is the L11 discipline layer that runs across CLI-dispatched model teammates.** It is not a dispatcher, not a provider adapter library, not a re-implementation of Claude Code features. It owns:

- The L11 rubric and verdict protocol (Machine Result blocks, double-SHIP rounds, anti-yes-man rules)
- Per-(feature, role) persistent state via the sidecar
- Mailbox-based peer messaging between roles
- Audit trails for every reviewer turn
- The skill prose (brainstorming, writing-plans, TDD, SDD, autopilot, debugging) that drives the discipline

**Dispatch — the act of actually invoking a model with a system prompt and getting text back — is delegated to external CLIs** (`codex`, `claude`, `gemini`, `qwen`, future vendors). The plugin spawns those CLIs as subprocesses, the way Zen-MCP's `clink` does. Each CLI handles its own auth, sandboxing, retries, context window, and vendor-specific quirks. The plugin layers L11 discipline on top.

**The plugin's value:** any role × any CLI × persistent state × structured verdicts × audit. Roles are routed per-project so each domain expert runs on the model best at that domain.

## 2. Roles

Nine logical roles. Each is routed at dispatch time via a **preference ladder + graceful degradation** model — the plugin does **not** hardcode a single CLI per role.

### Preference ladders (recommendations, not defaults)

The plugin ships `lib/codex-bridge/role-recommendations.json` (validated by tests, exposed via `bin/codex-paired list-recommendations`). Each role declares an ordered preference of CLIs the plugin's authors believe best-suited to that domain, **plus a rationale**:

| Role | Preference ladder | Rationale | Permissions |
| --- | --- | --- | --- |
| `paired-reviewer` | `codex` → `claude` | L11 anti-yes-man rigor benefits from cross-model; codex strongest | read-only |
| `expert-architecture` | `codex` → `claude` | Deep architectural reasoning; codex strongest | read-only |
| `expert-ai-harness` | `codex` → `claude` | AI pipeline correctness; codex strongest | read-only |
| `expert-test` | `codex` → `claude` | Test boundary design; codex strongest | read-only |
| `expert-security` | `codex` → `gemini` → `claude` | Threat-model thinking; codex/gemini strongest | read-only |
| `expert-backend` | `codex` → `qwen` → `claude` | Storage/transactions; codex or qwen-coder when installed | read-only |
| `expert-ui` | `claude` → `gemini` → `ollama{kimi-k2.6}` → `ollama{glm-5.1}` → `codex` | Render hazards and prop surfaces; claude well-suited; deep fallback ladder for cross-model diversity when desired | read-only |
| `expert-ux` | `claude` → `gemini` → `ollama{kimi-k2.6}` → `ollama{glm-5.1}` → `codex` | Workflow alignment and copy judgment; same deep ladder as `expert-ui` | read-only |
| `implementer` | `claude` → `codex` | Code writing; either works; claude default fits Claude Code session | **write-allowed** |

### Adapter variants (`ollama` for cloud-hosted models)

Ollama Cloud hosts large models (Kimi k2.6, GLM 5.1, Qwen, DeepSeek) too big to run locally. The plugin treats Ollama as its own **stateless one-shot adapter** with model variants.

**The exact one-shot CLI form is verified at implementation time.** Per [docs.ollama.com/cli](https://docs.ollama.com/cli) and [docs.ollama.com/api](https://docs.ollama.com/api), Ollama's `ollama launch` command is for **launching/configuring integrations** (e.g., wiring a Claude Code session to use Ollama as backend), not for stateless inference. Stateless inference is typically `ollama run <model> "<prompt>"`. The Ollama adapter contract is "a stateless model turn against an Ollama-hosted model"; whether that's `ollama run`, the HTTP API at `/api/generate`, or a different form is determined when the adapter is built.

Placeholder config shape (final form verified during v0.9.0 implementation):

```json
// cli-clients/ollama.json (PLACEHOLDER — verified during v0.9.0 implementation)
{
  "name": "ollama",
  "command": "ollama",
  "invocation": "TBD — stateless one-shot, likely `ollama run <model> -p <prompt>` or HTTP /api/generate",
  "variants": {
    "kimi-k2.6":  { "model_name": "kimi-k2.6:cloud" },
    "glm-5.1":    { "model_name": "glm-5.1:cloud" },
    "qwen3.5":    { "model_name": "qwen3.5:cloud" }
  }
}
```

The companion `cli-clients/claude.json` stays simple — just the default Anthropic-backed Claude with no variants. Clean separation between "Anthropic Claude" (`claude`) and "cloud-hosted model via Ollama" (`ollama{<variant>}`).

Preference ladder entries are `(cli, variant?)` tuples; `ollama{kimi-k2.6}` resolves to `cli: "ollama", variant: "kimi-k2.6"`. Availability detection runs:

- `ollama --version` (binary present)
- `ollama list` or `ollama whoami` (cloud auth configured)

A variant counts as available iff both pass. Adding new cloud models (Qwen, DeepSeek, future Ollama-hosted) is a config-only change: add a variant entry to `ollama.json`; no adapter code needed.

**Ollama is explicitly stateless.** No session continuity claims, ever. Each turn is a fresh invocation with full `sidecarParticipantState` re-injected via the prompt. (Session continuity for any adapter is deferred to v0.9.1 entirely.)

**Note on local Ollama:** the `ANTHROPIC_BASE_URL=http://localhost:11434` env-var path is for **local-only** models (small enough to run on the user's machine via `ollama serve`). v0.9.0 focuses on the cloud path; local-Ollama can be added later as a separate `cli-clients/ollama-local.json` if needed.

### Resolution at dispatch time

The dispatcher resolves each role to a concrete CLI at invocation, using availability data from doctor's cached CLI inventory:

```
resolveAdapter(role, availableCLIs):
  if role has explicit override in .codex-paired/role-routing.json:
    cli = override.cli
    if cli not in availableCLIs:
      HARD HALT with "role X explicitly routed to <cli>, but <cli> is not installed. Install it or remove the override."
    return cli
  
  for cli in role_recommendations[role].preference:
    if cli in availableCLIs:
      return cli
  
  HARD HALT with "no-supported-cli-for-role: role X has no available CLI in its preference ladder; no fallback possible."
```

**Claude is not assumed always-available** — it is detected like any other CLI. If running outside a Claude Code session (no `claude` binary, no Claude subagent transport), dispatch halts with `no-supported-cli-for-role` rather than silently breaking.

**Explicit override + unavailable CLI = hard halt.** A user opting into a specific CLI made an explicit routing decision; substituting silently would violate intent and make model behavior non-obvious. Walk the ladder only when no override is set.

### Implementer is the only default write-capable role

Reviewer/expert roles inherit `permissions: "read-only"`, which the dispatcher enforces by refusing to pass dangerous CLI flags (`--dangerously-bypass-approvals-and-sandbox` on codex, `--yolo` on gemini, etc.). Users can override per-role in routing config, but the dispatcher emits a loud audit warning for any `reviewer-role + write-allowed` combination.

### Role selection per skill

Selection uses the existing `composeExperts` logic from v0.8.0. The dev-team is **role-routed when invoked, not always-fan-out.** Skills get composer-selected experts (typically 2–4); fan-out rationale is required when `selected.length >= 6`.

## 4. Adapter Contract

The dispatcher exposes a single function:

```ts
dispatch(role: string, systemPrompt: string, userPrompt: string, options?: DispatchOptions)
  -> DispatchResult
```

`DispatchResult` shape:

```ts
interface DispatchResult {
  responseText: string;     // verbatim model reply, free-form
  exit: number;             // 0 = success, nonzero = failure
  warnings: string[];       // adapter-normalized warnings
  sessionId?: string;       // adapter's native session/thread id; OPTIONAL cache only
  adapterMeta: object;      // raw adapter-specific data for debugging
  duration_ms: number;
}
```

Each CLI has its own adapter module:

- `lib/codex-bridge/cli-adapters/codex.js` — parses `--json` event stream
- `lib/codex-bridge/cli-adapters/claude.js` — parses text + session line
- `lib/codex-bridge/cli-adapters/gemini.js` — parses text/events (v0.9.1)
- `lib/codex-bridge/cli-adapters/qwen.js`, `ollama.js`, etc. — v0.9.2+

Adapters export `{ dispatch(systemPrompt, userPrompt, options) -> DispatchResult }`. The dispatcher front-end reads role-routing, picks the adapter, applies permission flags, spawns, normalizes output.

**Permission flag mapping** is declared per-adapter in `cli-clients/<name>.json`:

```json
{
  "name": "codex",
  "command": "codex",
  "permissions": {
    "read-only":     { "args": ["--sandbox", "read-only"] },
    "write-allowed": { "args": ["--dangerously-bypass-approvals-and-sandbox"] }
  },
  "roles": {
    "codereviewer": { "prompt_path": "..." }
  }
}
```

The dispatcher refuses to mix `read-only` role config with write-mode flags. Unknown adapter names and unknown permission modes are rejected at config-load time, not at dispatch time.

**Config merge order:** bundled defaults < project `.codex-paired/cli-clients/` < project `role-routing.json`. Users can override any layer; the dispatcher resolves left-to-right.

### Runtime kinds + mailbox injection

v0.9.0 has **two runtime kinds** for dispatching role turns:

| `runtime_kind` | When used | Adapter examples | Mailbox injection |
| --- | --- | --- | --- |
| `claude-task` | Roles routed to `claude` (Anthropic-backed only) | Claude Code's `Agent` tool with `general-purpose` subagent | Spawn-prompt embedding (turn-atomic) |
| `cli-harness` | Roles routed to `codex`, `gemini` (v0.9.1), `ollama{<variant>}`, future CLIs | New `lib/codex-bridge/cli-harness/` Node module | Spawn-prompt embedding (turn-atomic) |

**Both runtime kinds embed unread mailbox messages in the spawn prompt before dispatch.** No runtime kind gets mid-turn DM injection for experts — they're all turn-atomic. Concretely:

1. Orchestrator reads each role's unread mailbox via `readUnreadMessages(repoRoot, identity.id)` BEFORE spawn.
2. `assembleSpawnPrompt` (from v0.8.0, unchanged) embeds the messages verbatim under `## Unread Mailbox Messages` in the prompt text.
3. Whether dispatched via Task or CLI harness, the subagent receives the messages as part of its input prompt.
4. After parse-success, `markManyAsRead` flips the injected messages' `read_at` field.

### The single hook exception: slice implementer in worktree

The v0.7.3.1 `hooks/mailbox-inject.sh` PostToolUse hook fires for actors running inside `.git-worktrees/slice-N/`. The hook's `inferActorAndRepoRoot()` only matches that path pattern (verified by reading `lib/codex-bridge/hook-mailbox-inject.js`). This means:

| Role | Runs in worktree? | Mid-turn DM via hook? |
| --- | --- | --- |
| `implementer` (when routed to `claude` via Task) | Yes — dispatched into `.git-worktrees/slice-N/` | ✓ Yes |
| `implementer` (when routed to `codex-background-bash`) | Yes — runs in slice worktree | ✓ Yes (existing path) |
| Any expert (`expert-ui`, `expert-ux`, etc.) | No — runs from arbitrary cwd | ✗ No — turn-atomic only |
| `paired-reviewer` | No — runs from arbitrary cwd | ✗ No — turn-atomic only |

**The hook is a feature of the slice-implementer path, not a generic expert mechanism.** Experts are turn-atomic across both `claude-task` and `cli-harness` runtime kinds.

The peer-DM `drainPeerDMs` scheduler from v0.8.0 still creates **within-round multi-turn dialog** — expert-A returns with a DM to expert-B; scheduler immediately re-dispatches expert-B (with DM in its mailbox); expert-B returns with counter-DMs; scheduler immediately re-dispatches; until convergence or cap. This is turn-taking-via-re-dispatch, not mid-turn injection — and it works identically for both runtime kinds.

### Why this asymmetry (slice implementer gets mid-turn, experts don't)

The slice implementer runs **inside its own git worktree** for a meaningful duration (minutes to hours): it writes code, tests, commits. During that span, the orchestrator may have new messages for it (e.g., `slice-blocker-from-mailbox` events). Mid-turn injection via PostToolUse hook makes that delivery responsive.

Experts run **per-dispatch** (~10–60s per turn). Their context is bounded by the turn itself. Between-turn message delivery is sufficient for the L11 review discipline. Mid-turn injection would require either:
- A new identity mechanism for experts running outside worktrees (e.g., `CPS_EXPERT_ID=expert-ui` env var set by the orchestrator), AND
- Long-running CLI sessions to receive injected messages

Both are deferred. If real users want mid-turn expert DM, that's a v1.0.0+ design conversation.

### Why LiteLLM is not adopted

LiteLLM is a Python SDK/proxy that provides an OpenAI-compatible interface to 100+ model providers (OpenAI, Anthropic, Cohere, Ollama, etc.) with router fallbacks, spend tracking, and gateway features. It's an **excellent fit for teams building REST-based LLM gateways** — but it's the wrong abstraction for this plugin:

- We explicitly delegate model dispatch to vendor CLIs, not REST APIs. Each CLI handles its own auth, retries, sandboxing — that's the point. LiteLLM only helps if we bypass the CLIs.
- LiteLLM's auto-vendor-fallback feature would **violate** our routing-audit invariant (round 3): explicit override of an unavailable CLI is a **hard halt**, not a silent substitution. Silent fallback is the kind of surprise that breaks trust.
- LiteLLM is canonical Python; our plugin is Node ESM. Bridging requires either a long-running LiteLLM proxy daemon (new failure surface) or a partially-maintained JS port.
- The user's question ("can we use LiteLLM or something like it?") is really about **subprocess lifecycle management**, not API abstraction. LiteLLM doesn't solve that.

**Source for the rejection rationale:** [docs.litellm.ai](https://docs.litellm.ai/) frames it as a gateway/SDK for API callers; nothing in it addresses CLI subprocess management.

### Panel mode (high-stakes multi-model consensus)

For specific high-stakes review phases, the dispatcher supports a **panel mode** that dispatches a single role to **multiple available CLIs in parallel** for cross-model consensus. Designed for foundational decisions where single-model bias is unacceptable (test design especially).

**Dispatcher contract (panel mode):**

```js
dispatch({
  role:            'expert-test',
  mode:            'panel',         // ← new field; default 'single'
  panel_min_size:  2,               // hard floor; <2 is not a panel
  panel_max_size:  3,               // default; bounds cost (N-times single)
  prompt:          testListReviewPrompt,
})
```

**Resolution (deterministic, no silent degradation):**

1. Walk the role's preference ladder; collect ALL available CLIs.
2. If `available.length < panel_min_size` → **HARD HALT** with `panel-quorum-unavailable`. Sidecar warning lists which CLIs were missing. **No silent degradation to single mode.**
3. Take first `min(available.length, panel_max_size)`. Audit `skipped_candidates` for any preference-ladder entries beyond `panel_max_size`.
4. **Snapshot the member list once.** Used unchanged across the initial panel AND any consensus round. Do not re-resolve adapters between rounds unless a member hard-failed.
5. Fan out N parallel dispatches with same `assembleSpawnPrompt` output, same role.

**Quorum:** `max(2, floor(N/2) + 1)`. For N=2, both must succeed. For N=3, 2/3. For N=4, 3/4. For N=5, 3/5.

**Verdict aggregation (deterministic; no semantic overlap detection):**

| Panelist verdicts | Panel outcome |
| --- | --- |
| All parsed SHIP | `panel-SHIP` — proceed |
| All parsed REVISE | `panel-REVISE` — orchestrator surfaces findings |
| Mixed SHIP/REVISE (panel_size >= 3) | One automatic consensus round (panelists see each others' findings; re-dispatch with same member list) |
| Mixed SHIP/REVISE (panel_size = 2) | `panel-disagreement` — user arbitrates |
| Parse failure (post-repair) | Degraded-N proceeds if quorum still met; else `panel-quorum-lost` halt |
| After 1 consensus round, still mixed | `panel-disagreement` — user arbitrates (no open-ended consensus loops) |

**Findings preservation:** sidecar records each panelist's `blocking_findings` + `nonblocking_findings` arrays verbatim. No semantic dedup, no overlap detection (those are not machine-reliable in v0.9.0). Tests assert preservation; orchestrator synthesizes when surfacing to user.

**Panel peer DMs are suppressed.** Three `expert-test` panelists DM'ing `expert-architecture` independently would cause duplicate-delivery noise and conflicting messages. So:

- In panel mode, `peer_messages_requested` from any panelist is **not enqueued**.
- Sidecar records the suppression: `panel_peer_messages_suppressed: [{to: '<target>', body_hash: '<sha>'}]` per panelist, preserving the audit trail of what would have been sent.
- Cross-panel exposure happens **only via the consensus round** — panelists see each others' findings in the re-dispatch, not via mid-panel mailbox.

**Default panel-mode roles (narrow, signal-driven, user-controlled):**

| Skill / phase | Role | Panel default | Rationale |
| --- | --- | --- | --- |
| `writing-plans` TDD-mandatory phase | `expert-test` | **Always panel** | Test design is foundational; cross-model consensus mandatory |
| `autopilot` release gate | `paired-reviewer` | Panel | Final go/no-go decision |
| `writing-plans` slice with `high_stakes: true` frontmatter flag | `expert-security` + `expert-architecture` | Panel per role | User opts slice into panel review via spec frontmatter |
| All other dispatch | All roles | Single | Default; cost-efficient |

`high_stakes: true` is a **user-controlled per-slice frontmatter field** in the plan — no silent escalation via keyword detection. Future versions may add composer-suggested warnings ("this slice looks high-stakes; consider `high_stakes: true`") but execution stays explicit.

**v0.9.0 panel adapters:** `codex` + `claude-task` + `ollama{<variant>}`. **Grok deferred** — no first-party Grok CLI per xAI docs (API/SDK only). Grok admission requires the same gate as Gemini: fake-CLI fixtures + installed-smoke + verified invocation contract. Likely v0.9.x post-Gemini.

**Cost impact:** panel mode is N× cost vs single (3× for default panel_max_size=3). For TDD-mandatory test review, this is acceptable for foundational decisions. Users can override per-slice if cost matters.

### The `cli-harness` module

v0.9.0 ships its own minimal subprocess-management harness for non-Claude CLIs, vendored in `lib/codex-bridge/cli-harness/`:

```
lib/codex-bridge/cli-harness/
├── harness.js              ~80 LOC  (spawn + AbortController + bounded buffers + timeout)
├── adapters/codex.js       ~50 LOC  (parse --json event stream → DispatchResult)
├── adapters/ollama.js      ~40 LOC  (parse text + Ollama Cloud stderr semantics)
├── adapters/gemini.js      (v0.9.1) ~50 LOC  (parse text/events)
├── transcript.js           ~30 LOC  (sidecar replay-grade transcript per turn)
└── normalizer.js           ~30 LOC  (DispatchResult shape enforcement)
```

**Total: ~230 LOC for v0.9.0** (codex + ollama adapters). Adding Gemini in v0.9.1 is one adapter file. No external Node dependencies beyond what's already in the plugin (`child_process.spawn`, `AbortController`).

**v0.9.0 dispatch is stateless one-shot.** Session continuity for any adapter is deferred to v0.9.1. Each turn pays the spawn cost; context comes from `sidecarParticipantState` re-injected into the prompt (v0.8.0 mechanism).

### Availability detection + cache

`doctor` (existing skill) extended to probe each CLI declared in `cli-clients/`:

```
- codex:  PASS  (codex auth ✓, v0.42.0, /opt/homebrew/bin/codex)
- claude: PASS  (running this session, v0.21.0, /usr/local/bin/claude)
- gemini: NOT INSTALLED — `expert-ux` will fall back to `claude` (preference[1])
- qwen:   NOT INSTALLED — `expert-backend` will fall back to `codex` (preference[0])
- ollama: NOT INSTALLED
```

Result is cached at `.codex-paired/cli-availability.json`. **Cache key includes** `plugin_version` + per-CLI `resolved_path` so the cache invalidates on plugin upgrade OR when a CLI's install location moves. **TTL 24h** (regenerated by doctor or first dispatch).

Per-CLI cached payload:

```json
{
  "name":           "codex",
  "status":         "available",        // "available" | "missing" | "broken"
  "version":        "0.42.0",
  "resolved_path":  "/opt/homebrew/bin/codex",
  "checked_at":     "2026-05-11T22:00:00.000Z",
  "plugin_version": "0.9.0"
}
```

Invalidation triggers: TTL expired, plugin version mismatch, or `resolved_path` missing from disk.

## 5. Safety / Failure Semantics

**Stateless replay is canonical truth.** Each turn's sidecar record contains everything needed to reconstruct it:

```json
{
  "runtime_kind":          "claude-task",
  "adapter":               "claude-task",
  "variant":               null,
  "requested_role":        "expert-ux",
  "resolved_cli":          "claude",
  "resolution_source":     "recommendation",
  "preference_index":      0,
  "preference_ladder":     ["claude", "gemini", "ollama{kimi-k2.6}", "ollama{glm-5.1}", "codex"],
  "unavailable_candidates": [],
  "fallback_reason":       null,
  "role_prompt_version":   "v0.8.0-r2",
  "role_prompt_hash":      "sha256:...",
  "spec_path":             "/abs/path",
  "spec_snippet_hash":     "sha256:...",
  "mailbox_message_ids":   ["msg-..."],
  "inputs_hash":           "sha256:...",
  "response_text_inline":  "<verbatim, if <50KB>",
  "response_ref":          "responses/sha256-<hash>.txt (if overflow)",
  "response_hash":         "sha256:...",
  "cli_session_id":        null,
  "duration_ms":           45123,
  "cli_warnings":          []
}
```

**`runtime_kind`** distinguishes Task-dispatched (`claude-task`) from CLI-harness-dispatched (`cli-harness`) turns. Identical sidecar shape across both. The `adapter` field tells you which underlying CLI ran the turn (`claude-task` for Task; `codex`, `ollama`, `gemini` for CLI harness). `variant` is populated for Ollama (`kimi-k2.6` etc.) and null otherwise. `cli_session_id` stays `null` in v0.9.0; session continuity is a v0.9.1 concern.

**Resolution audit fields** (`requested_role` through `fallback_reason`) capture WHY this CLI was chosen for this turn. They answer "why did Claude do this instead of Gemini?" in the audit trail. `resolution_source` is one of `recommendation` (walked the ladder), `override` (explicit `.codex-paired/role-routing.json` entry), or `implicit-fallback` (no ladder match; landed on Claude default — rare and worth warning on).

Native CLI session IDs are an optimization cache — never load-bearing for audit. If a vendor's continuation breaks, replay from recorded inputs. Cross-CLI migration is possible: a codex-recorded turn can be replayed via claude (with model-output drift visible against `response_hash`).

**Response storage:** verbatim text inline up to 50KB (`sidecar.max_inline_response_bytes`, configurable). Overflow goes to content-addressed file `responses/sha256-<hash>.txt` in `.superpowers-codex-paired/`. Content addressing dedupes identical responses across turns.

**Concurrency cap:** default `max-concurrent-cli: 4`. Parallel expert dispatches queue past 4. Matches the peer-DM-cap pattern from v0.8.1.1.

**Fail-closed defaults.** Reviewer roles failing to dispatch (CLI crash, adapter error, malformed output) halt the round with `cli-dispatch-failed` (new halt reason, `HALT_REASONS_PRESERVE`). They do NOT silently succeed. The orchestrator surfaces failed targets via sidecar `cli_warnings` for triage.

**Read-only enforcement is ON by default in v0.9.0.** No warn-only release. This is a safety boundary, not a convenience migration. Reviewer-with-write-permissions requires explicit per-role override and emits a sidecar audit warning every turn.

**Migration from v0.8.x:** silent on-load upgrade. `codex_session: <id>` becomes `role_sessions.paired-reviewer: <id>`. The CLI that loaded the sidecar exposes a single migration warning; library reads do not emit per-call stderr noise. Three-release back-compat (alias removed in v0.12.0).

**Role-prompt versioning:** both. Each `role-prompts/<name>.txt` has frontmatter `version: <human-readable>` + `role-prompts.lock.json` records SHA-256 hashes at plugin install. The dispatcher audits both `role_prompt_version` and `role_prompt_hash`. Mismatch (e.g., user edited a prompt without bumping version) emits a sidecar audit warning.

## 6. Out-of-Scope (Forever)

The plugin will **not** build:

- **In-process provider adapters calling REST APIs.** We shell to CLIs. `codex` / `claude` / `gemini` / future vendors handle their own auth, retries, rate limits, context windows. We don't reimplement what `@anthropic/sdk`, `@google/genai`, OpenAI SDK, etc. already do.
- **MCP server hosting.** This plugin is a Claude Code plugin, not an MCP server in its own right. (The bundled `codex` MCP server is a third-party we depend on, not something we maintain.)
- **Native Claude Code re-implementations.** Task/Agent subagent dispatch, file I/O, hook registration, slash-command dispatch — Claude Code does these. We use them; we don't replace them.
- **Vendor SDK integration in-process.** No npm install of `@anthropic-ai/sdk` or `@google/genai`. Adapters always spawn the vendor's CLI.
- **Custom auth flows.** Each CLI owns its own login (`codex auth login`, `claude` Anthropic auth, `gemini` Google auth, etc.). Doctor verifies they're configured; doesn't proxy.

## 7. Test Strategy

How v0.9.0 ensures the dispatch contract works correctly. **Seven tiers, each with explicit pass criteria.** Following the existing codex-paired discipline: every fix gets a regression test at the failure boundary; every test asserts observable behavior, not mock-invocation counts.

### Tier 1 — Unit tests (pure logic)

Per-function, no I/O. Fast (~30s for the full suite).

| Module | What it asserts |
| --- | --- |
| `role-recommendations.json` validation | Schema correctness, no duplicate roles, every preference entry references a known CLI or variant, every entry has a rationale |
| `role-routing.json` parsing + merge | Bundled defaults < project cli-clients < project routing; unknown CLI/variant rejected at load |
| `resolveAdapter(role, availableCLIs)` | Override path (CLI present → use; CLI missing → hard halt); ladder walk (picks first available); no-fallback path (halt with `no-supported-cli-for-role`); explicit override of unavailable CLI = hard halt |
| Permission flag mapping | `permissions: "read-only"` refuses to emit dangerous flags per CLI; `permissions: "write-allowed"` emits write flags; mismatch (reviewer + write) emits sidecar warning |
| `DispatchResult` normalization | Fixed shape across adapters; missing fields default sanely; verbatim `response_text` preserved |
| Resolution audit fields in sidecar | All required fields present per turn (`requested_role`, `resolved_cli`, `resolution_source`, `preference_index`, `preference_ladder`, `unavailable_candidates`, `fallback_reason`) |
| Stateless replay reconstruction | Given a recorded turn, can rebuild assembled prompt; hashes match |

### Tier 2 — Adapter tests with fake CLI fixtures

Per-adapter, fake binary in `tests/fixtures/fake-cli/{codex,claude,gemini}.sh`. Tests run with `PATH` modified to put fake CLIs first.

**Each adapter MUST pass tests for happy path AND three failure paths** (Codex round-2 critique was explicit on this):

| Adapter | Happy path | Nonzero exit | Stderr warning | Malformed/partial stdout |
| --- | --- | --- | --- | --- |
| `codex` | parses `--json` event stream, returns DispatchResult | normalizes to `{exit: <nonzero>, responseText: ''}` | warning passes through `warnings[]` | parse error → `exit: 1, warnings: ['malformed-output']` |
| `claude` | parses text + session line | same | same | same |
| `claude{kimi-k2.6}` | env vars set + model flag passed + Ollama backend mock responds | mock backend down → `exit: 1, warnings: ['ollama-unreachable']` | same | same |
| `gemini` (v0.9.1) | parses text/events | same | same | same |

**Adapter readiness gate** (Codex round-2): a new adapter is NOT "ready" until fake-CLI fixtures cover ALL FOUR paths AND an installed-smoke test passes.

### Tier 3 — Integration tests (dispatcher end-to-end)

Whole-pipeline: dispatch → spawn fake CLI → parse → return → audit. ~5–10s per test.

- Single dispatch with fake codex → DispatchResult populated, sidecar turn recorded with full resolution audit
- Parallel dispatch (4 concurrent fake CLIs) → all complete, audit records distinct
- Concurrency cap (config: `max-concurrent-cli: 2`) → 5 dispatches queue properly, none lost
- Permission enforcement: read-only role's dispatch refuses to pass `--dangerously-bypass-...` even if cli-clients config declares it
- Resolution audit captures `resolution_source: "recommendation"` when ladder walked, `"override"` when explicit, `"implicit-fallback"` when default landed
- Doctor cache hit: dispatch reads cached availability instead of re-probing every spawn
- Doctor cache invalidation: plugin version bump triggers re-probe; moved binary path triggers re-probe

### Tier 4 — Installed-smoke tests (CI feature flag)

Only run when real CLIs are installed. Guarded by `CPS_INSTALLED_SMOKE=1` env var so they don't fail CI runs that lack the binaries.

- `codex` installed-smoke: spawn real `codex` with `--sandbox read-only`, send a 2-turn prompt, assert response shape + non-empty text
- `claude` installed-smoke: spawn real `claude` (default variant) with read-only mode, same assertion
- `claude{kimi-k2.6}` installed-smoke: requires running Ollama Cloud session; spawn `claude` with env vars + model flag, assert response shape AND that the response did NOT originate from Anthropic (cross-model verification via response metadata)
- Per-CLI smoke also covers: ctrl-C handling, timeout enforcement, session continuity (where supported)

### Tier 5 — Replay tests

Given a recorded sidecar turn, can we re-dispatch and reconstruct? Validates stateless-replay invariant.

- Record a turn under fake-CLI → reload sidecar → call replay helper → assembled prompt matches `inputs_hash`
- Hash assertion: replay's `response_text` matches recorded `response_hash` (modulo non-deterministic model output — assert only `inputs_hash` and adapter signature equality)
- Cross-CLI replay: record under codex → replay under claude → flag adapter mismatch in audit (warning, not fail)

### Tier 6 — Honest-reporting compliance

All 6 codex-paired skills write the activation marker on entry. The Stop/PreToolUse hook fires on unsourced claims. v0.8.1 / v0.8.1.1 tests cover this; v0.9.0 extends with:

- Resolution audit fields counted as evidence (mentioning a CLI ladder index is verifying state)
- Cross-adapter prose (e.g., "we VERIFIED via codex" vs "we VERIFIED via claude{kimi-k2.6}") routes through the same evidence-proximity rules

### Tier 7 — Live verification (release gate)

Before v0.9.0 tags: end-to-end run with **at least two real CLIs** (codex + claude minimum; ideally + gemini if installed; + `claude{kimi-k2.6}` if Ollama Cloud configured) against a contrived spec.

**Pass criteria:**

1. `composeExperts` selects ≥2 distinct roles
2. Each role's dispatch resolves to a different CLI (cross-model diversity actually observed)
3. Sidecar resolution-audit fields fully populated per turn
4. At least one peer DM enqueued and consumed by recipient role
5. Final round emits double-SHIP across all selected advisors
6. Replay from sidecar reconstructs round-1 prompt identically

If any criterion fails, the release gate does NOT pass. Modeled on v0.8.0's release-gate-doc pattern with PASS/PENDING/FAIL annotations.

### What does NOT get tested at the unit/integration tier

- **Live model output quality.** We don't assert "codex's review caught bug X." That's a meta-evaluation; release gate is the closest thing, and it's a sanity check, not a correctness gate.
- **Specific model latencies.** Timing varies. Tests assert `duration_ms` is recorded, not that it's under N seconds.
- **CLI-vendor-specific bugs.** If `codex --json` changes its event schema, our adapter breaks; we catch that in installed-smoke + add a regression test pinning the event-schema contract at the boundary.

### CI shape

- **Default CI run:** Tiers 1–3 + Tier 6. ~3 minutes. Fake CLIs, no external dependencies.
- **Smoke run** (`CPS_INSTALLED_SMOKE=1`): adds Tier 4. ~5 minutes additional. Requires real CLIs installed in CI runner.
- **Release run** (manual gate): adds Tier 5 + Tier 7. Operator-driven.
