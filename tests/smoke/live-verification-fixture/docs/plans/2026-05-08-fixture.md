# Fixture Implementation Plan

**Spec:** docs/specs/2026-05-08-fixture-design.md

**Goal:** Implement the two user-visible features of the Live Verification Fixture app so Phase E autopilot can exercise its full lifecycle against a real (intentionally buggy) implementation.

---

## Slice 1: Implement Save Display Name

**live-verification:** not skipped (Phase E should run and catch the bug)

**Validation:** critical

### Tasks

- [ ] **1a: Implement the Save Display Name form** — `index.html` input + Save button + current-name label. The button calls `POST /save` with `{ displayName }` and re-reads `/display-name` to update the label.

- [ ] **1b: Implement `POST /save` on the server** — reads the JSON body and writes `displayName` to the backing store. (NOTE: the current implementation has an intentional bug: it writes to `store.name` instead of `store.displayName`, so the label never updates. Phase E exists to catch this.)

- [ ] **1c: Implement `GET /display-name`** — returns `{ displayName: store.displayName }` from the backing store.

- [ ] **1d: Implement `GET /saves`** — returns `{ saves: store.saves }` from the backing store (powers the Recent Saves list).

- [ ] **1e: Implement `GET /healthz`** — returns `{ ok: true }` (ready-signal for Phase E launcher).

- [ ] **1f: Implement reset-store script** — `scripts/reset-store.js` clears `store.json` back to `{ saves: [], displayName: "" }`.

**Expected Phase E outcome:** Scenario A (Save Display Name) fails because the label does not update after save. Server logs `ERROR: missing field 'displayName'` when the backing store has no `displayName` field written. Fix-subagent corrects the field key. All scenarios re-run; both pass. Slice ships.
